68000 Assembly

The Programming Model

Before any instruction can do anything, there has to be something for it to do it to. That something is the 68000's programmer-visible state: sixteen registers, a program counter, a status register, and a flat memory map. Every instruction in the language reads or writes some part of it. This page is the map of that state, the place the rest of the track is built on. Every example is real, pulled from the Sonic 2 disassembly and re-verified line by line, unless a snippet is marked as a teaching example.

In a hurry, and you already know the mnemonic you want? Skip to the instruction index at the bottom: type rte, pea, or any other, and land on its entry.

Notation, if any of it is new: $ is hexadecimal and % is binary; # marks an immediate constant (#$100 is the number itself); parentheses mean the memory at an address ((a0) is what a0 points at, not a0 itself); and the size suffixes .b, .w, .l select 8, 16, or 32 bits. An assembly line normally begins with an instruction mnemonic followed by its operands, and in this wiki's syntax the source comes first and the destination second: move.w source,destination puts the value into the right-hand operand. Some source omits a size suffix when the instruction or its operands already fix it. The Addressing Modes page carries the full key.

1 · The register file

The 68000 gives you sixteen general registers to work in, split into two even groups of eight, plus the program counter and the status register off to the side. Eight data registers, d0 through d7, hold the values a program computes with: a ring count, a coordinate, a byte of flags, a loop counter. Eight address registers, a0 through a7, hold pointers: the address of an object, a table, a buffer. And one program counter tracks where execution currently is. Every data register, every address register, and the program counter is 32 bits wide (the status register is the exception, at 16 bits), and all of them live inside the CPU itself. That is worth stating plainly, because it is the source of a lot of early confusion: unlike a byte of memory, a register has no address of its own. You name it (d0, a3) rather than point at it, and no memory address ever reaches one. This is the whole working surface: instructions pull values in from memory, do the shuffling and computing in these registers (far quicker to reach than memory itself), and write the results back out.

Two parts of this state deserve special attention. a7 is an address register, but it also serves as the stack pointer, which subroutine calls, returns, and exceptions use to push and pop values, so it is written sp just as often as a7; the two names refer to the same register. The status register is not general-purpose at all. Its dedicated job is to hold the condition codes and the system-control bits. The diagram below is the core programmer-visible state; poke any part of it to see what that part is for.

New here? · What is a register, and why not just use memory?

A register is a tiny storage slot built right into the CPU, big enough for one number. The 68000 has a fixed, small set of them, and they have names baked into the language (d0, a3, and so on) rather than addresses. That fixed small set is the point: because a register lives inside the processor, reading or writing one costs no extra bus traffic, while every touch of memory means a trip out across the bus and back, several times slower.

So the shape of almost all 68000 code is the same dance: load the values you need from memory into registers, do the arithmetic and logic and comparisons there where it is cheap, and write the finished results back out to memory. A register is scratch space you can reach instantly; main memory is the filing cabinet you visit only when you must.

New here? · If they are all 32 bits, what do .b, .w, and .l mean?

Every data register is physically 32 bits wide, but most instructions let you say how much of that width to touch. The size suffix goes on the mnemonic: .b means a byte (the low 8 bits), .w means a word (the low 16 bits), and .l means a longword (all 32 bits). So move.b operates on the bottom byte of its destination, move.w on the bottom 16 bits, and move.l on all 32. (What happens to the bits a partial size does not touch is the very next explainer.)

Picking the size that fits matters, but not the way beginners assume: a register-to-register move is 2 bytes and 4 cycles at every size, so the suffix alone is not smaller or faster. Size pays off with memory transfers (a byte or word often moves in fewer bus cycles) and with immediates. A byte or word immediate both occupy one extension word, while a long immediate needs two, so the real saving is byte-or-word versus long, not byte versus word.

New here? · What does a .b or .w write leave behind?

On a data register, a .b or .w operation writes only the low 8 or 16 bits and leaves the upper bits exactly as they were. It does not clear them. (The "low byte" is the rightmost two hex digits of the 32-bit value.)

Teaching example 
	move.l	#$12345678,d0	; d0 = $12345678
	move.b	#$AB,d0	; d0 = $123456AB  (low byte only)
	move.w	#$CDEF,d0	; d0 = $1234CDEF  (low word only)

An address register behaves the opposite way. A word moved into one is sign-extended to fill all 32 bits:

Teaching example 
	movea.w	#$8000,a0	; a0 = $FFFF8000  (top bit copied up)

Data and address registers look alike, both 32 bits, but the 68000 treats them differently on purpose, and the difference is worth pinning down early because it shapes which instruction you reach for.

New here? · Why split data registers from address registers at all?

You might expect one pile of sixteen identical registers, and some CPUs work that way. The 68000 splits them because the two jobs want different behavior. A data register is for a value you compute on, so it supports every size and sets the condition flags as a side effect: subtract into d0 and the Z flag tells you whether the result was zero. An address register is for a pointer, so it has the opposite habits: it does not deal in bytes (only word and long), address-register arithmetic never touches the flags (you rarely want a pointer calculation to disturb a comparison in progress), though an address register used as an operand in a comparison sets them like any other value, and a word written into it is automatically sign-extended, its top bit copied up to fill all 32 bits (the arithmetic is worked out in section 4), because a pointer must always be a full-width address.

You will feel the split in the instruction names. Loading a value uses MOVE; loading a pointer uses MOVEA. Adding to a value uses ADD; adding to a pointer uses ADDA. The A suffix is the CPU reminding you which world you are in.

2 · The status register and condition codes

An interrupt is a hardware request that pauses the current code, runs a designated handler, and normally returns to the interrupted work with rte; higher-numbered interrupt levels have higher priority.

The status register is 16 bits, but it is really two half-registers with very different audiences. The low byte is the condition code register (CCR): the five flags X N Z V C that ordinary code lives on. The high byte is the system byte: the supervisor bit, the interrupt mask, and the trace bit, which only supervisor code may write directly (reading the SR is not privileged). Of the 16 bits, only 10 actually carry anything: 5 in the CCR and 5 in the system byte (the trace bit, the supervisor bit, and the 3-bit interrupt mask). The other 6 are unused and reserved: they read as zero on the 68000, but treat them as reserved, write them as zero and do not rely on the value. That is why a full status-register constant like #$2700 is mostly zeros. The 16-bit ruler in section 1 shows exactly which bit is which. The MOVE that initializes the SR during boot writes both halves at once:

	movem.l	(a6),d0-a6
	move	#$2700,sr	; set the status register
s2disasm/s2.asm:272-273 (the standard console init, just before the game starts)

The five condition codes

Most instructions update the CCR as a side effect of whatever they just computed. On their own the flags do nothing; their purpose is that a later conditional instruction (a branch, or an Scc, DBcc, and the like) can read them and decide what to do, as long as nothing changed the relevant flags in between. This is how the CPU makes decisions without a separate "test" step: subtract, compare, or even just move a value, and the flags already describe the result.

It is worth taking the phrase "side effect" literally, because it clears up the usual confusion about when the flags get set. An instruction like add produces two outputs at once: the result, which goes to the destination you named, and a flag update, which goes into the CCR. You asked for the sum; the flag update comes with it, every time, whether you wanted it or not. That is the side effect. It is also why LEA and MOVEA feel different: they were built to have only the first output and touch no flags, so a branch placed after one is reading whatever the last flag-setting instruction left behind.

The timing follows from that. The flags are written as the instruction completes, from the result it just produced, and each flag holds its value until an instruction that affects that particular flag changes it. Different instructions touch different flags, so at any moment the five can reflect more than one earlier instruction (X in particular survives most operations). Take add.b #2,(a0) where the byte at (a0) holds $FE (254). The byte add lands on $100, which does not fit in a byte, so it wraps to $00, and in the same step the CPU records how that came out: Z is set (the result was zero), C and X are set (the add carried out past the top bit), and N and V are cleared. A beq on the very next line then acts on that Z with nothing in between, because the add already did the testing as part of adding.

Nnegative
A copy of the result's top bit. Set means the result, read as a signed number, is negative. BMI branches on it directly; the signed less-than branch BLT combines it with V (it tests N XOR V).
Zzero
Set when the result was exactly zero. This is the flag beq and bne read.
Voverflow
Answers the signed question: did the true result fall outside the signed range? For an 8-bit value that range is -128 to 127; if the correct answer lands outside it, V is set. Signed branches like BLT read it (via N XOR V).
Ccarry
Answers the unsigned question: did the true result fall outside the unsigned range? For an 8-bit value that range is 0 to 255; C is set when an add goes above 255 or a subtract needs to go below 0. Unsigned branches like BCS / BLO read it.
Xextend
A carry-like flag, updated alongside C by most arithmetic but a separate flag (a cmp changes C while leaving X untouched), so extended, multi-word math (addx, roxl) can thread a carry across several longwords even when a compare or branch sits between the steps.
Teaching example · one add, two interpretations: V and C

The CPU does not do a "signed add" or an "unsigned add." It does one binary addition and sets both flags. Your code decides which reading matters: V answers the signed question, C the unsigned one. Two additions make the point, one that trips signed overflow and one that trips unsigned carry.

Signed overflow, $7F + $01:

  0111 1111    $7F
+ 0000 0001    $01
-----------
  1000 0000    $80

All eight bits fit, so there is no carry: C=0. But read as signed, $80 is -128, not +128. 127 + 1 cannot fit in the signed range -128 to 127, so the sign came out wrong and V=1.

Unsigned carry, $FF + $01:

  1111 1111    $FF
+ 0000 0001    $01
-----------
1 0000 0000    $100

The complete answer is $100, which needs nine bits, but the byte holds only eight. So it stores the low eight (0000 0000 = $00) and records the extra bit by setting C=1. That leftmost 1, sitting past the left edge of the 8-bit field, is the carry made physical. It is not that 256 fits in 0000 0000: 0000 0000 is only the part that fit, and the lost ninth bit is exactly why carry is set. Read as signed, -1 + 1 = 0 fits fine, so V=0.

The same two additions, side by side:

AdditionSigned readingUnsigned readingFlags
$7F + $01 = $80127 + 1 should be 128, outside -128 to 127127 + 1 = 128, fits 0 to 255V=1, C=0
$FF + $01 = $00-1 + 1 = 0, fits -128 to 127255 + 1 should be 256, outside 0 to 255V=0, C=1

Both additions produce a valid 8-bit bit pattern. The flags just tell you whether that pattern is a valid answer under the interpretation your program cares about.

Which flag do you check? The CPU sets both because it does not know what the bits mean to your program. Check V when the value is meant to be signed, and C when it is meant to be unsigned. A signed velocity might run -128 to 127: if adding acceleration turns 127 into -128, V reports that the signed answer did not fit. An unsigned counter might run 0 to 255: if bumping 255 gives 0, C reports that the answer needed a ninth bit. The very same instruction can be checked either way:

	add.b	#1,d0
	bvs.s	SignedOverflow		; d0 is signed (a velocity)
	bcs.s	UnsignedOverflow	; d0 is unsigned (a counter)

The arithmetic is identical; the correct branch depends on what d0 represents. The gap shows up most sharply in comparisons, where the signed and unsigned "less than" can disagree:

	cmp.w	d1,d0			; compare d0 against d1
	blt.s	D0IsLower		; signed less-than
	blo.s	D0IsLower		; unsigned lower

Say d0 = $FFFF and d1 = $0001. Signed, those are -1 and 1, so d0 is lower and blt branches. Unsigned, they are 65535 and 1, so d0 is not lower and blo does not. Same bits, same compare, opposite answer: the branch you choose declares which interpretation you mean. For quick reference, the common signed and unsigned branches:

New here? · What does "the result reads as negative" mean? There is no minus sign.

Right, there is no minus sign in hardware; a register only holds bits. Negative numbers work by wrap-around, like an odometer rolling backward past zero. The bit pattern $FF in a byte means 255 or -1 depending entirely on how the code chooses to read it, and the tell is the top bit: read as signed, patterns with the top bit clear are zero and up, and patterns with the top bit set are the negatives. The N flag is nothing more than a copy of that top bit. So "N is set" literally means "the top bit of the result is 1," which signed code reads as negative and unsigned code reads as just a large value.

That is also why the same flag serves two readings without the CPU knowing which you meant: the hardware sets N, Z, V, and C mechanically from the bits, and your choice of branch (a signed blt versus an unsigned bcs) decides which flags to believe. The full odometer treatment, including why a negative needs more F digits at a bigger size, is on the Moving Data page.

This page explains what the codes mean. Watching them actually flip as instructions run, picking an operation and seeing X N Z V C change before and after, is the job of the live flags demo on the Math & Logic page (coming), which is where that toy lives so it can sit next to the instructions that set the flags in interesting ways.

Gotcha Not every instruction touches the flags, and assuming they all do is a classic bug. A plain MOVEA or an adda into an address register changes no condition codes, so a branch after one is reading whatever the last flag-setting instruction left behind, not the address you just loaded. When you need the flags set from a value, use a data-register operation or an explicit tst.

The system byte: control, not a side effect

Everything above is the CCR, the low half, written for you as a side effect of arithmetic. The high half, the system byte, works the opposite way: no ordinary add or move ever touches it. It holds mode and control state, not a record of results, and it changes only on purpose, when the CPU takes an exception or interrupt, when rte returns from one, or when privileged code writes the SR directly with move #x,sr. So an instruction that reshapes the CCR leaves the whole top byte exactly as it was. Its three fields: the trace bit T, which when set makes the CPU trap after every instruction so a debugger can single-step; the interrupt mask I2 I1 I0, a priority threshold saying which interrupts are currently allowed to break in; and the supervisor bit S, the privilege switch, which is the one piece the Genesis actually leans on.

Supervisor and user mode

The supervisor bit (S, bit 13 of the status register) picks which of two worlds the CPU runs in. In supervisor mode everything is allowed: the privileged instructions, the system byte of the SR, the lot. In user mode, writing the system half of the SR is privileged (reading the SR is not, on the MC68000) and the privileged instructions trap instead of running, so a misbehaving program cannot mask interrupts or reprogram the machine out from under the operating system. The two modes also get their own stack pointers, the detail the "two a7 registers" paragraph below pins down.

Hardware choice · two modes, one register name Motorola built the supervisor / user split so a real operating system could protect itself from application code. The Genesis has no operating system: Sonic 2 comes up in supervisor mode at reset and stays there for the entire run, so the user stack pointer is never really used. The boot code still initializes it, to a clean zero, before switching the SR to its running value: 
	movea.l	d0,a6	; clear a6
	move.l	a6,usp	; set usp to $0
s2disasm/s2.asm:217-218 (d0 was just zeroed by the line above) The takeaway for reading Genesis code: assume supervisor mode throughout, and treat the whole user-mode machinery as inherited plumbing rather than something the game switches into.
New here? · If the game never leaves supervisor mode, why does the split matter to me?

Two reasons it is worth knowing even though Sonic stays in one mode. First, it explains why some instructions "mysteriously" refuse to run: move <ea>,sr, stop, reset, and the USP moves are privileged, and if you ever saw one trap you would be hunting a ghost without knowing the rule. On the Genesis they work precisely because the game is always supervisor.

Second, the split is why the 68000 has that odd doubled stack pointer (spelled out in the paragraph just below). Since the Genesis lives in supervisor mode, "the stack" in every other section of this track means the supervisor stack, and you never have to think about the other one.

That doubled stack pointer is worth stating outright: the 68000 physically has two a7 registers, a user stack pointer (USP) and a supervisor stack pointer (SSP). The name a7 always refers to whichever one belongs to the current mode, and it switches automatically the moment the S bit changes. Exceptions and interrupts always run on the supervisor stack; supervisor code can reach the inactive user one explicitly with MOVE USP. Because the Genesis stays in supervisor mode, the user stack barely appears in ordinary game code, but this split is why the register diagram shows one a7 while the manual keeps talking about two stacks.

3 · The program counter and labels

The program counter is the CPU's finger on the page: a 32-bit register holding the address of the next instruction to fetch. Run an instruction and the finger has already slid past it to the following one. That is the entire mechanism of "running a program", fetch, advance, fetch, advance, and every other trick, loops, calls, jumps, is just some instruction reaching over and setting the finger somewhere else.

Hardware choice · the PC is not a general register The 68000 deliberately keeps the program counter out of the d0-d7 / a0-a7 file. You cannot write move.l #$1000,pc; the assembler will not even encode it. The instructions that directly express ordinary control flow are the ones whose entire job is to change the PC: the branches (bra, bcc), the jumps (jmp), the calls (bsr, jsr), and the returns (rts, rte). This is not a limitation to work around: control flow moves only through the instructions built for it (branches, jumps, calls, returns), which is simply how the 68000 is designed. A jmp ($00000206).l is the 68000's way of spelling "put this address into the PC", and rts is "pop four bytes off the stack into the PC."
New here? · So how does a loop or an "if" happen at all?

Entirely by writing the PC. Normally each instruction lets the finger slide to the next in line. A branch is an instruction that, instead, sets the PC to some other address, so the next fetch comes from there. An unconditional branch (bra) always does it; a conditional branch (beq, bne, and the rest) first checks a condition code and only redirects the finger if the flag says so, otherwise it falls through to the next instruction. A loop is a conditional branch pointing backward; an "if" is a conditional branch pointing forward over the code you want to skip.

A subroutine call adds one idea: before redirecting the finger, it writes down where the finger was (the return address) so a later rts can restore it. Where it writes that down is the stack, which is why section 5 and the PC are so tightly linked. All of this is the Flow & System page's territory (coming); here the point is narrower: the PC is just a register, and control flow is just instructions assigning to it.

The PC also has a day job as an addressing base. A PC-relative operand like SetupValues(pc) means "this far from wherever the PC is right now." Both absolute and PC-relative labels survive ordinary source edits, because the assembler recomputes them every build. PC-relative's extra benefit is that an already-assembled block of code and its nearby data can be relocated together without patching embedded absolute addresses. That mode, and why it is read-only, is covered on the Addressing Modes page; the register itself is the point here.

Labels: names the assembler turns into addresses

You almost never write a raw PC value, or any raw address. You write a label, a name like EntryPoint, Vint_routine, or ErrorTrap, and let the assembler work out the number. This is not a runtime feature; it is plain bookkeeping done once at build time, and understanding it removes a whole category of confusion about where addresses "come from."

New here? · Who decides a label's address?

The assembler does, by counting. It walks your source from the top, adding up the size of every instruction and every piece of data it emits, keeping a running total of "how many bytes have I placed so far." When it reaches one of your labels, whatever that running total has reached IS the label's address, from then on, everywhere the name appears. You never type the number; you type the name, and the assembler substitutes the count. That is how a location label gets its value, from the assembler's location counter; other symbols can instead be assigned explicitly with equ or =, and origins or sections can place code and data at fixed addresses.

Two consequences fall straight out of that. First, add one instruction above a label and the label's address quietly shifts, because the running count reached it later; nobody edits it by hand, it just moves. Second, a label is only a name: at runtime the machine sees the resolved number and nothing else. jmp EntryPoint and jmp ($00000206).l assemble to the identical bytes if EntryPoint sits at $206 and the same addressing form is selected (an assembler may pick absolute-short versus absolute-long). The name is for you; the number is for the CPU. This is the same bookkeeping whether the label names a spot in the code or a slot in RAM, which is why the next section can talk about RAM variables and code addresses in one breath.

System_Stack, EntryPoint, and ErrorTrap are all labels the assembler resolves to real addresses. At reset the 68000 does something delightfully literal: it reads the first longword of the ROM into a7 (that is the initial stack pointer) and the second longword into the PC (that is where execution begins). No magic entry point, just "the address written in the second slot of the table." Every other slot is an address the CPU jumps to when something goes wrong, and this ROM has pointed almost all of them at ErrorTrap, a choice section 6 comes back to.

New here? · What does a reset actually set up?

Less than you might guess. Reset does not zero the general registers. It puts the CPU in supervisor mode, clears the trace bit, sets the interrupt mask to 7 (all ordinary interrupts masked), loads the supervisor stack pointer from the longword at $000000 and the PC from the longword at $000004, and leaves d0-d7 and a0-a6 undefined.

That is exactly why boot code does so much clearing and setup: none of the register or hardware state it intends to rely on is guaranteed by reset, so it has to establish that state itself before the game can start.

4 · Pointers and the memory map

An address register holds a pointer, and a pointer is just a number that names a location. Address registers are pointer-oriented, not pointer-typed: they can hold any 32-bit pattern, and the CPU never verifies that the pattern is a sensible address, which is why a bad pointer produces such chaotic failures. The 68000's addresses are 32-bit numbers, so in principle they range over four gigabytes. In practice the chip is wired to something much smaller, and the gap between the two is the source of both a useful trick and a lurking trap.

Hardware choice · 32 bits stored, 24 bits decoded The 68000 keeps addresses as full 32-bit values inside its registers, but its package only has 24 address pins, so when it reaches out to memory it uses the low 24 bits and ignores the top 8. That fixes the real, physical address space at 2^24 = 16 MB, $000000 to $FFFFFF. 24 pins make for a cheaper package, while the full-width registers left room for later chips (the 68020 and up) to wire all 32. On the Genesis, always a plain 68000, only those 24 bits ever matter; the full consequences of that gap are the "two spellings, one location" note at the end of this section.

Not every address selects stored memory. Some Genesis addresses select hardware registers, so writing to $C00000, for example, sends data to the VDP rather than storing it in RAM. This is called memory-mapped I/O. The Genesis fills that 16 MB map with ROM down low, the hardware chips in the middle, and the 68000's work RAM right at the top, from $FF0000 to $FFFFFF. That placement carries a benefit worth understanding, because it rests on a small piece of arithmetic that explains a notation you see on nearly every line of Genesis code.

New here? · Why is the RAM at the very top of the map, not the bottom?

Because of how the cheapest way to name an address works. The 68000 lets an instruction carry a memory address as either a full 32-bit number (four extra bytes, slow) or a compact 16-bit short form (two extra bytes, fast). But 16 bits can only count to $FFFF, so how does a short address reach anywhere useful? The CPU sign-extends it: it copies the 16-bit value's top bit up through all the high bits. A short address from $0000 to $7FFF stays low, but one from $8000 to $FFFF gets its top bit copied up and becomes $FFFF8000 to $FFFFFFFF, which after the 24-bit decode is $FF8000 to $FFFFFF, the top 32 KB of the map.

So the short form can only reach two regions: the bottom 32 KB and the top 32 KB. Placing the Genesis work RAM at the top of the address map makes its upper 32 KB ($FFFF8000-$FFFFFFFF) reachable through this cheap two-byte form, and Sonic-style engines lean on that by keeping many frequently-touched globals up there. The lower half of work RAM, below $FFFF8000, needs the four-byte long form. That is why RAM references are written (Something).w all over the source: the .w is asking for the short, sign-extended address, and it reaches those globals because they sit where sign extension lands. The absolute-short mode on the Addressing Modes page walks the same arithmetic.

You can watch a work-RAM variable get named the short way in almost any routine. This one reads the current frame's video-interrupt routine number and then sets the next one:

	move.b	(Vint_routine).w,d0
	move.b	#VintID_Lag,(Vint_routine).w
s2disasm/s2.asm:500-501 (the main loop's wait for the next frame)

Vint_routine is a label the assembler resolves to $FFFFF62A, a byte up in work RAM. Because the code writes it as (Vint_routine).w, only the low 16 bits, $F62A, are stored in the instruction; at run time the CPU sign-extends that back to $FFFFF62A (the top bit of $F62A is set) and reaches the right byte. Two bytes in the instruction, the whole top of the map in range. That is the upshot of that placement.

The two suffixes on move.b (Vint_routine).w,d0 answer different questions. move.b is the data size, one byte; (Vint_routine).w is the addressing form, absolute-short. The .w describes the address, not the data. (The Addressing Modes page covers the addressing form in full.)

Byte order: the 68000 is big-endian

When a word or longword is stored in memory, the 68000 puts the most-significant byte at the lowest address, with successively less-significant bytes at increasing addresses. This is called big-endian, and it is the order you read a number in on paper.

	move.l	#$12345678,($FF0000).l

This is not a curiosity: reading ROM or RAM hex dumps, debugger memory windows, the layout of the stack, and the byte order of data tables all depend on knowing that the high byte comes first.

Alignment: words and longs go on even addresses

Byte accesses may use any address at all. But word and longword accesses must start at an even address on the 68000. A word or long access to an odd address raises an address error, the same hardware exception family as fetching from an odd PC. This is also why a byte push or pop steps a7 by 2 rather than 1: the extra step keeps the stack word-aligned so the words and longs stored on it always land on even addresses.

The top byte is dead weight, so never smuggle data in it

The flip side of "the top 8 bits are ignored" is a trap that has bitten real software for decades. Since the 68000 throws those bits away when it dereferences, you can set them to anything, a flag, a type tag, a counter, and the pointer still works. It is tempting: free storage riding along in every address. And on the 68000 it genuinely does work, which is exactly what makes it dangerous.

Gotcha Never store flags in the top byte of a pointer, even though the 68000 lets you get away with it. The bits are ignored only because this chip decodes 24 bits; a 68020 or later decodes all 32, so the same code that "worked" now dereferences $12FF0000 instead of $00FF0000 and reads garbage. Keep addresses as pure addresses; keep flags in their own byte.
Aside · why some 1980s software broke on later CPUs

This is not hypothetical. A wave of early Macintosh software did exactly this, tucking flags or type tags into the top byte of pointers because the 68000 threw those bits away on every dereference. It worked perfectly until the hardware caught up: when Macs moved to the 68020 and later, which decode all 32 bits, those addresses suddenly pointed somewhere else and the programs read garbage. The industry had to laboriously rewrite that code to be "32-bit clean," the exact rework the gotcha above is telling you to avoid up front.

The ROM header wants the address in its physical 24-bit form, so Sonic 2 masks the RAM start address down before storing it there.

	dc.l RAM_Start&$FFFFFF		; Start address of RAM
s2disasm/s2.asm:175 (the cartridge header's RAM range)

The label RAM_Start lives at $FFFF0000 as a full 32-bit value; the &$FFFFFF keeps only the low 24 bits, $FF0000, so the header field records the physical address the hardware decodes rather than the padded 32-bit one the register holds. That mask is specific to the header field. It is not evidence that the engine strips the top byte from its live pointers: it does not. Its work-RAM symbols are written full-width, like $FFFFF62A, top $FF and all, and dereferenced that way at run time.

Logical vs physical · two spellings, one location Address registers and the PC are 32 bits wide, but the MC68000 exposes only a 24-bit physical address bus, so Genesis hardware decodes just the low 24 bits of any address. That gap has a visible consequence in the source: assemblers commonly write high work-RAM addresses in sign-extended form, like $FFFFF600, even though the physical address is $FFF600. Both $00FFF600 and $FFFFF600 name the same location, because the top byte is discarded before the address ever reaches memory. The header mask above is the one place a physical 24-bit address is actually stored; everywhere else the padded logical form is normal and correct.

5 · The stack

The stack is not a special chip or a separate memory; it is just an ordinary region of work RAM that a7 points into, used with a simple discipline. Pushing means "step a7 down and write below where it was"; popping means "read where a7 points and step it back up." The 68000 builds this right into two addressing modes: -(sp) predecrements before writing (a push) and (sp)+ postincrements after reading (a pop). The stack grows downward, from high addresses toward low, which is why section 3's vector table hands the CPU an initial a7 up near the top of RAM.

New here? · What is the stack actually used for?

Three everyday jobs, all variations on "set something aside and get it back in reverse order." The first is subroutine return addresses: bsr and jsr push the address to come back to, and rts pops it. The second is saving registers: a routine that needs to use d0-d3 but must not disturb its caller's copies pushes them on entry and pops them on exit. The third is passing values between routines when there are more of them than fit comfortably in registers.

The reason a stack works for all three is that calls nest: routine A calls B calls C, and they finish in the opposite order, C then B then A. A stack is exactly the structure that gives back the most recently saved thing first, so each rts naturally finds its own return address on top, and each register-save is unwound by its matching restore. The whole scheme rests on one rule, below.

That one rule is balance: whatever a routine pushes, it must pop before it returns, on every path out. The rts at the end blindly pops four bytes into the PC and jumps there; it has no idea whether those four bytes are the return address you meant or a register you forgot to pop. If the pushes and pops do not balance, rts loads the wrong four bytes and the PC goes wrong, which is exactly the failure section 6 is about. A clean routine brackets its work with matching save and restore. Here it is at its simplest:

Teaching example · a balanced call, step by step

A constructed example (not lifted from the disassembly), with toy addresses to make the discipline concrete. Say the call is made from pc = $00001000 with sp = $FFFFFF00, and the routine saves one register, works, and returns:

	pc = $00001000, sp = $FFFFFF00
	bsr.w	Sub		; push return address $00001004 (just past this 4-byte bsr.w), sp drops to $FFFFFEFC
	; inside Sub:
	move.l	d0,-(sp)	; save d0, sp drops to $FFFFFEF8
	; ... work ...
	move.l	(sp)+,d0	; restore d0, sp rises to $FFFFFEFC
	rts			; pop $00001004 into pc, sp rises to $FFFFFF00

Because the one push of d0 is matched by exactly one pop, sp is back at $FFFFFF00 when rts runs, so the four bytes it reads are genuinely the return address, not the saved d0. Drop the move.l (sp)+,d0 and rts would pop d0's old value into the pc and jump into nowhere. Balance is the whole game.

Step through it: watch the return address land in memory, the balance hold or break, and see that an exception uses the very same stack.

Real routines do the very same thing at scale. This decompressor brackets its work by saving and restoring thirteen registers in one instruction:

NemDec:
	movem.l	d0-a1/a3-a5,-(sp)
	; ... decompression body omitted ...
	movem.l	(sp)+,d0-a1/a3-a5
	rts
s2disasm/s2.asm:1806-1807, 1844-1845 (NemDec, the Nemesis art decompressor)

The save pushes thirteen registers (d0-d7, then a0-a1, then a3-a5, 52 bytes on the stack); the restore just before rts pops the same thirteen, so a7 is back precisely where it started and the four bytes rts reads are genuinely the return address. Note d0-a1 runs from d0 straight through a1, crossing out of the data bank into the address bank. Get that list even one register out of balance and rts returns into the middle of nowhere. The trap is sharpest when a routine has an early exit: every return path, not just the last, has to undo the same pushes, which is why the balance is a property of the whole routine and not just its final line.

Gotcha Because the return address is just four bytes sitting at (sp), ordinary code can overwrite it, and rts will happily "return" wherever the new value points. Done by accident, that is the balance bug above: the CPU cannot tell a corrupted return slot from a valid one, because it never knew those four bytes were a return address in the first place. One subtlety: a restore list with a different set of registers but the same count still balances sp (it just restores the wrong values into the wrong registers); only a different total byte count actually unbalances sp and misdirects rts.
Aside · the same slip, weaponized: speedrun arbitrary code execution

Done on purpose, from outside the game, the same overwritable return slot is the backbone of speedrun "arbitrary code execution" (ACE): feed the game inputs that make it write a chosen address over a return slot, and the next rts jumps into attacker-controlled data, turning a platformer into a tiny programmable machine. Same mechanism as the balance bug, one an accident and one an exploit; the CPU treats them identically because it never distinguished the four bytes as a return address at all.

One last distinction the stack makes easy to blur: the CPU defines the mechanism, not the policy. bsr and jsr push a return address, rts pops it, and exception entry and rte do their fixed dance, all of that is wired into the silicon. But which registers a routine must preserve, where its parameters go, how it hands back a result, and who pops arguments off the stack afterward are software conventions the engine chooses, not rules the 68000 enforces. That is why one Sonic routine carefully saves registers on entry while another freely clobbers them: each is following (or ignoring) a convention the code agreed on, and the CPU is indifferent either way.

6 · Exceptions and when the PC goes wrong

Because control flow is nothing but "some instruction writes the PC," a wrong value in the PC is one of the ways a 68000 program dies. It is worth knowing exactly how it dies, because the failure has two distinct shapes and neither one looks like the tidy crash you might expect.

A crash, though, is just an exception the game did not want, and every exception (wanted or not) follows the same six steps. This is the sequence that ties together the supervisor stack, the vector table, and rte that the rest of this page has met one piece at a time:

  1. The exception fires: a bus or address error, an illegal instruction, a trap, an interrupt, and so on.
  2. The CPU switches into supervisor mode.
  3. It switches to the supervisor stack, so a7 now points at the SSP.
  4. It pushes the current SR and PC onto that stack, saving enough to resume the interrupted work later.
  5. It reads the handler's address from this exception's slot in the vector table and loads it into the PC.
  6. The handler runs; if execution is meant to continue, it ends with rte, which pops the saved SR and PC back and carries on where it left off.

This is the common conceptual sequence; it excludes reset, which builds no saved frame at all. The exact stack frame varies by exception, and bus and address errors save extra diagnostic information beyond SR and PC.

ErrorTrap, below, is a handler that does run (step 6's "handler runs" part) but deliberately never reaches the rte: it loops forever, which is the silent freeze.

New here? · How can the program counter end up wrong in the first place?

The usual way is a corrupted return address. A subroutine call stashes its return address on the stack, and if the routine writes past the end of a buffer, pushes and pops that do not balance, or simply computes a jump target wrong, the value the rts loads into the PC is no longer a real instruction address. The finger lands somewhere it should not, and the CPU, which has no idea anything is amiss, tries to fetch and run whatever is there.

The 68000 has no memory protection and no concept of "invalid code." It will faithfully execute whatever bytes the PC points at. So the failure is never a clean "illegal jump" message; it is the machine earnestly running nonsense. What happens next depends entirely on whether that nonsense address is even or odd.

An odd PC is an instant, specific crash. The 68000 fetches instructions in 16-bit words, and it requires every instruction fetch (and every word or long data access) to sit at an even address. Point the PC at an odd address and the very next fetch raises an address error, a hardware exception, before a single wrong instruction runs. This is the tidier of the two failures, because the CPU caught it.

An even but wrong PC is worse: the CPU runs your data as code. If the bad address is even, nothing objects. The processor reads whatever bytes live there, decodes them as an opcode, and executes them. A table of sprite coordinates, a string, a chunk of compressed art: any of it becomes "instructions," and the machine charges off doing arbitrary things. It MAY eventually hit an illegal instruction, a bad address, or another exception, but nothing guarantees it: the CPU can just as easily loop forever or quietly corrupt memory. There is no guardrail; the discipline that keeps the PC pointing at real code lives entirely in the code, not the hardware.

When the PC does trip an exception, an exception means "jump to the address in this slot of the vector table." Which brings back the table from section 3, and the choice this particular ROM made about where those slots point:

ErrorTrap:
	nop	; delay
	nop	; delay
	bra.s	ErrorTrap	; Loop indefinitely.
s2disasm/s2.asm:188-191 (where stock Sonic 2 sends every error)

Stock Sonic 2 points its bus-error, address-error, illegal-instruction, and nearly every other fault and unused vector at ErrorTrap (the horizontal and vertical interrupt vectors keep their own handlers, H_Int and V_Int), and ErrorTrap is a three-instruction infinite loop that does nothing and says nothing. So a corrupted PC on real hardware produces a silent freeze: the screen holds its last frame, the music usually keeps playing, since the Z80 sound driver runs on its own, but it stops responding to the frozen game, and there is not one word about what went wrong or where. For a shipped game that is a reasonable "fail safe and stop" choice. For someone hacking the game, it is maddening, because every mistake looks identical: a frozen screen with no clue.

Practice · hacks replace the silent trap Because the stock handler throws away every scrap of diagnostic information, the near-universal first move in a Sonic ROM hack is to repoint those vectors at a real error handler. The community standard is Vladikcomper's MD Debugger and Error Handler, which replaces ErrorTrap with code that catches the exception and draws a readable crash screen: which exception fired, the PC and the whole register file at the moment of death, and a stack trace back to the call that went wrong. The same silent freeze that told you nothing becomes a screen that points straight at the bug. It is the single highest-leverage thing to install before doing any serious work on 68000 game code. Before moving on

Five quick checks on the state model this page built. Try to answer each before revealing it.

What does move.b #$AA,d0 preserve?

The upper 24 bits of d0. Only the low byte is written.

Why can (Vint_routine).w reach $FFFFF62A?

Its high $FFxxxx address sign-extends from a single 16-bit word ($F62A becomes $FFFFF62A), so absolute-short reaches the top 32 KB of work RAM.

What does reset load from $000000 and $000004?

The initial supervisor stack pointer from $000000 and the initial PC from $000004.

Why must word and long accesses start on even addresses?

The 68000 requires them to be even-aligned; an odd word or long access raises an address error.

What happens if rts runs while a saved register is still on top of the stack?

rts pops those four bytes into the PC as if they were the return address and jumps into garbage. That is the balance bug.
