68000 Assembly

Instructions: Shifts & Bits

This is where the CPU works at the level of individual bits: sliding a register's bits sideways to multiply or divide by powers of two, wrapping them around in a circle, and reaching in to test or flip a single bit by number. The shift and rotate instructions double as fast arithmetic; the single-bit instructions are how flags in a byte get read and set one at a time. This page covers the shift, rotate, and single-bit families; the flow and branch instructions that act on their results live on another page. Every example is real, pulled from the Sonic 2 disassembly and re-verified line by line.

1 · Shifts

A shift moves every bit of a value sideways by a fixed number of places. Each step, the bit pushed off one end lands in the carry (C) and, for the shift instructions, also in the extend bit (X); the far end fills back in with a new bit. That fill is the whole difference between the four instructions: the logical shifts (LSL, LSR) and the left arithmetic shift (ASL) fill the vacated end with 0, while the right arithmetic shift (ASR) copies the sign bit (the top bit) back in so a negative number stays negative.

The reason to reach for a shift is that it is a cheap multiply or divide by a power of two. LSL #n multiplies by 2^n (every bit steps toward the high end, a 0 enters at the bottom, so the value doubles per step). LSR #n divides an UNSIGNED value by 2^n (bits step down, a 0 enters at the top). ASR #n divides a SIGNED value by 2^n: it is the sign copy that makes this correct, because a negative number must keep its top bits set as it shrinks. That is why Sonic's signed position and velocity math uses asr where address arithmetic uses lsr: the same halving, but one preserves the sign and the other does not.

ASL and LSL are the odd pair: they move the bits identically (left, 0 into the bottom, top bit out into C and X) and produce the same result and the same N, Z, and C. The ONLY difference is V. LSL always clears V; ASL sets V if the sign bit changed at any point during the shift, which is exactly the signed-overflow condition (the number doubled past what a signed value of that size can hold). So use ASL when you are doing signed arithmetic and want the overflow flagged, and LSL when you just want to move bits and do not care about sign.

New here? · A shift is a cheap multiply or divide by a power of two

In decimal, sliding all the digits of 45 one place left gives 450, ten times bigger, and sliding them right gives 4 (with the 5 falling off), a divide by ten. Binary works the same way, except each place is worth 2 instead of 10. So sliding a binary number one place left DOUBLES it, and one place right HALVES it. Shift by n places and you multiply or divide by 2^n: lsl.b #3,d0 multiplies by 8, lsr.w #6,d0 divides by 64.

This matters because a real multiply (MULU) costs around 70 cycles while a shift costs a handful, so whenever the factor is a power of two the code shifts instead. The one catch is division of NEGATIVE numbers: a plain lsr pulls a 0 into the top and turns a negative into a large positive, which is wrong. asr exists for exactly that case: it copies the sign bit back in so a negative value halves correctly. Pick lsr for unsigned values (counts, addresses) and asr for signed ones (positions, velocities).

n is the shift count, so a longer shift genuinely costs more (2 cycles per place), which is why code shifts by the smallest count it can. The memory form shifts a word by exactly 1. c = CPU cycles, b = bytes; Genesis memory adds wait states.

Operands
Register form #n,DnDn,Dn count is an immediate 1 to 8, or a data register (taken mod 64)
Memory form (An)(An)+-(An)d(An)d(An,Xn)(xx)DnAn#n

Two shapes: shift a data register by an immediate (1 to 8) or by a count in another data register, OR shift a WORD in memory by exactly 1 (lsl.w (a0)). There is no byte or long memory shift and no immediate destination. The destination of the register form is always a data register; you cannot shift an address register.

LSL slides the bits left, dropping a 0 into the bottom each step and pushing the top bit out into C and X. That is a multiply by 2^n, the everyday way to scale an index into a byte offset. This one, from the Enigma decompressor, multiplies a count by 8 to turn it into a table stride:

	lsl.b	#3,d4		; multiply by 8
s2disasm/s2.asm:2337 (EniDec, the Enigma decompressor)

See it move the bits in the shift visualizer (pick lsl), and see it set N, Z, C, and X in the flags demo.

Gotcha The size sets the boundary the top bit exits from. lsl.b pushes bit 7 out, lsl.w bit 15, lsl.l bit 31, and only the chosen field moves (a .b shift leaves the rest of the register alone). If the value could grow past the size you picked, the high bits just fall off into C and are gone, so shift in the size you actually mean.
Cheaper cousins Need signed-overflow flagging on the same left shift? That is ASL: identical bit movement, but it also sets V when the sign changes. For an unsigned scale where overflow does not matter, LSL is the plain choice. To divide instead of multiply, shift the other way with LSR.
Encoding

The register form is 1110 ccc d ss i 01 rrr: bits 11-9 the count or count-register, bit 8 the direction (1 = left), bits 7-6 the size, bit 5 immediate-vs-register count, bits 2-0 the data register. A count field of 0 means 8. The memory form is 1110 001 d 11 mmm rrr and always shifts by one.

LSR Logical Shift Right LSR.b/.w/.l #n,Dn  |  LSR.b/.w/.l Dn,Dn  |  LSR.w <ea>

Flags Xset Nset Zset Vclear Cset C and X take the last bit shifted out; N is always 0 (a 0 fills the top)
Cost 6 + 2n cycles 2 bytes same timing as LSL: 6 + 2n for a register shift, 8+ for a memory word shift-by-1.
Operands
Register form #n,DnDn,Dn count is an immediate 1 to 8, or a data register (mod 64)
Memory form (An)(An)+-(An)d(An)d(An,Xn)(xx)DnAn#n

Same two shapes as LSL: a register shifted by an immediate or a register count, or a word in memory shifted by 1. Address registers and immediates are never the destination.

LSR slides the bits right, dropping a 0 into the top each step and pushing the bottom bit out into C and X. That is an UNSIGNED divide by 2^n: it treats the value as a plain nonnegative number, which is why it suits counts and addresses but not signed positions. This one converts a byte offset into an object slot index by dividing by $40 (64 bytes per object):

	lsr.w	#6,d5			; divide by $40
s2disasm/s2.asm:9471 (SSAllocateObjectAfterCurrent)

Shift right by 6 is a divide by 64, turning the distance in bytes into a count of objects. Watch it in the shift visualizer (pick lsr) and see it set the flags in the flags demo.

Gotcha LSR is for UNSIGNED values only. Because it feeds 0 into the top, a negative number (top bit set) becomes a large positive one: lsr.b #1 of $FE (-2 signed) gives $7F (127), not -1. To halve a signed value use ASR, which copies the sign bit back in. This mix-up is a classic position-math bug.
Cheaper cousins Dividing a SIGNED value? Use ASR instead: same rightward movement, but it fills the top with a copy of the sign bit so negatives halve correctly. The difference between LSR and ASR is entirely in that fill bit. To multiply, shift left with LSL.
Encoding

The register form is 1110 ccc 0 ss i 01 rrr, the same as LSL but with the direction bit (bit 8) cleared to 0 for right. The memory form is 1110 001 0 11 mmm rrr, shifting a word by one. A count field of 0 means 8.

ASL Arithmetic Shift Left ASL.b/.w/.l #n,Dn  |  ASL.b/.w/.l Dn,Dn  |  ASL.w <ea>

Flags Xset Nset Zset Vset Cset the only shift that sets V: 1 if the sign bit changed at any point
Cost 6 + 2n cycles 2 bytes same as LSL; the extra V-computation is free. Memory word shift-by-1 is 8+.
Operands
Register form #n,DnDn,Dn count is an immediate 1 to 8, or a data register (mod 64)
Memory form (An)(An)+-(An)d(An)d(An,Xn)(xx)DnAn#n

Identical operand shapes to LSL. ASL and LSL even produce the same result bits; they differ only in the V flag, so the assembler keeps them as separate mnemonics.

ASL moves the bits exactly as LSL does (left, 0 into the bottom, top bit out into C and X), and produces the same result. The one thing it adds is V: it sets the overflow flag if the sign bit changed at any point during the shift, which is precisely a signed overflow (the value doubled past what a signed number of that size can hold). It is the shift to use in signed arithmetic when you need to know the result overflowed. This one, from the pseudo-random number generator, multiplies the RNG state by 5 using a shift-and-add (x*4 + x):

	asl.l	#2,d1
	add.l	d0,d1
	asl.l	#3,d1
s2disasm/s2.asm:3980-3982 (the RNG, multiply by 41 via shift-and-add)

Each asl.l is a multiply by a power of two; the interleaved adds build the odd factor. See it in the shift visualizer (pick asl and watch V light when the top bit flips), and see V set in the flags demo.

Gotcha ASL and LSL are NOT interchangeable just because they move bits the same way. If you shift a value that overflows its signed range, ASL sets V and LSL does not, so signed code that branches on V must use ASL. For pure bit packing where sign is meaningless, LSL is the honest choice and reads as "not arithmetic".
Cheaper cousins Do not need the V overflow flag? LSL is the same speed and same result, just without the signed-overflow detection. The right-shift signed divide is ASR. Note the asymmetry: ASL and ASR are NOT mirror images (ASL sets V, ASR clears it, and their fills differ).
Encoding

The register form is 1110 ccc 1 ss i 00 rrr: like LSL but with the type field (bits 4-3) 00 instead of 01, which is what tells the CPU to compute V. Direction bit 8 is 1 for left. The memory form is 1110 000 1 11 mmm rrr, a word by one.

ASR Arithmetic Shift Right ASR.b/.w/.l #n,Dn  |  ASR.b/.w/.l Dn,Dn  |  ASR.w <ea>

Flags Xset Nset Zset Vclear Cset fills the top with a copy of the sign bit, so a negative stays negative; V is 0
Cost 6 + 2n cycles 2 bytes same timing as the other shifts; a memory word shift-by-1 is 8+.
Operands
Register form #n,DnDn,Dn count is an immediate 1 to 8, or a data register (mod 64)
Memory form (An)(An)+-(An)d(An)d(An,Xn)(xx)DnAn#n

Same operand shapes as the other shifts. ASR is the right shift used for signed division; its distinguishing behavior is the fill bit, not the operand set.

ASR slides the bits right and, instead of a 0, copies the SIGN bit (the top bit) back into the top each step. That keeps a negative number negative, so ASR is a correct SIGNED divide by 2^n. It is the shift for signed position and velocity math, where a plain LSR would turn a small negative into a huge positive. This one halves a signed delta, and the disassembly even labels it:

	asr.w	#1,d0 ; Divide d0 by 2
s2disasm/s2.asm:15826 (the cloud-layer scroll math)

Because d0 here is a signed delta between camera and cloud positions, asr (not lsr) is what keeps a leftward delta leftward. See the sign copy fill the top in the shift visualizer (pick asr, try a byte like $FE) and the flags in the flags demo.

Gotcha ASR rounds toward negative infinity, not toward zero, which surprises people used to C integer division. asr.w #1 of -3 ($FFFD) gives -2 ($FFFE), not -1: the halving floors. If you need round-toward-zero behavior you must adjust before shifting. Also, do not reach for ASR on unsigned data (the sign copy would be wrong); that is LSR's job.
Cheaper cousins Dividing an UNSIGNED value? Use LSR, which pulls in 0 at the top instead of the sign bit. The two right shifts, ASR and LSR, differ ONLY in that fill: sign copy versus 0. To multiply a signed value, shift left with ASL.
Encoding

The register form is 1110 ccc 0 ss i 00 rrr: type field 00 (arithmetic), direction bit 8 0 (right). It shares ASL's 00 type but the direction bit picks the sign-extending right shift. The memory form is 1110 000 0 11 mmm rrr, a word by one.

2 · Rotates

A rotate is a shift whose fallen-off bit reappears at the other end. A shift loses bits: the one pushed off the edge lands in the carry and the vacated end fills with something new (a 0, or the sign). A rotate loses nothing. The bit that leaves one end comes back in at the far end, so after a full trip around you have the exact same value you started with. Nothing is created, nothing is destroyed, the bits just orbit.

There are two flavours, and the difference is the size of the ring. Plain rotate (ROL and ROR) turns a byte as an 8-bit ring: the bit that wraps is the one that reappears at the other end, and a COPY of it is also dropped into C so you can branch on it, but X is left completely alone. Rotate-with-extend (ROXL and ROXR) splices the X bit into the ring as a ninth cell: for a byte the ring is now 9 bits wide (8 data bits plus the X cell). The bit shoved out of the byte does not wrap straight back, it parks in X for one step, and the OLD contents of X enter the byte at the vacated end. C ends up equal to the new X.

Why does rotate-with-extend exist at all? To chain a rotate across more than one word. The 68000 rotates a register at a time, but sometimes you want to rotate a value wider than one register. You rotate the low word first with a plain shift or ROXL, and its top bit exits into X. Then you ROXL the high word, and that parked X bit slides into the high word's bottom, threading the carry across the boundary. The Kosinski decompressor does exactly this: lsr.w #1,d5 pushes a compression-stream bit into X, then roxl.w #1,d3 pulls that same X bit into the bottom of the count being built. X is the bridge between the two operations.

New here? · What does "through the X bit" actually mean

The tempting wrong picture is that "through X" means the bit is copied into X and ALSO stays in the byte, or that it is lost. Neither is true. Here is the dumb, literal mechanism for roxl.b #1: line up nine cells in a row, the 8 bits of the byte plus the X cell parked at the low end. Everything shifts one place to the left. The byte's top bit moves out into X. The old X value moves into the byte's bottom bit. That is the whole operation: a nine-cell ring turned by one, done once per count.

So the bit that leaves the byte is not lost and not duplicated. It parks in X for exactly one step, and on the NEXT rotate (or the next instruction that reads X, like ROXL on the high word) it re-enters at the other end. C is just a readable copy of whatever X becomes, so a branch can look at it. Plain ROL and ROR skip the X cell entirely: their ring is only 8 wide, the wrapped bit comes straight back around, and X keeps whatever it held before.

Register form #n,DnDn,Dn count is an immediate 1 to 8, or a data register (taken mod 64)
Memory form (An)(An)+-(An)d(An)d(An,Xn)(xx)DnAn#n

The same two shapes as the shifts: rotate a data register by an immediate (1 to 8) or by a register count, or rotate a WORD in memory by exactly 1. No byte or long memory form, no immediate destination, and never an address register.

ROL slides the bits left; the top bit wraps around into the bottom instead of being lost, and a copy of that wrapped bit lands in C. X is left alone. It is the tool for repacking bits: bring a field around to where you can mask it off. This one, from the scattered-ring animation, rotates a running counter to derive the current frame:

	rol.w	#7,d0
	andi.w	#3,d0
s2disasm/s2.asm:6105-6106 (ChangeRingFrame, the ring-spill animation)

The rotate brings the wanted bits down to the bottom, then andi keeps just those. Watch the wrap in the rotate visualizer (pick rol) and note X stays put; see N, Z, and C in the flags demo.

Gotcha ROL does NOT touch X, unlike every shift (which copies the exiting bit into both C and X). If your code relies on X for a following add-with-extend or a ROXL chain, a plain ROL will leave X holding whatever it had before. Use ROXL when you actually want the exiting bit to feed X.
Cheaper cousins Want the exiting bit to thread through X (to chain across words, or feed add-with-extend)? That is ROXL: a 9-bit rotate through the X cell. To rotate the other way, use ROR. If you just want to scale a value and do not need the bits to wrap, a shift (LSL) is the usual choice.
Encoding

The register form is 1110 ccc 1 ss i 11 rrr: bits 11-9 the count or count-register, bit 8 the direction (1 = left), bits 7-6 the size, bit 5 immediate-vs-register count, type field (bits 4-3) 11 which selects rotate-without-extend, bits 2-0 the data register. A count field of 0 means 8. The memory form is 1110 011 1 11 mmm rrr and rotates a word by one.

ROR Rotate Right ROR.b/.w/.l #n,Dn  |  ROR.b/.w/.l Dn,Dn  |  ROR.w <ea>

Flags Xunaff Nset Zset Vclear Cset C takes the bit that wrapped (the old bottom bit); X is UNAFFECTED; V is always 0
Cost 6 + 2n cycles 2 bytes same timing as ROL: register rotate by n, memory word rotate-by-1 is 8+.
Operands
Register form #n,DnDn,Dn count is an immediate 1 to 8, or a data register (mod 64)
Memory form (An)(An)+-(An)d(An)d(An,Xn)(xx)DnAn#n

Same two shapes as ROL: a data register rotated by an immediate or a register count, or a word in memory rotated by 1. Address registers and immediates are never the destination.

ROR slides the bits right; the bottom bit wraps around into the top, and a copy of it lands in C. X is left alone. Because a right rotate by k is the same as a left rotate by (width minus k), it is also a handy way to reorder bytes within a long. This one, from the vertical-scroll math, reorders a computed longword so the wanted word sits where a following move.w can grab it:

	ror.l	#8,d3
	move.w	d3,(a4)
s2disasm/s2.asm:18256-18257 (ScrollVerti, the vertical-scroll difference)

The ror.l #8 spins the long around by a whole byte, no bits lost, so the byte-ordered result lines up for the word store. Watch the wrap in the rotate visualizer (pick ror) and see the flags in the flags demo.

Gotcha Like ROL, ROR leaves X UNAFFECTED. Only C receives the wrapped bit. If you need the exiting bit to enter X (to chain into a high word), reach for ROXR instead. Also remember the count is mod 64 for a register count, so a rotate by a data register can wrap more than once around the value.
Cheaper cousins Need the exiting bit threaded through X for a multi-word chain? Use ROXR, the 9-bit right rotate through the X cell. To rotate the other direction, use ROL. For an unsigned divide (bits fall off, do not wrap), the shift LSR is what you want.
Encoding

The register form is 1110 ccc 0 ss i 11 rrr, the same as ROL but with the direction bit (bit 8) cleared to 0 for right; type field (bits 4-3) is 11 for rotate-without-extend. The memory form is 1110 011 0 11 mmm rrr, a word by one. A count field of 0 means 8.

Register form #n,DnDn,Dn count is an immediate 1 to 8, or a data register (mod 64)

Same operand shapes as the other rotates. What sets ROXL apart is not its operands but the ring: the X bit is spliced in as a ninth cell (for a byte), so it both reads and writes X.

ROXL slides the bits left through the X cell: the top bit of the operand moves into X, the OLD X moves into the bottom, and C ends up equal to the new X. For a byte that is a 9-bit ring (8 data bits plus X). The whole point is chaining a rotate across more than one register: rotate the low part, its top bit exits into X, then ROXL the high part so that parked bit slides into its bottom. The Kosinski decompressor threads a compression-stream bit exactly this way, feeding X from an lsr into a count being built:

	lsr.w	#1,d5		; bit shifted out goes into X
	roxl.w	#1,d3		; get repeat count bit by shifting X in
s2disasm/s2.asm:2621,2629 (KosDec, the Kosinski decompressor)

The lsr pushes a bit into X, then roxl pulls that same X bit into the bottom of d3: X is the bridge between the two instructions. See a bit park in X and re-enter in the rotate visualizer (pick roxl and watch the X cell across runs), and see N, Z, C set in the flags demo.

Gotcha ROXL reads the CURRENT X, so whatever set X last (an add, a shift, an lsr, a previous ROXL) decides the bit that enters at the bottom. Set X deliberately before the chain: to start a multi-word rotate cleanly, clear X first (for instance with an and that clears C and X, or arrange the feeding shift). A stale X quietly corrupts the low bit.
Cheaper cousins Do not need the bit to pass through X? Plain ROL is a simpler 8-bit wrap that leaves X alone and just copies to C. To thread through X in the other direction, use ROXR. If you want the exiting bit in both C and X but not wrapped back in, that is a shift (LSL).
Encoding

The register form is 1110 ccc 1 ss i 10 rrr: like ROL but with the type field (bits 4-3) 10 instead of 11, which selects rotate-WITH-extend (the X cell in the ring). Direction bit 8 is 1 for left. The memory form is 1110 010 1 11 mmm rrr, a word by one. A count field of 0 means 8: for ROXL that special count-0 case sets C to the current X and leaves the operand and X unchanged.

Flags Xset Nset Zset Vclear Cset X PARTICIPATES: it is the ninth cell of the ring; C ends equal to the new X; V is 0
Cost 6 + 2n cycles 2 bytes same timing as ROXL; a memory word rotate-by-1 is 8+.

Register form #n,DnDn,Dn count is an immediate 1 to 8, or a data register (mod 64)

Same operand shapes as ROXL. It is the mirror of ROXL: the ninth cell (X) sits at the top of the ring instead of the bottom, so it too reads and writes X.

ROXR is ROXL going the other way: the bottom bit of the operand moves into X, the OLD X moves into the top, and C ends equal to the new X. For a byte that is the same 9-bit ring, just turned right. It chains a rotate across words in the high-to-low direction: rotate the high word first, its bottom bit exits into X, then ROXR the low word so that bit enters at the top, threading a 32-bit rotate through two 16-bit rotates. No clean real example of ROXR appears in the Sonic 2 disassembly (its sibling ROXL is used for the same chaining), so here is the two-word rotate-right idiom spelled out:

And count 0 (a register count of exactly 0) does not rotate at all: it sets C to the current X and leaves the operand and X unchanged.
Cheaper cousins Do not need X in the ring? Plain ROR is the simpler 8-bit right wrap that leaves X alone. To thread through X in the other direction, use ROXL. To drop bits off the end instead of wrapping them, a shift (LSR or ASR) is the tool.
Encoding

The register form is 1110 ccc 0 ss i 10 rrr: type field (bits 4-3) 10 (rotate-with-extend), direction bit 8 0 (right). The memory form is 1110 010 0 11 mmm rrr, a word by one. A count field of 0 means 8; a register count of exactly 0 sets C from the current X and leaves everything else alone.

3 · Single-bit operations

These four instructions reach into ONE numbered bit and act on it by name, where the shifts and rotates move a whole register at once. You name a bit (bit 5 of a status byte, the priority bit of a render flag) and the CPU tests it, and then BSET sets it to 1, BCLR clears it to 0, BCHG flips it, or BTST stops after the test. All four do the test FIRST: before touching anything they read the current value of that bit and record it, which is how flag bytes get read one bit at a time. The single most useful fact about them is what they leave in Z.

Z reports the OLD value of the tested bit, and it reports it INVERTED: Z=1 means the bit WAS clear (0), Z=0 means the bit WAS set (1). So btst #n,x then beq branches exactly when the bit was 0, and bne branches when it was 1. That is the everyday joypad test: btst #button_A,(Ctrl_1_Press) then beq skips ahead when A was NOT pressed. Contrast this with masking on the Math & Logic page: AND, OR, and EOR edit a WHOLE byte through a mask and set Z from the whole result; these name a single bit and additionally report that one bit's prior state. None of the four touch X, N, V, or C: only Z moves.

New here? · Why is Z "backwards" (Z=1 when the bit was 0)

The intuitive guess is that Z copies the bit: bit set, so Z set. That guess is wrong and it will make every branch you write go the wrong way. Throw it out. The rule is the opposite: Z is the complement of the tested bit's OLD value. Z=1 when the bit was 0, Z=0 when the bit was 1.

The name Z stands for "zero", and it helps to read btst #n,x as "compute the value of just bit n, was that ZERO?" If the bit was 0, the tested value is zero, so Z=1, and beq (branch if equal, meaning branch if Z=1) fires. If the bit was 1, the value is nonzero, Z=0, and bne fires. So the reliable reading is: btst then beq takes the branch when the bit was CLEAR; btst then bne takes it when the bit was SET. The op you pick after (set/clear/flip) does not change this: BSET, BCLR, and BCHG all set Z from the OLD bit exactly as BTST does, THEN change the bit.

Play with all four below. Pick an op, set a byte and a bit number 0 to 7, and run it: the toy marks the selected bit, shows the byte after (the changed bit highlighted), and puts Z next to it so you can watch Z come out as the COMPLEMENT of the bit's old value. The byte is held across runs, so a bset then a later bclr both act on the current value.

BTST Test a Bit BTST #n,<ea>  |  BTST Dn,<ea>  (byte in memory, or long in a Dn)

Flags Xunaff Nunaff Zset Vunaff Cunaff ONLY Z moves: Z = the complement of the tested bit's value (Z=1 if the bit was clear)
Cost 4 to 8 cycles 2 to 4 bytes immediate bit number on a register is 6c/4b; on memory 4c/2b plus the operand fetch.
Full timing and size

BTST is the cheapest of the four because it never writes back. A register long-bit test is 6c; a byte in memory adds the addressing-mode fetch. c = CPU cycles, b = bytes; Genesis memory adds wait states.

Operands
Bit number #nDn immediate, or a data register; taken mod 8 for a byte operand, mod 32 for a Dn
Destination Dn(An)(An)+-(An)d(An)d(An,Xn)(xx)An

The bit number is either an immediate #n or a value in a data register Dn (two encodings of the same instruction). The destination is a byte in memory OR a long in a data register: a Dn destination tests one of its 32 bits (bit number mod 32), any memory destination tests one of the 8 bits of a byte (bit number mod 8). You cannot BTST an address register.

BTST reads a named bit and sets Z from it, without changing anything: Z=1 if the bit was clear, Z=0 if it was set, so a following beq branches when the bit was 0. This is the canonical joypad button test, straight from the pause handler:

	btst	#button_A,(Ctrl_1_Press).w	; is button A pressed?
	beq.s	Pause_ChkBC		; if not, branch
s2disasm/s2.asm:1600-1601 (the pause handler, slow-motion cheat)

The button-A bit of the freshly pressed controls is tested; beq takes the branch (skips the A action) when the bit was 0, that is, when A was NOT pressed. Try it in the bit-poke toy (pick btst) and watch Z come out as the complement of the bit; see how Z drives a branch in the flags demo.

Gotcha Z is INVERTED, and this is the classic bug. btst then beq branches when the bit was CLEAR (0), not when it was set; bne branches when it was SET. If you read it as "Z equals the bit" every branch inverts. Say it out loud: beq after btst means "the bit was zero".
Encoding

Two forms. The register-bit-number form is 0000 rrr 100 mmm ddd: bits 11-9 the data register holding the bit number, bits 5-0 the destination effective address. The immediate form is 0000 1000 00 mmm ddd followed by an extension word whose low byte is the bit number. Bits 7-6 (00) select the "test" operation, distinguishing BTST from BCHG/BCLR/BSET.

BSET Test a Bit and Set BSET #n,<ea>  |  BSET Dn,<ea>  (byte in memory, or long in a Dn)

Flags Xunaff Nunaff Zset Vunaff Cunaff ONLY Z moves: Z = the complement of the bit's PRIOR value (Z=1 if it was clear), then the bit is set to 1
Cost 8 to 12 cycles 2 to 4 bytes register form 8c/2b (12c with an immediate bit number); memory adds the write-back and fetch.
Operands
Bit number #nDn immediate, or a data register; mod 8 for a byte, mod 32 for a Dn
Destination Dn(An)(An)+-(An)d(An)d(An,Xn)(xx)An

Same shape as BTST, but the destination is now read AND written. A Dn destination pokes one of 32 bits (mod 32); a memory destination pokes one of the 8 bits of a byte (mod 8). No immediate destination, no address register.

BSET first tests the bit (setting Z from its OLD value, Z=1 if it was clear) and THEN forces it to 1. It is how a status flag gets raised by name. This one raises the player's "in-air" bit at the top of a jump:

	bset	#status.player.in_air,status(a1)	; set "in-air" bit
s2disasm/s2.asm:5496 (the jump routine, marking the player airborne)

The named bit of the status byte becomes 1 regardless of what it held; Z is left reflecting the old value (usually ignored here, since the code just wants the bit set). Poke it in the bit-poke toy (pick bset, note Z comes from the OLD bit while the byte changes), and see Z in the flags demo.

Gotcha Z reflects the bit's state BEFORE the set, not after. After bset the bit is always 1, but Z tells you whether it had ALREADY been 1 (Z=0) or was newly raised (Z=1). That is genuinely useful for test-and-set locks, but if you branch on Z expecting the new state you will be a step behind.
Cheaper cousins Setting SEVERAL bits of a byte at once? That is a whole-byte OR with a mask on the Math & Logic page: ori.b #mask raises every 1 in the mask in one instruction. BSET is the single-named-bit tool that ALSO reports the old value in Z. To clear a bit instead of setting it, use BCLR; to flip it, BCHG.
Encoding

The register-bit-number form is 0000 rrr 111 mmm ddd; the immediate form is 0000 1000 11 mmm ddd plus an extension word holding the bit number. Bits 7-6 (11) select "set", the field that distinguishes BSET from BTST (00), BCHG (01), and BCLR (10).

BCLR Test a Bit and Clear BCLR #n,<ea>  |  BCLR Dn,<ea>  (byte in memory, or long in a Dn)

Flags Xunaff Nunaff Zset Vunaff Cunaff ONLY Z moves: Z = the complement of the bit's PRIOR value (Z=1 if it was already clear), then the bit is set to 0
Cost 10 to 14 cycles 2 to 4 bytes register form 10c/2b (14c immediate); BCLR is 2c dearer than BSET on a register. Memory adds fetch + write.
Operands
Bit number #nDn immediate, or a data register; mod 8 for a byte, mod 32 for a Dn
Destination Dn(An)(An)+-(An)d(An)d(An,Xn)(xx)An

Same operands as BSET; the destination is read and written. Dn clears one of 32 bits (mod 32), memory one of a byte's 8 bits (mod 8). No immediate destination, no address register.

BCLR tests the bit (setting Z from its OLD value, Z=1 if it was already clear) and THEN forces it to 0. It is how a flag gets lowered by name. This one clears the title-card bit out of the game mode as the level proper begins:

	bclr	#GameModeFlag_TitleCard,(Game_Mode).w ; clear $80 from the game mode
s2disasm/s2.asm:5078 (Level_StartGame, leaving the title-card sequence)

The named bit of Game_Mode becomes 0, dropping the level out of its title-card phase; Z is left reflecting the old bit. See it in the bit-poke toy (pick bclr) and watch Z reflect the OLD bit while the byte drops that bit; check Z in the flags demo.

Gotcha Like BSET, Z is the bit's OLD value inverted, read BEFORE the clear: after bclr the bit is always 0, but Z tells you whether it had been clear already (Z=1) or you actually just cleared a set bit (Z=0). Also note BCLR must read-modify-write, so it is a touch slower than BTST and even BSET.
Cheaper cousins Clearing SEVERAL bits at once? That is a whole-byte AND with an inverted mask on the Math & Logic page: andi.b #~mask knocks out every named bit in one go. BCLR is the single-named-bit tool that ALSO reports the old value in Z. To set a bit, use BSET; to flip it, BCHG.
Encoding

The register-bit-number form is 0000 rrr 110 mmm ddd; the immediate form is 0000 1000 10 mmm ddd plus an extension word with the bit number. Bits 7-6 (10) select "clear". Clearing a bit is implemented as a read, an AND with the inverted bit mask, and a write-back, which is why it costs a little more than BTST.

BCHG Test a Bit and Change BCHG #n,<ea>  |  BCHG Dn,<ea>  (byte in memory, or long in a Dn)

Flags Xunaff Nunaff Zset Vunaff Cunaff ONLY Z moves: Z = the complement of the bit's PRIOR value (Z=1 if it was clear), then the bit is flipped
Cost 8 to 12 cycles 2 to 4 bytes register form 8c/2b (12c immediate), like BSET; memory adds the read-modify-write and fetch.
Operands
Bit number #nDn immediate, or a data register; mod 8 for a byte, mod 32 for a Dn
Destination Dn(An)(An)+-(An)d(An)d(An,Xn)(xx)An

Same operands as BSET and BCLR; read and written. Dn flips one of 32 bits (mod 32), memory one of a byte's 8 bits (mod 8). No immediate destination, no address register.

BCHG tests the bit (setting Z from its OLD value, Z=1 if it was clear) and THEN flips it: a 1 becomes 0, a 0 becomes 1. It is the toggle, the one-instruction way to alternate a state each time through. This one flips the region bit, swapping the console between Japanese and international:

	bchg	#7,(Graphics_Flags).w
s2disasm/s2.asm:4644 (the region-toggle cheat, Japanese vs international)

Each time the code runs, bit 7 of Graphics_Flags flips to the other state, toggling the trademark symbols and the Tails/Miles name; Z reflects the value before the flip. Try the toggle in the bit-poke toy (pick bchg and run it twice to see the bit come back), and see Z in the flags demo.

Gotcha BCHG is the one whose result you genuinely cannot predict from the op alone: the bit ends up as the OPPOSITE of whatever it was, so run it twice and you are back where you started. Z (the OLD value inverted) is therefore the only way to know which way it just flipped. For a whole-byte toggle of several bits at once, that is EOR with a mask.
Cheaper cousins Toggling SEVERAL bits at once? That is a whole-byte EOR with a mask on the Math & Logic page: eori.b #mask flips every named bit in one instruction. BCHG is the single-named-bit toggle that ALSO reports the old value in Z. To force a bit one way instead of flipping, use BSET (to 1) or BCLR (to 0).
Encoding

The register-bit-number form is 0000 rrr 101 mmm ddd; the immediate form is 0000 1000 01 mmm ddd plus an extension word with the bit number. Bits 7-6 (01) select "change". Like BCLR and BSET it is a read-modify-write; the change is an EOR with the bit mask.
