Instructions: Math & Logic

This is where the CPU actually computes: adding and subtracting, multiplying and dividing, comparing, and working on bits with AND, OR, and their kin. Almost every one of these instructions does two jobs at once, it produces a value AND it reports on that value through the condition codes, which is how the very next branch decides where to go. This page covers the arithmetic, compare, and logic families; the shifts, rotates, and single-bit instructions live on the next page. Every example is real, pulled from the Sonic 2 disassembly and re-verified line by line.

1 · How instructions set the flags

Before any single instruction, one idea ties this whole page together. Nearly every arithmetic and logic instruction here does its computation AND, as a side effect, updates the five operand condition codes, X N Z V C, to describe the result it just produced. On their own those flags do nothing. Their entire purpose is that the very next instruction, usually a conditional branch, reads them and decides what to do. So on the 68000 you rarely test a value with a separate step: you add, subtract, or compare, and the flags already say whether the result was zero, negative, or overflowed.

New here? · What are the X N Z V C flags?

The CPU keeps five status bits called the condition codes, or flags, packed into the low byte of the status register. Most instructions overwrite them as a side effect of whatever they just computed: N (the result was negative, its top bit set), Z (the result was zero), V (a signed overflow happened, the sign came out wrong), C (an unsigned carry or borrow out of the top bit), and X (the extend bit, a saved copy of the carry that multi-precision math reuses). The Programming Model page covers the status register in full; here what matters is the pattern below.

Each instruction's Flags row shows which bits it touches: set means updated from the result, clear means forced to zero, unaff. means left exactly as it was. Watch for the instructions that touch NO flags at all (ADDA, SUBA, and the quick forms into an address register): a branch after one of those is reading whatever the last real flag-setter left behind, which is a classic bug.

New here? · Signed or unsigned: which flag do I read?

The hardware sets all five flags mechanically from the bits; it has no idea whether you meant a number to be signed or unsigned. Your choice of branch decides which flags to believe. If you are treating values as unsigned (0 and up, like a ring count or an address), you branch on the carry: bcc / bcs, bhi / bls. If you are treating them as signed (negatives allowed, like a velocity), you branch on the N and V pair: bpl / bmi, bgt / blt. Same instruction, same flags, two readings. This is why the demo below shows both the unsigned and the signed value of every result.

Play with it. Pick an instruction, set the two operands, and run it: the destination register and all five flags update, with the note spelling out both readings of the result.

With that model in hand, the rest of the page is the instructions themselves, grouped by what they do. Each entry gives its exact flag behavior, its cost, which operands it accepts, a real Sonic example, and, where the lesson is something other than the flags, its own small toy.

2 · Adding and subtracting

The bread and butter. Every frame, positions gain velocities, timers count down, scores climb, and pointers step through tables, and nearly all of it is ADD and SUB. The family looks large because the 68000 gives each job a specialized, cheaper form: a quick version for tiny constants (ADDQ), an immediate version for larger ones (ADDI), and an address-register version that stays out of the way of the flags (ADDA). Learn the plain ADD once and the rest are variations on when to reach for which.

New here? · Why are there four different ways to add?

It looks redundant, but each form is a specialization that saves space or stays out of the flags' way. Plain ADD is the general case: add any value to a data register. ADDQ is a shortcut for tiny constants (1 to 8) that packs the whole instruction into one word. ADDI is for larger constants added in place. And ADDA adds to an ADDRESS register, which the CPU deliberately treats differently: no flags, no byte size.

The good news is you rarely choose consciously. Write add.w d0,a1 and the assembler quietly encodes ADDA because the destination is an address register; write a 1-to-8 constant and a good assembler reaches for ADDQ. The forms exist so the machine has a cheap encoding for each common shape; knowing they are all "add" is enough to read the code. The same story repeats for subtract (SUB, SUBQ, SUBI, SUBA).

ADD always has one data register as an operand; the grid shows what the OTHER operand (the effective address) may be, in each of the two directions. † an address register source is fine in .w/.l but not .b (a byte cannot come from an An). Adding into an address register (add.w d0,a0) is really ADDA, which the assembler swaps in for you; a constant or PC-relative location is not writable, so add.w d0,#5 is refused.

ADD adds a value to a data register (or a register's value into memory) and reports fully on the result through all five flags. It is the workhorse of the page: how a position gains a velocity, a counter climbs, an index advances. Because it sets N, Z, V, and C from the sum, you can branch on the result immediately, no separate compare needed. Reach for plain ADD when the amount is a runtime value in a register or memory; when it is a small constant, its cheaper cousins below win.

This is the exact position-plus-velocity step, from Obj5F (a moving object), and it shows the reason ADD sets flags: the very next lines branch on the new position to delete the object once it drifts off screen.

	add.w	d1,x_pos(a0)
	add.w	d0,y_pos(a0)
	cmpi.w	#0,x_pos(a0)
	blt.w	JmpTo_DeleteObject
s2disasm/s2.asm:9641-9644 (Obj5F, a moving object)

The two ADDs move the object; the cmpi/blt that follows reacts to where it landed. To watch ADD set the flags for yourself, run it in the flags demo at the top: try $8000 + $8000 and see V light (two negatives summing to a positive), or $FFFF + $0001 and see C light (unsigned carry off the top).

Gotcha Size matters for the flags. add.w sets N, Z, V, and C from the 16-bit result and leaves the upper half of the destination alone, so a carry out of bit 15 sets C even though the full 32-bit register did not overflow. If you mean 32-bit math, use add.l. And remember ADD into an address register is silently ADDA, which sets no flags at all.
Cheaper cousins Adding a small constant (1 to 8) to a data register or memory? ADDQ does it in one word (4c vs ADDI's 8c). A larger constant is ADDI. Advancing an address register is ADDA, and if the offset is fixed and you want the flags left alone, LEA (lea 4(a0),a0) does the same bump flag-free.
Encoding

ADD is 1101 rrr ooo mmm rrr: bits 15-12 are 1101, bits 11-9 the data register, bits 8-6 the opmode (which also picks the direction, 000/001/010 = <ea>+Dn→Dn for byte/word/long, 100/101/110 = Dn+<ea>→<ea>), and bits 5-0 the effective address.

add.w d0,d1 assembles to $D240: register d1 (001), opmode word-into-Dn (001), source d0 (mode 000, register 000).

The destination is always an address register; the source may be anything. ADDA is what the assembler quietly uses whenever you write an ADD whose destination is an An, which is why you rarely type adda yourself but see its behavior constantly.

ADDA advances an address register by a value, and unlike ADD it changes NO condition codes. That is the whole reason it exists as a separate instruction: you routinely need to step a pointer in the middle of a comparison, and you do not want that step to disturb the flags the next branch is going to read. Reach for ADDA (or let the assembler reach for it) to move a pointer by a runtime amount; for a fixed offset, LEA reads cleaner and is equally flag-free.

A real double-indexed pointer walk, from the special-stage track renderer: two ADDAs in a row advance a1 to the exact animation frame, and neither disturbs the flags around them.

	adda.w	(a1,d3.w),a1	; Add offset so a1 points to animation data
	adda.w	d2,a1	; Offset into current animation frame
s2disasm/s2.asm:7045-7046 (SSTrack_Draw)
Gotcha Two silent traps in one instruction. First, ADDA sets no flags, so a beq right after it reads a stale result. Second, adda.w sign-extends its word source to 32 bits, so adding #$8000 does not add 32768, it adds -32768 and moves the pointer backward. When in doubt about the high bits, use adda.l.
Cheaper cousins Bumping a pointer by a fixed compile-time offset? LEA (lea 4(a0),a0) is the same 8 cycles, reads more clearly, and is equally flag-free. ADDA wins when the amount is a runtime value in a register or memory.
Encoding

ADDA is 1101 rrr ooo mmm rrr, sharing ADD's 1101 prefix; the opmode bits 011 (word) and 111 (long) mark the address-register destination in bits 11-9. adda.w a0,a1 = $D2C8.

The source is always the immediate constant; the destination is a data register or memory, never an address register. addi.w #4,a0 is refused, that is ADDA's job (adda.w #4,a0).

ADDI adds a compile-time constant straight to a destination, and it earns its place by doing the one thing plain ADD cannot: adding an immediate directly to MEMORY. ADD's memory-writing form (add Dn,<ea>) requires its source to be a register, so there is simply no add #n,(memory) instruction; ADDI is how you bump a RAM variable by a constant without first loading it into a register. Its only real competition is ADDQ, which is smaller but caps at 8, so ADDI takes over for any constant of 9 or more. (Adding a constant into a data register can go through either ADD or ADDI; the memory case is where ADDI is the only option.) The oil-slide friction routine below is a textbook use: the constant $40 is well past ADDQ's range, and the destination is a field in RAM.

From the oil-slide friction code: a guarded nudge that only fires when the object is not yet at its target speed.

	cmp.b	d0,d1
	bge.s	+
	addi.w	#$40,inertia(a1)
s2disasm/s2.asm:5579-5581 (the oil-slide friction routine)

The cmp/bge gate decides whether to nudge; the ADDI adds $40 to the object's inertia field in RAM. To watch ADDI set flags, use the flags demo (pick add; an immediate adds the same way a register value does).

Gotcha If your constant is 1 to 8 and the destination is data-alterable, writing ADDI wastes a word and a couple of cycles versus ADDQ; most assemblers will NOT silently switch it for you the way they switch ADD to ADDA, so the wasteful encoding is exactly what you asked for.
Cheaper cousins Constant of 1 to 8 into a data-alterable destination? ADDQ, always (2 bytes and 4 cycles versus 4 bytes and 8). Value already in a register? plain ADD. Targeting an address register? ADDA.
Encoding

ADDI is 0000 0110 ss mmm rrr followed by the immediate word(s): bits 7-6 are the size, bits 5-0 the destination effective address. addi.w #$40,d1 = $0641 then the immediate word $0040.

The source is a tiny quick constant, 1 to 8 only, encoded in three bits of the opcode itself (which is why there is no extension word). † an An destination is legal in .w/.l only, sets no flags, and updates the whole register. Ask for #9 or more and the assembler cannot encode ADDQ, so it falls back to ADDI.

ADDQ adds a constant from 1 to 8 in a single opcode word, no extension needed, making it the cheapest way on the chip to nudge a value up by a little. This is the everyday increment: bump a counter, step an index, advance a pointer by a few bytes. If the amount fits in 1 to 8, ADDQ is what you want; the only catch is the range wall.

The classic guarded increment, from ring collection: bump the ring count by one, but only if it is below the 999 cap.

	cmpi.w	#999,(Ring_count).w	; does the player 1 have 999 or more rings?
	bhs.s	+			; if yes, skip the increment
	addq.w	#1,(Ring_count).w	; add 1 to the ring count
s2disasm/s2.asm:24894-24896 (CollectRing)

The cmpi/bhs is the guard; the ADDQ is the increment. This exact shape, compare against a cap, branch away if reached, otherwise ADDQ, is everywhere in the engine.

Gotcha Two surprises. The amount is 1 to 8, full stop; addq #9 is not an ADDQ and the assembler will quietly encode ADDI instead (bigger and slower). And addq #n,An costs 8 cycles, not 4, and sets NO flags, one of several places where an address-register operation behaves differently from the data-register one right next to it.
Cheaper cousins ADDQ is the floor for adding 1 to 8; nothing beats it. Going the other way: 9 or more is ADDI, and a value already in a register is plain ADD. Note addq #4,An is a common flag-preserving pointer bump, its selling point over add.
Encoding

ADDQ is 0101 ddd 0 ss mmm rrr: bits 11-9 are the data field (001-111 = 1 to 7, 000 = 8), bit 8 is 0 (marking ADDQ, not SUBQ), bits 7-6 the size, bits 5-0 the destination. addq.w #1,d0 = $5240; addq.w #8,d0 = $5040.

Same as ADD: one operand is a data register, the grid is the other. † An source is legal except in .b. Subtracting into an address register is SUBA.

SUB subtracts and sets all the flags, and the one thing to fix in your head is the direction: sub.w <ea>,Dn computes Dn - <ea>, so the source is the amount taken away, not the thing being reduced. Read left to right it looks backward. Its most common job is computing a gap, "how far is this from that", and then branching on the sign of the result to pick a direction. Reach for plain SUB with a runtime value; small constants go to SUBQ.

This single block from the water-level animation shows SUB, NEG, and ADD working together, and it is the clearest "compute a gap, branch on its sign" example in the game:

	move.w	(Water_Level_3).w,d0
	sub.w	(Water_Level_2).w,d0
	beq.s	++	; rts
	bcc.s	+
	neg.w	d1
+	add.w	d1,(Water_Level_2).w
s2disasm/s2.asm:5356-5362 (DynamicWater)

SUB computes the gap and sets the flags; beq handles "already there", bcc reads the sign to decide which way to move, NEG flips the step when needed, and ADD applies it. Run SUB in the flags demo to watch a borrow set C: try $0000 - $0001.

Gotcha The operand order is the reverse of how you might say it out loud. sub.w a,b is b - a, not a - b. Get it backward and the sign of every branch that follows is inverted. When you only want the flags and not the difference, that is CMP.
Cheaper cousins Subtracting 1 to 8 from a data-alterable destination? SUBQ (4c vs 8c). From an address register? SUBA (or a negative-offset LEA). Only need the flags, not the result? CMP.
Encoding

SUB is 1001 rrr ooo mmm rrr, the same layout as ADD with prefix 1001. Opmode 000/001/010 = Dn-<ea>→Dn (b/w/l), 100/101/110 = the memory direction. sub.w d0,d1 = $9240.

A single operand, the thing to negate, which must be data-alterable. You cannot NEG an address register; move it into a data register first, or subtract it from zero another way.

NEG replaces a value with its two's-complement negative: 0 - value, flipping the sign. It is how a positive speed becomes a leftward one, a downward step becomes an upward one. Unlike CLR or MOVE, NEG sets the carry and extend for any nonzero input, which matters if it is part of a multi-word negate; and negating the most-negative value ($8000 as a word) sets V, because its positive counterpart does not fit.

A real sign-flip, from the Buzzer badnik's main routine: reverse the horizontal velocity and mirror the sprite at the same moment, so it turns around.

	neg.w	x_vel(a0)
	bchg	#render_flags.x_flip,render_flags(a0)
s2disasm/s2.asm:24527-24528 (Obj28, the Buzzer badnik)
Gotcha NEG is arithmetic (two's complement), not NOT (one's complement): negating $0005 gives $FFFB (-5), while NOT gives $FFFA. And negating the most-negative value returns it unchanged with V set, the one input NEG cannot actually flip.
Cheaper cousins If the value came out of a subtraction, reordering the operands of that SUB often avoids the NEG entirely (compute the gap the other way round). To flip every bit rather than the sign, that is NOT.
Encoding

NEG is 0100 0100 ss mmm rrr: bits 7-6 the size, bits 5-0 the destination. neg.w d0 = $4440, neg.l d0 = $4480.

3 · Comparing and testing

These four instructions exist to set the flags and nothing else. A compare is a subtraction that throws its result away and keeps only the condition codes, so the branch on the next line can act on "were these equal / which was bigger / is this zero" without disturbing either value. They are the setup half of nearly every if and every loop in the engine.

New here? · Why subtract if you throw the answer away?

Because subtraction is exactly the operation whose flags answer "how do these two compare." If a - b comes out zero, they were equal (Z set). If it borrowed, a was smaller (C set, for unsigned). Its sign tells you which was bigger for signed values (N and V). So the CPU does not need a separate "compare" circuit: it runs a subtract, sets the flags from it, and simply does not store the difference. That is all CMP is. TST is the same idea against a fixed zero.

The payoff is that you keep both operands intact. cmp.w d1,d0 tells you how d0 relates to d1 and leaves both untouched, so the following code can still use them. If you actually wanted the difference, you would use SUB instead.

New here? · CMP, CMPA, CMPI, TST: which one do I use?

They are one idea (subtract, keep only the flags) aimed at different operands, and the suffix tells you which. CMP compares against a DATA register. CMPA compares against an ADDRESS register (the A). CMPI compares against an IMMEDIATE constant (the I), and can reach a value in memory directly. TST is the special case of comparing against zero, the most common test of all, so it gets its own tiny instruction.

Mostly the assembler picks for you from what you wrote: cmp #5,d0 becomes CMPI because the source is a constant, and comparing against an An becomes CMPA. The one you reach for on purpose is TST, whenever the question is simply "is this zero or negative."

Cheaper cousins Comparing against zero? TST (4 cycles and no immediate word, versus cmpi #0's 8). Comparing against an immediate? CMPI. Against an address register? CMPA.

4 · Multiplying and dividing

The 68000 can multiply and divide in hardware, which was a real luxury in 1988, but the operations are slow and full of sharp edges. A multiply takes tens of cycles; a divide well over a hundred, the most expensive instructions on the whole chip. Both work on 16-bit operands and produce a wider result, and divide has two ways to go wrong that you must guard against yourself.

New here? · Why are these so much slower than add?

Because the CPU has no single-step multiply or divide circuit; it works them out the long way, one bit at a time, the same way you would do long multiplication or long division on paper. A 16-bit multiply is up to sixteen add-and-shift steps, so it lands around 40 to 70 cycles depending on the number's bit pattern. A divide is even more work: DIVU runs up to about 140 cycles and DIVS, the signed version, up to about 158, making it the single slowest instruction the 68000 has.

The timing is data-dependent, which is unusual: the exact cycle count depends on the values, not just the instruction. This is why performance-sensitive code avoids these when it can, reaching for a shift when it only needs to multiply or divide by a power of two (that is the cheaper cousin on every entry here). When you genuinely need a general multiply or divide, they are there, just budget for the cost.

New here? · What does "16 by 16 into 32" mean, and what is signed vs unsigned here?

Two 16-bit numbers can multiply to almost 32 bits (two values near 65535 multiply to over four billion), so the product needs the WHOLE register, not just its low half. That is why MULU and MULS write all 32 bits of the destination and you read the answer as a long. Divide runs the other way: it takes a 32-bit number and a 16-bit one and hands back a 16-bit quotient plus a 16-bit remainder, packed together into the one register.

The U and S endings pick how the bits are read. MULU and DIVU are UNSIGNED (every value is zero or positive); MULS and DIVS are SIGNED (a top bit set means negative). They give different answers the moment a top bit is involved, so you use the pair that matches how you mean the numbers: unsigned for counts and sizes, signed for anything that can go negative, like a velocity.

MULU multiplies two unsigned 16-bit words and writes the full 32-bit product back into the data register. The important word is full: two 16-bit values can multiply up to nearly 32 bits, so the result uses the whole register, and the upper half, previously ignored, now holds real data. Code that keeps treating it as a word silently drops the top half. Also: only the low word of the destination is used as the input factor; whatever was in its high word is overwritten by the result.
Cheaper cousins Multiplying by a power of two? A left shift is far cheaper, roughly 6 to 8 cycles against MULU's 38 to 70. Sonic's own code branches on exactly this: it uses lsl when the object size is a power of two and MULU otherwise. Shifts live on the Shifts & Bits page (coming).

5 · Bitwise logic

The last family works on individual bits rather than on numbers as a whole. AND, OR, and EOR combine two bit patterns bit by bit; NOT flips every bit of one. This is how you pack several yes/no flags into a single byte and then read or change them: mask bits off with AND, switch them on with OR, flip them with EOR. Every one of these also sets N and Z from the result and forces V and C to zero, so you can branch on whether the outcome was zero.

New here? · What is a bit mask, and why pack flags into one byte?

Registers and memory are precious, and most game state is yes/no: on the ground or not, facing left or right, invincible or not. Rather than spend a whole byte on each, the engine packs eight of these into the eight bits of a single "flag byte." Each bit is one flag.

A mask is a second bit pattern that says which bits you care about. Pair a mask with the right logic instruction and you edit just those bits: AND with a mask clears the bits that are 0 in it, OR sets the bits that are 1, EOR flips them, and the bits you did not name are left exactly alone. That is why these three show up constantly on status bytes, hardware registers, and the art-tile words that carry flip and palette flags.

Teaching example · a flag byte and its three edits

Toy bits, to make masking concrete. Say a status byte packs three flags, and right now the player is on the ground, not jumping, not invincible:

	status = %00000001	; bit 0 = on-ground, bit 1 = jumping, bit 2 = invincible

Three edits, each touching only the bit its mask names:

or.b #%00000010,status  sets jumping on  → %00000011
and.b #%11111011,status  clears invincible (already 0 here)  → %00000011
eor.b #%00000100,status  toggles invincible on  → %00000111

Every other bit is left as it was. That is the whole point of masking: change the bits you name, keep the rest. (The % prefix is real assembler syntax for a binary constant, the way $ marks hex.)

One real routine shows nearly the whole family working together. This is the joypad reader, building the "pressed this frame" byte from the raw controller state, and it uses AND, ANDI, OR, EOR, and NOT in a tight sequence:

	or.b	d1,d0	; Fuse them into one controller bit array
	not.b	d0
	move.b	(a0),d1	; Get button press data
	eor.b	d0,d1	; Toggle off held buttons
	move.b	d0,(a0)+	; Store raw controller input for held button data
	and.b	d0,d1
	move.b	d1,(a0)+	; Store pressed controller input
s2disasm/s2.asm:1380-1386 (Joypad_Read)

OR merges, NOT inverts the pad's inverted logic, EOR finds what changed since last frame, and AND keeps just the new presses. The result is the byte every object reads to see which buttons were tapped. Play with each operation in the bitwise visualizer, which shows the bits themselves:

The key asymmetry on this page: EOR's SOURCE must be a data register, only. Unlike AND and OR, you cannot EOR a memory value or an immediate into a register. eor.w (a0),d0 is refused (move it to a Dn first); eor.w #$FF,d0 is refused (that is EORI).

EOR (exclusive-OR) sets a result bit wherever the two inputs DIFFER, which means a 1 in the mask toggles the matching destination bit and a 0 leaves it alone. Two uses fall out of that: flipping chosen bits, and finding which bits changed between two values. The joypad reader uses the second, to detect new presses:

	move.b	(a0),d1	; Get button press data
	eor.b	d0,d1	; Toggle off held buttons
	move.b	d0,(a0)+	; Store raw controller input for held button data
	and.b	d0,d1
s2disasm/s2.asm:1382-1385 (Joypad_Read)

EOR of this frame against last frame yields exactly the bits that changed; the following AND narrows that to the buttons that went from up to down. See eor in the bitwise visualizer to watch bits toggle.

Gotcha EOR is the one logic instruction whose source cannot be memory or an immediate: it must be a data register. Reach for a memory or constant operand and you need EORI or a load first. (One neat trick this enables: eor.w d0,d0, a register against itself, drives every bit to 0, though moveq #0 is the cheaper way to zero a register.)
Cheaper cousins Flipping a SINGLE bit is BCHG (Shifts & Bits page). Zeroing a register is moveq #0, not a self-EOR.
Encoding

EOR is 1011 rrr ooo mmm rrr: it shares the 1011 prefix with CMP, but uses only the 100/101/110 opmodes (the Dn,<ea> direction), which is exactly why EOR has no memory-source form, those opcodes are CMP. eor.b d0,d1 = $B101.

EORI Exclusive OR Immediate EORI.b/.w/.l #<data>,<ea>

Flags Xunaff. Nset Zset Vclear Cclear
Cost 8 cycles 4 bytes immediate into a data register; memory destinations cost more.
Operands
Dest (with #n) DnAn(An)(An)+-(An)d(An)d(An,Xn)(xx)#nd(PC)

Unlike plain EOR, the immediate form CAN target memory directly, so it is how you toggle bits with a constant mask against a register or a RAM byte.

EORI toggles bits with a constant mask, and it reaches memory that plain EOR cannot. A clean example flips a one-bit page flag in RAM every pass, alternating between two buffers:

	eori.b	#1,(SS_Alternate_PNT).w	; Toggle flag
s2disasm/s2.asm:897 (special-stage double buffering)

Each pass flips the low bit of the flag, so it reads 0, 1, 0, 1..., which the next pass tests to pick the other buffer. One instruction, no load, straight to RAM.

Cheaper cousins A single bit is BCHG (Shifts & Bits page), the clearer one-instruction toggle.
Encoding

EORI is 0000 1010 ss mmm rrr plus the immediate. eori.b #1,d0 = $0A00 then $0001.

NOT Logical Complement NOT.b/.w/.l <ea>

Flags Xunaff. Nset Zset Vclear Cclear
Cost 4 cycles 2 bytes on a data register, byte or word (.l is 6c). Memory destinations cost more.
Operands
Dest DnAn(An)(An)+-(An)d(An)d(An,Xn)(xx)#nd(PC)

A single operand, data-alterable. No address register, no immediate.

NOT flips every bit of its operand, the one's complement. Its two everyday uses appear right next to each other in the code: building an inverted mask for a following AND (as in the AND entry), and correcting the joypad's inverted logic. The pad reports 0 for a pressed button, so NOT turns the whole byte around to the more natural 1-means-held:

	or.b	d1,d0	; Fuse them into one controller bit array
	not.b	d0
s2disasm/s2.asm:1380-1381 (Joypad_Read)

After the NOT, a set bit means "this button is down", which the rest of the reader can work with directly. See not in the bitwise visualizer to watch all eight bits invert at once.

Gotcha NOT is one's complement (flip every bit), not NEG's two's complement (arithmetic negate). not.b on $05 gives $FA; neg.b gives $FB. They differ by one, and confusing them corrupts any signed arithmetic that follows.
Cheaper cousins Flipping a SINGLE bit is BCHG (Shifts & Bits page). To negate arithmetically, use NEG.
Encoding

NOT is 0100 0110 ss mmm rrr: bits 7-6 the size, bits 5-0 the operand. not.b d0 = $4600.
