68000 Assembly

Addressing Modes

Every instruction has to say where its operand lives. The 68000 gives you a small grammar for that: fourteen ways to name a value, from a plain register to a memory cell computed from a base, an index, and an offset. This page lists all of them, shows exactly what each one computes and what it costs in cycles and bytes, and is where every instruction page's timing note points when it says "plus the price of the addressing mode." Every example is real, pulled from the Sonic 2 disassembly and re-verified line by line.

1 · How an instruction names its operand

An instruction like move.w <source>,<dest> is two questions and a verb: copy a word, from where, to where. Each of those two "wheres" is an effective address: a small field inside the instruction that spells out how to find the operand. It might be a register (the value is right there), a plain memory address, or an address the CPU has to build on the spot by adding a base and an offset together. An addressing mode is one recipe for answering "where."

One idea carries through every mode below, so it is worth pinning down once. Each entry names an address (that is the "under the hood" step); the operand the instruction actually reads or writes is the value at that address, one step further in. Address and value are not the same thing, and the parentheses are what tell them apart:

Teaching example · address versus value 
	move.b	#$20,($FFFFF000).w	; store the byte $20 at $FFFFF000

After that store, the same number means two different things depending on how you write it:

#$FFFFF000  resolves to  $FFFFF000, the address (a plain number, a location)
($FFFFF000).w  resolves to  $20, the value that location now holds

The leading # gives you the number itself; the parentheses send the CPU to fetch what lives there. That one difference is the whole of this page.

Real Sonic code writes a name (Vint_routine).w, instead of the raw ($FFFFF000).w. A label is just a readable alias the assembler swaps for the address at build time; the machine sees the same number either way. The examples below use the real names, with the raw address worked out beside each.

New here? · What is an effective address, really?

Think of the instruction as a sentence and the effective address as its noun phrase. "Copy a word" needs a noun: copy a word from what? The 68000 does not force you to write out a full 32-bit memory address every time; that would be slow and huge. Instead it lets you pick a short recipe. "The value in register d0." "The 16 bits that a0 is pointing at." "The 16 bits eight bytes past where a0 points." Each recipe is an addressing mode, and the actual number it works out to, the real memory address, is the effective address the mode produces.

The key idea, and the one thing this whole page is about: some recipes are just a register (no address at all, the value is already in the CPU), some name a fixed address, and some compute an address by adding pieces together at the moment the instruction runs. The computing ones are what make loops and object code compact: point a register at a thing once, then reach every field of it with a small offset. That is why the same three instructions show up over and over in real code with different modes hanging off them.

The additive rule (the one thing to carry away)

Here is the single fact every instruction page leans on. The time an instruction takes is not one number; it is a sum:

Teaching example · not engine code

Total cycles = the instruction's base cost + the price of reaching the source operand + the price of reaching the destination operand. Each addressing mode has a fixed price, and that price is the same table no matter which instruction it hangs off.

Take move.w, whose base is 4 cycles. Reaching a data register is free (0). Reaching a word in memory through (a0) costs 4. So:

	move.w	d0,d1        = 4  +  0 (source d0)  +  0 (dest d1)
	move.w	(a0),d1      = 4  +  4 (source (a0)) +  0 (dest d1) = 8
	move.w	(a0),(a1)    = 4  +  4 (source (a0)) +  4 (dest (a1)) = 12

Nothing here is special to MOVE. Swap in any instruction's base, add the same two mode prices from the table below, and you have its cost. That is why the instruction pages do not each reprint a giant timing matrix: they state a base, then point here for the per-mode add-ons.

The same addition governs SIZE. The opcode itself is one 16-bit word. A mode that needs extra information, a displacement, an absolute address, an immediate constant, tacks on extension words after the opcode, and those are the only reason an instruction grows past two bytes.

New here? · Why does the same value cost different amounts depending on where it lives?

Because "reaching" an operand can mean very different amounts of actual work. If the value is already in a register, the CPU has it: zero extra bus activity. If the value is in memory, the CPU has to run a bus cycle to go fetch it, and a bus cycle on the 68000 moves 16 bits and takes about 4 clock periods. So a word in memory costs roughly 4, and a longword costs about 8, because it is two 16-bit trips.

Modes that first have to compute the address add a little more on top for the internal arithmetic: a displacement mode has to read the displacement and add it; an indexed mode also has to pull in the index register. That is the whole shape of the cost table: registers are free, memory is a few cycles per 16 bits, and computing the address adds a small, fixed surcharge.

New here? · What is an extension word?

An instruction starts as one 16-bit opcode word. That word has room to name a mode and a register number, but not room for a whole 32-bit address or a 16-bit displacement. When a mode needs one of those, the assembler puts it in the words right after the opcode, and the CPU reads them as part of decoding the instruction. Those trailing words are extension words.

So move.w d0,d1 is 2 bytes (opcode only), move.w 8(a0),d1 is 4 bytes (opcode plus one extension word holding the displacement 8), and move.w ($00FF0000).l,d1 is 6 bytes (opcode plus two extension words holding the 32-bit address). Reading each extension word is also a bus cycle, which is exactly why the modes that use them cost more time as well as more space. Space and time move together here.

So how many modes are there?

You will see the 68000 credited with both twelve and fourteen addressing modes, and both numbers are correct once you know what each is counting. The Motorola manual groups its modes into six categories and totals them at fourteen. Twelve of those fourteen are the general effective-address forms, the ones that fit in the 6-bit address field an instruction carries for each operand, so they can appear in most operand slots. This page's chip key and price list cover exactly those twelve. The remaining two are special cases that are not written as a free-standing operand you pick, so they never appear in the grid.

This is the one true per-mode cost table. Every timing note on the instruction pages that says "plus the addressing-mode price" is adding the number from one of these rows. Cycles are the effective-address calculation times from the Motorola MC68000 User's Manual: the clocks spent computing and reaching the operand, added on top of the instruction's own base. They are core-CPU figures; a real Genesis access to ROM or the VDP adds wait states beyond these. The .b/.w column covers byte and word (a byte still moves in one 16-bit bus cycle); the .l column doubles the memory trips. Extension bytes are per operand: #n is 2 bytes for a byte or word immediate and 4 for a long. Registers cost nothing and add nothing; every step down the table is the CPU doing more work to find the value.

One footnoted exception, worth knowing because MOVE is everywhere: as a MOVE destination, -(An) costs the same as (An) (4/8, not 6/10), because MOVE overlaps the decrement with its own work. So move.w d0,-(a0) is 8 cycles, not 10. The 6/10 row applies everywhere else: as a source for any instruction, and as the destination of read-modify-write instructions like add.w d0,-(a0).

Hardware choice · the 6-bit address field The reason there are exactly these forms, and exactly this many, is that each operand's mode is encoded in only 6 bits of the opcode: 3 bits for the mode and 3 bits for a register number. Seven of the eight mode values name a register-based form (Dn through d(An,Xn)), one per register. The eighth mode value, 111, has no register to name, so its 3 register bits are reused to select among the five "special" forms instead: (xx).w, (xx).l, d(PC), d(PC,Xn), and #n. That single design decision, pack the whole "where" into 6 bits, is what fixes the menu at twelve effective-address forms and gives absolute, PC-relative, and immediate their slightly cramped, tacked-on feel.
Reference · the exact 6-bit encoding (mode field + register field)

Every effective address is six bits: a 3-bit mode field and a 3-bit register field. Mode values 000 through 110 each name a register-based form, and their register field carries the register number n (0 to 7). Mode 111 has no register to name, so its register field sub-selects one of the special forms.

Reference · which modes each instruction accepts (data / memory / control / alterable)

Instructions do not accept every mode. Motorola sorts the modes into four capability classes, and each instruction's operand is specified as one of them (or a combination). The four:

  • Data: the operand carries data. Everything except An (an address register is not a data source).
  • Memory: the operand lives in memory, not a register. Everything except Dn and An.
  • Control: a plain memory address with no size or auto-step implied. This is what jmp, jsr, lea, and pea require, which is why you cannot jmp (a0)+ or lea d0,a1.
  • Alterable: the operand can be written. Everything except the two read-only PC-relative modes and the immediate constant.

3 · Register and register-indirect modes

These seven modes all answer "where" with a register: either the value is the register, or the register holds the address, possibly with an offset or an auto-step. They are the workhorses of object code, because a single address register can point at an object and every field of it is one cheap offset away.

New here? · How can one register reach every field of an object?

A Sonic object is just a block of memory: 64 bytes, with the x position at a known spot inside it, the y velocity at another, the collision radius at another. Those "known spots" are fixed offsets from the start of the block: x position at 0, y velocity at some byte further in, and so on. They are the same for every object because every object uses the same layout.

So the routine that runs an object is handed one thing: the address of that object's block, kept in an address register (a0, by convention). From there it never needs another full address. The x position is x_pos(a0), the y velocity is y_vel(a0), each a tiny fixed offset off the one pointer. You might expect each field to need its own address loaded; it does not, and that is the payoff of the displacement mode. Point once, then reach the whole structure for the price of a small constant. This is why d(An) is the single most common addressing mode in game code, and why so much of it threads a0 from routine to routine.

Dn Data register direct

You writeA data register, d0 through d7.
Under the hoodNo address to compute. The register itself is the operand.
You work withThe contents of that register, at the size you name (.b/.w/.l).
Price0 cycles, 0 extension bytes. The cheapest possible operand.
Reach forAnything already in hand: a counter, an intermediate result, a value you are about to combine with another. Most arithmetic keeps both operands in data registers precisely because it is free.

Here a decoded code value in d0 is doubled to turn it into a word-sized table index, then used to index that table. Both operands of the add are data registers, named directly:

	move.b	(a0)+,d0	; get code
	add.w	d0,d0	; each code gets a word-sized entry in the table
	move.w	d7,(a1,d0.w)	; store the entry for the code

Gotcha There is no byte size for an address register. move.b a0,d0 will not assemble: An can only be read or written as a word or a long, because a byte cannot meaningfully hold an address. Also, a word write to An sign-extends to the full 32 bits (that is the MOVEA behaviour on the Moving Data page), and operations on An do not set the condition flags, so you cannot branch on the result of moving one.

New here? · Postincrement and predecrement: what moves the pointer, and when?

Both of these modes do two things in one instruction: they access memory, and they nudge the address register. The only difference is the order, and the order is the whole story.

(a0)+ accesses first, then steps a0 forward: read the value here, then move to the next slot, ready for the next read. That is how you walk a list start to end. -(a0) steps a0 backward first, then accesses: move back one slot, then write there. Run a few of each in the toy below and the asymmetry is visible: post-increment leaves a0 pointing just past what it touched, pre-decrement pointing right at it. That mirror is not an accident. It is exactly what makes a stack work: push with -(sp) (make room, then store) and pop with (sp)+ (read, then reclaim the room), and every push is undone by a matching pop. Neither step ever touches the condition flags, so the value you branch on afterward is the one you moved, never the pointer.

ProcessDMAQueue walks a1 through the queued video-transfer commands, reading one word and advancing with each access:

	move.w	(a1)+,d0
	beq.s	.done	; branch if we reached a stop token
	; issue a set of VDP commands...
	move.w	d0,(a5)		; transfer length
	move.w	(a1)+,(a5)	; transfer length
	move.w	(a1)+,(a5)	; source address

New here? · What does "base plus index plus displacement" actually compute?

Picture a table in memory: a row of entries, one after another, starting at some address. To read the entry you want, you need three facts. Where does the table start? That is the base, held in an address register. How far into the table is your entry? That is the index, a number in a register that changes at runtime (slot 3, frame 7, whatever). And is there a fixed step from the register to the actual first entry, a header to skip or a named field the table sits inside? That is the displacement, a small constant the assembler bakes in.

The mode adds all three and reads there: address = base + index + displacement. The reason it is one instruction instead of an add-then-read is that indexing a table is so common the CPU does the arithmetic for free as it computes the address. One caution that trips people: on the plain 68000 the index is not scaled, so if each entry is a word (2 bytes), the index has to already be doubled. That is exactly the add.w d0,d0 you saw in the very first example, turning a slot number into a byte offset before it is used as an index.

The sound-driver feed reads one slot of a queue, where the table base is a field (SoundQueue.SFX0 off a0) and the slot number lives in d1:

	move.b	SoundQueue.SFX0(a0,d1.w),d0
	beq.s	.skip

Gotcha Two size traps live in this mode. The displacement is only 8 bits signed (-128 to 127), far smaller than the plain d(An) mode's 16, so a large constant offset does not fit here. And the index register can be taken as a word or a long: d1.w uses the low 16 bits sign-extended, so a "negative" index reaches backward, while d1.l uses all 32 bits. Picking .w when the value can exceed a signed 16-bit range is a classic off-into-the-weeds bug. There is no automatic scaling of the index on the 68000: the register value is added as-is, so a table of word entries needs the index pre-doubled, exactly what the Dn example above was doing with add.w d0,d0.

4 · Absolute, immediate, and PC-relative modes

The remaining five modes are the "mode 111" family: they do not hang off an address register at all. Two name a fixed memory address, one carries a constant, and two measure from the program counter (the CPU's running address of the next instruction; the pocket version is below and the full story is on the Moving Data page). They are how code reaches named variables, bakes in constants, and finds its own tables.

New here? · A quick reminder: labels, the PC, and the # sign

Three things this section leans on are taught in full on the Moving Data page; the pocket versions:

A label like Vint_routine is just a name for an address; the assembler works out the number at build time from where the label sits, and you never type the number yourself. The program counter is the CPU's finger on the page, the address of the instruction it is fetching; the PC-relative modes measure an offset from it. And a leading # means "the constant itself," not "the memory at this address": #4 is the number 4, while a bare 4 would mean the bytes at address 4.

Vint_routine names the work-RAM byte $FFFFF62A; tst.b tests the value there, $06, and the beq jumps to the lag path if it is zero. The .w is the address size; the byte is what gets tested.

Hardware choice · why the short form sign-extends A 16-bit address by itself can only name the bottom 64KB of memory. The 68000 instead treats the stored 16 bits as signed and extends the top bit across the upper half, so an address like $FF00 becomes $FFFFFF00, not $0000FF00. That splits the short form's reach into two 32KB windows: the very bottom of the map ($00000000 up) and the very top ($FFFF8000 up). The Genesis wires its work RAM and I/O into that top window on purpose, so nearly every RAM variable and hardware register is reachable in the cheap 2-byte form. The tradeoff is the hole in the middle: anything from $00008000 to $FFFF7FFF needs the full 4-byte absolute long.

Gotcha This is plain immediate. The small constant inside moveq, addq, and subq is a different animal, the "quick immediate" mode, packed right into the opcode word with no extension at all (and limited: signed 8-bit for moveq, 1 to 8 for addq/subq). That is one of the two modes that push Motorola's count from twelve to fourteen. When a constant fits, the quick forms are smaller and faster than a plain #n; when it does not, you fall back to this mode.

5 · PC-relative versus absolute: the choice

Whenever code wants to reach a fixed location, it faces a fork: name it by its absolute address ((xx).w or (xx).l), or name it as a distance from the program counter (d(PC)). Both can reach the same spot; they differ in cost, in whether the code survives being moved, and in one hard limit that decides the matter for RAM.

Cost. For a nearby target the PC-relative form is the cheaper of the honest options. lea Table(pc),a0 spends one extension word (4 bytes, and the LEA runs in 8 cycles); the absolute-long equivalent lea (Table).l,a0 spends two extension words (6 bytes, 12 cycles). Over a whole ROM those saved words add up, which is why Sonic-era source reaches for (pc) so often.

New here? · What does "position-independent" mean, and why would code ever move?

Position-independent code is code that works no matter what address it is loaded at, because it never hard-codes where it is. Instead of "jump to address $001234," it says "jump 40 bytes ahead of me"; instead of "read the table at $002000," it says "read the table 12 bytes past me." Every reference is relative to the code's own position, so you can pick up the whole block and drop it somewhere else and every distance still lands where it should.

Why would code move? Because ROM layout shifts constantly during development: add a routine above this one, or the assembler packs things differently, and every fixed address below it slides to a new spot. Code written with PC-relative references does not care; code that baked in absolute addresses to its own nearby tables would now point at the wrong place. That is the deeper reason Sonic-era source reaches for (pc) whenever it addresses a table that travels with the code: not just to save a couple of bytes, but so the block stays correct when everything around it moves.

Relocatability. An absolute reference bakes in a fixed number: move the code and the reference still points at the old spot, now wrong. A PC-relative reference says "this far from me," so if the whole block is relocated in ROM, the distance to a table that moved with it is unchanged and the code still works. That is what "position-independent" means, and it is the deeper reason to prefer (pc) for code-space data.

The 32KB reach limit. The catch is that the PC-relative displacement is 16 bits signed, so d(PC) can only name a target within 32KB of the instruction. Reach farther and it will not assemble; you must fall back to an absolute address. So PC-relative is the tool for near data, and absolute long is the tool for anything far.

Why RAM can never be PC-relative. Put those two facts together and a rule drops out. Code lives in ROM at the bottom of the map; work RAM lives at the top, at $FFxxxx, megabytes away. No PC-relative displacement can span that gap, so a RAM variable is never reachable as d(PC). RAM is always named by an absolute address, and because it sits in the top window, the cheap (xx).w short form reaches it.

The result is a house pattern you will see everywhere in real 68000 source, three modes sorted by what they are naming:

TargetModeWhy
Nearby code-space data (tables, setup arrays)Label(pc)Cheapest and relocatable, within 32KB
Work RAM variables and low hardware(Label).wAbsolute is mandatory for RAM; the short form reaches the top window in 2 bytes
Far-off ROM data, out-of-window hardware(Label).lOnly the full 32-bit address reaches it
Hardware choice · PC-relative is read-only You can read through d(PC) and d(PC,Xn), but you cannot write through them: the 68000 leaves both PC-relative modes out of the destinations any instruction will accept, so move.w d0,Table(pc) is refused at assembly time. The reasoning is that PC-relative addressing exists to fetch constants and tables that sit next to the code, in ROM, which is not writable anyway; a writable position-independent reference would have to survive relocation while pointing at mutable data, which the simple "distance from PC" scheme cannot promise. So writes always go to an absolute address or through an address register, and PC-relative stays a read-only, fetch-from-my-neighborhood tool.
