68000 Assembly

Instructions: Moving Data

Before a CPU can add, compare, or branch on anything, it has to get a value into a place it can work with. This page covers the ten 68000 instructions that do exactly that: they copy, load, clear, and rearrange values between registers, memory, and the stack, but never change what the values mean. Every example on this page is real, pulled from the Sonic 2 disassembly and re-verified line by line.

1 · The MOVE family

Every one of these three instructions copies a value from somewhere to somewhere else. What separates them is where that somewhere is allowed to be, and what the CPU does to its status flags along the way. A register is the 68000's fastest working storage: eight general-purpose data registers (D0-D7) for values, and eight address registers (A0-A7, with A7 doubling as the stack pointer) for pointers. Reading or writing a register costs nothing extra in bus time, unlike memory, which is why so much 68000 code is a dance of getting values into registers, working on them there, and writing results back out.

New here? · What do .b, .w, and .l mean?

Almost every 68000 instruction can be told how much data to move: a .b byte (8 bits), a .w word (16 bits), or a .l longword (32 bits). The suffix goes right after the mnemonic, so move.b copies one byte, move.w copies two, and move.l copies four. Registers are always 32 bits wide internally, but an instruction sized .b or .w only touches the low part of that 32 bits and leaves the rest of the register alone, which is a common source of stale-data bugs for anyone assuming a byte operation clears the whole register the way it might on other chips.

Picking the smallest size that fits the job matters on real hardware: a .b move is both smaller in ROM and faster to execute than the equivalent .l move to the same address, because the 68000's bus does less work for a byte transfer.

New here? · What is an "effective address"?

Almost every instruction on this page names its source or destination as an effective address (often abbreviated <ea> in syntax), which is really a small grammar for "where the value lives," not just a single number. It might be a register itself (d0), a plain memory address ((Some_Label).w), an address held in a register with a symbol added to it (x_pos(a0), the classic "struct field" pattern), or an immediate constant baked right into the instruction (#4). The 68000 supports fourteen distinct effective-address forms in total; this page uses several of them in its examples without dwelling on the mechanics of each, since a dedicated Addressing Modes page covers exactly how every form resolves to a real address.

New here? · What's an immediate value?

An immediate is a constant baked directly into the instruction's own bytes in ROM, written with a leading #, like #0 or #'SEGA'. It is the opposite of an address: move.l #4,d0 puts the number 4 into d0, while move.l 4,d0 (no #) would instead read whatever four-byte value sits at memory address 4. Mixing these up is one of the most common typos in 68000 code, and one the assembler cannot catch for you, since both are syntactically legal instructions that just do very different things.

MOVE MOVE.b/.w/.l <ea>,<ea>

Copies data from a source effective address to a destination effective address, in any of the three sizes. It is the single most common instruction in any 68000 program: the general-purpose "get this value from here to there."

Flags Xunaff. Nset Zset Vclear Cclear
New here? · What are these X N Z V C flags?
The CPU keeps five status bits called the condition codes, or flags. Most instructions update them as a side effect of whatever they just computed: N (the result was negative, its top bit set), Z (the result was zero), V (a signed overflow happened), C (an unsigned carry or borrow happened), and X (the extend bit, a saved copy of carry that multi-precision math uses). On their own they do nothing; their whole point is that the branch instructions read them a moment later to decide where to go. So "MOVE sets N and Z" means that right after a MOVE you can branch on whether the value was negative or zero without a separate compare, which is exactly what the second example below does. The Flags row on each instruction shows which bits it touches: set means updated from the result, clear means forced to zero, unaff. means left exactly as it was. The Programming Model page (coming) will cover the status register in full.

Real example, satisfying the TMSS check right at boot, an immediate longword written straight to a hardware register:

	beq.s	SkipSecurity				; If the console has no TMSS, skip the security stuff.
	move.l	#'SEGA',Security_Addr-Z80_Bus_Request(a1) ; Satisfy the TMSS
s2disasm/s2.asm:211-212

A second, equally idiomatic use is reading a value purely to react to it, using MOVE's own flag side effects instead of a separate comparison:

SkipSecurity:
	move.w	(a4),d0	; check if VDP works
s2disasm/s2.asm:214-215
Gotcha Because MOVE sets N and Z from the value it just moved, it doubles as a free "test and branch": move.w (a4),d0 followed by a branch on Z or N needs no separate tst. The privileged forms (MOVE <ea>,SR, MOVE USP,An) are supervisor-only; attempting one from user mode traps instead of executing.

MOVEQ MOVEQ #<data>,Dn

Loads an 8-bit immediate into a data register, sign-extended to fill the entire 32 bits, and does it in a single word of opcode instead of MOVE's longer encoding. There is no size suffix: MOVEQ always writes a full long.

Flags Xunaff. Nset Zset Vclear Cclear

The classic full-register zero idiom, right in the same boot sequence:

	move.w	(a4),d0	; check if VDP works
	moveq	#0,d0	; clear d0
s2disasm/s2.asm:215-216

And a small constant loaded as a countdown for a following loop:

	moveq	#VDPInitValues_End-VDPInitValues-1,d1 ; run the following loop $18 times
s2disasm/s2.asm:220
Gotcha The immediate is an 8-bit signed field, range -128 to 127. MOVEQ cannot express #200 or #$8000 at all; those need a plain MOVE instead. And despite encoding only a byte, MOVEQ always clears the entire 32-bit destination through sign extension, which is exactly why moveq #0,d0 is the standard way to zero a whole data register.

MOVEA MOVEA.w/.l <ea>,An

The address-register form of MOVE: copies a value into one of A0-A7. Plain MOVE cannot target an address register directly, so whenever the destination is An, this is the instruction that actually gets assembled, whether or not the source code spells out "movea" explicitly.

Flags Xnone Nnone Znone Vnone Cnone

Two absolute addresses loaded ahead of a checksum scan:

	movea.l	#EndOfHeader,a0	; start checking bytes after the header ($200)
	movea.l	#ROMEndLoc,a1	; stop at end of ROM
s2disasm/s2.asm:382-383

And MOVEA used purely to zero an address register, chained right after the MOVEQ example above:

	moveq	#0,d0	; clear d0
	movea.l	d0,a6	; clear a6
s2disasm/s2.asm:216-217
Gotcha MOVEA has no byte form at all, only .w and .l, since a byte cannot meaningfully hold an address. A .w source is sign-extended to 32 bits on the way in, so movea.w #$8000,a0 leaves a0 = $FFFF8000, not $00008000, which surprises anyone thinking in 16-bit address spaces. And because MOVEA touches no flags at all, you cannot branch on its result the way MOVE's example above does; a MOVEA immediately followed by bne is testing whatever flags were already set beforehand, not this instruction's outcome.

2 · Moving addresses and whole register sets

The three instructions below step outside the plain "copy a value" pattern. LEA and PEA move an address itself rather than the data sitting at that address, and MOVEM moves an entire list of registers in one shot, which matters most at the boundary of a subroutine or an interrupt handler, where every register in use has to be saved and later put back exactly as it was.

New here? · What's the difference between an address and the data at that address?

Think of an address as a house number and the data as what's inside the house. If a variable named Score lives at memory location $FF1000, then "Score's address" is the number $FF1000 itself, while "Score's value" might be some completely different number, say 4200, sitting at that location. move.w Score,d0 walks into the house and brings back what's inside (4200 lands in d0); lea Score,a0 just writes down the house number ($FF1000 lands in a0). Confusing the two is one of the most common early mistakes in 68000 code, because both instructions are one word long and look almost identical on the page.

New here? · What is the stack, and why does the CPU need one?

The stack is a region of work RAM that grows downward and is always addressed through A7, the stack pointer. "Push" means write a value at the address A7 currently points to and then decrease A7 by the value's size; "pop" (more properly, "pull") means increase A7 first and then read from there. Because it always operates on whatever A7 points to right now, the stack is naturally last-in-first-out: the most recent thing pushed is the first thing that comes back off.

That discipline is exactly what a subroutine call needs. jsr pushes a return address before it jumps, and rts pops that same address to come back, so nested calls just nest their return addresses on the stack automatically, no bookkeeping required by the programmer. MOVEM's predecrement and postincrement forms, covered below, lean on the same mechanism to save and restore whole banks of registers around a call.

LEA LEA <ea>,An

Computes an effective address and loads that address itself, not the data living there, into an address register. Always a longword operation; there is no size suffix.

Flags Xnone Nnone Znone Vnone Cnone

A PC-relative table load, right at reset, the standard position-independent way to address a local table:

	lea	SetupValues(pc),a5	; Load setup values array address.
s2disasm/s2.asm:206

And loading a hardware port address once, for repeated reuse through the register afterward:

	lea	(VDP_control_port).l,a5
s2disasm/s2.asm:704
Gotcha Only "control" addressing modes are legal LEA sources: no Dn, An, #immediate, or the auto-increment/decrement forms, since none of those name a memory address in the first place. Beginners commonly confuse lea (foo).w,a0 (load the address of foo) with move.w (foo).w,d0 (load the value stored at foo); the two-instruction difference between "where" and "what" is the entire point of this instruction.

PEA PEA <ea>

Computes an effective address exactly like LEA, then pushes that 32-bit address onto the stack instead of loading it into a register. Always pushes 4 bytes.

Flags Xnone Nnone Znone Vnone Cnone

PEA is genuinely rare in hand-written 68000 game code; it shows up far more often as compiler-generated "push this argument's address" glue than as something a person types. The one real use found in the Sonic 2 disassembly is not that at all: it is a deliberate, commented "fake call" trick. The routine pushes the address right after itself, then branches (not calls) into shared code, so that shared code's own rts returns to the label following the PEA, exactly as if the PEA'd branch had been a subroutine call all along:

	; This is just a fancy efficient way of doing 'if true then call this, else call that'.
	pea	+(pc)
	tst.w	(Two_player_mode).w
	beq.w	CalculateVRAMAddressOfBlockForPlayer1.AbsoluteXAbsoluteY
	bra.w	CalculateVRAMAddressOfBlockForPlayer1.AbsoluteXAbsoluteY_DoubleResolution
+
s2disasm/s2.asm:20004-20009 (DrawInitialBG_LoadWholeBackground_512x256)
Gotcha This is an unusual, advanced idiom, not PEA's typical job of parameter passing, and it is worth reading twice: the PEA supplies the return address a plain bsr/jsr would have pushed automatically, so a bra can be used instead, picking between two destinations with a branch rather than duplicating the call. A second, structurally identical use of the same trick exists later in the same file, at line 29666.

MOVEM MOVEM.w/.l <register list>,<ea>  |  MOVEM.w/.l <ea>,<register list>

Transfers a bitmask-selected list of registers to or from consecutive memory in a single instruction. This is the standard way to save every register a routine will clobber on entry, and restore them all on exit, without a separate MOVE per register.

Flags Xnone Nnone Znone Vnone Cnone

Saving every register at the very top of the vertical interrupt handler, predecrementing onto the stack:

V_Int:
	movem.l	d0-a6,-(sp)
	tst.b	(Vint_routine).w
	beq.w	Vint_Lag
s2disasm/s2.asm:481-484

And the matching restore, postincrementing back off the stack right before returning from the interrupt:

VintRet:
	addq.l	#1,(Vint_runcount).w
	movem.l	(sp)+,d0-a6
	rte
s2disasm/s2.asm:507-510

MOVEM is not limited to whole-stack saves; it also loads a fixed setup table in two passes, a word-sized group followed by a long-sized group, both postincrementing through the same pointer:

	lea	SetupValues(pc),a5	; Load setup values array address.
	movem.w	(a5)+,d5-d7
	movem.l	(a5)+,a0-a4
s2disasm/s2.asm:206-208
Gotcha With a predecrement destination (register-to-memory), the registers are stored in descending order, A7 down through D0, specifically so that a matching postincrement reload restores them in their original order. This is a real encoding quirk, not just a documentation note: the register-list bits mean something different for -(An) than they do for every other addressing mode, which surprises anyone reading raw opcode bytes expecting one consistent bit order. Separately, a .w load into a register sign-extends each 16-bit value to fill the full 32 bits, not zero-extends, a classic bug source when the intent was an unsigned word.

3 · Clearing and rearranging

The last four instructions don't move data between two different places at all; they reshape a value that is already sitting where it needs to be. CLR wipes a location to zero, SWAP and EXG rearrange bits or whole registers in place, and EXT changes how many bits of a register actually carry meaning, without touching memory at all.

New here? · What is sign extension, and why does it matter?

A signed byte like -1 is stored as $FF, and a signed word holding -1 is stored as $FFFF. If you simply copied that byte $FF into the low 8 bits of a 16-bit value and left the top 8 bits zero, you would get $00FF, which is 255, not -1: a completely different number. Sign extension fixes this by copying the value's top bit (the sign bit) into every new bit added above it, so $FF extended to 16 bits correctly becomes $FFFF, still -1.

This matters constantly on the 68000 because so much game data is packed into bytes to save ROM space (an object's radius, a velocity, an angle delta), but arithmetic on that data usually needs to happen at word or long size. EXT exists specifically to do this conversion correctly; getting it wrong (or skipping it, and doing math on an unextended byte) is a classic source of "works for positive values, breaks for negative ones" bugs.

CLR CLR.b/.w/.l <ea>

Sets the destination operand to zero, in any of the three sizes. It reads as "clear," and does exactly that: whatever was there before is gone, replaced with 0.

Flags Xunaff. Nclear Zset Vclear Cclear

Zeroing a flag byte before a batch of work runs:

loc_1072:
	clr.b	(Do_Updates_in_H_int).w
	movem.l	d0-a6,-(sp)
	bsr.w	Do_Updates
s2disasm/s2.asm:1260-1263

And the canonical "clear a struct field" pattern, stopping an object's velocity by zeroing both its axes directly:

	st.b	(CutScene+objoff_34).w
	clr.w	x_vel(a0)
	clr.w	y_vel(a0)
	bra.s	-
s2disasm/s2.asm:13594-13597
Gotcha On the original MC68000 specifically (not the later 68010 and up), CLR to a memory destination performs a read-then-write bus cycle, not a write-only one. That is invisible for ordinary work RAM, but it matters if the destination is a hardware register with a read side effect, since clearing it can silently trigger that side effect as an unwanted extra read. Later 680x0 family members fixed CLR to be write-only.

SWAP SWAP Dn

Exchanges the high 16 bits and the low 16 bits of a data register, in place. There is no size suffix; it always operates on the full 32-bit register, and it only ever targets a register, never memory.

Flags Xunaff. Nset Zset Vclear Cclear
high 16low 16 SWAP → low 16high 16

Both real uses here build a DMA command word out of two 16-bit halves computed separately, which is exactly the situation SWAP is built for. First, shaping a VRAM destination address into the top half of a DMA command:

	andi.l	#$FFFF,d2 ; command to specify destination address and begin DMA
	lsl.l	#2,d2
	lsr.w	#2,d2
	swap	d2
	ori.l	#vdpComm($0000,VRAM,DMA),d2 ; set bits to specify VRAM transfer
s2disasm/s2.asm:1744-1748

The same idiom again, in a different routine, moving a low-half command into position before writing the full longword to the VDP control port:

	ori.w	#vdpComm($0000,VRAM,WRITE)>>16,d0
	swap	d0
	move.l	d0,(a4)
s2disasm/s2.asm:2228-2230
Gotcha SWAP never touches memory, only a data register. It's a cheap way to assemble a 32-bit value out of two independently-computed 16-bit halves (compute the low half, swap it into the top, OR or add in the real low half) instead of reaching for shifted 32-bit arithmetic, which is exactly the pattern both examples above share: building a VDP command word half by half.

EXG EXG Dn,Dn  |  EXG An,An  |  EXG Dn,An

Exchanges the full 32-bit contents of two registers, in any mix of data and address registers. Always a full-long swap, regardless of the "logical" size of the values held.

Flags Xnone Nnone Znone Vnone Cnone

EXG is genuinely uncommon in hand-written 68000 game code. Most register swaps go through a spare register and a couple of MOVEs instead, since EXG doesn't obviously save time or registers in most contexts; across the entire 91,000-line Sonic 2 disassembly it appears only a handful of times, always as a plain two-register swap inside physics code. Here, swapping two floor-distance candidates so the smaller ends up where the following code expects it:

loc_1ECC6:
	move.b	(Secondary_Angle).w,d3
	cmp.w	d0,d1
	ble.s	loc_1ECD4
	move.b	(Primary_Angle).w,d3
	exg	d0,d1
s2disasm/s2.asm:43601-43606 (Sonic_CheckFloor)
Gotcha Three more near-identical exg d1,d2 call sites exist elsewhere in the same disassembly, at lines 47667, 47692, 54498, and 54548, all the same register-swap idiom in different physics routines. That rarity is itself a useful signal: reach for a temp-register MOVE first, and treat EXG as a specialist tool for the rare spot where swapping two full registers in one step is genuinely worth it, not a routine replacement for MOVE.

EXT EXT.w Dn  (byte→word)   |  EXT.l Dn  (word→long)

Sign-extends the low portion of a data register to fill the rest of the register, in place. EXT.w takes the low byte and extends it to a full word; EXT.l takes the low word and extends it to a full long.

Flags Xunaff. Nset Zset Vclear Cclear

A byte radius field read off an object, extended to a word before it takes part in word-sized arithmetic:

	moveq	#0,d0
	move.b	y_radius(a0),d0
	ext.w	d0
	add.w	d0,d2
s2disasm/s2.asm:43587-43590

And a decoded code-length byte extended for use in a word-sized subtraction, inside a compression decoder's bit-reading loop:

	move.b	(a1,d1.w),d0	; get code length in bits
	ext.w	d0
	sub.w	d0,d6	; subtracted from the shift value, such that the next time around the next code is read
s2disasm/s2.asm:1865-1867
Gotcha EXT only ever operates on a register, never a memory operand; sign-extending a byte value that lives in memory means MOVE.b-ing it into a register first, then EXT.w-ing it there, exactly the shape of both examples above. There is also no single-step byte-to-long form on the plain 68000: going all the way from a byte to a long takes two instructions, ext.w then ext.l. Reference material for later 680x0 chips sometimes mentions EXTB.L, a direct byte-to-long extension, but that instruction is 68020-and-up only and does not exist on the plain 68000 this page covers.
