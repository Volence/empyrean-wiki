68000 Assembly

Instructions: Moving Data

Before a CPU can add, compare, or branch on anything, it has to get a value into a place it can work with. This page covers the ten 68000 instructions that do exactly that: they copy, load, clear, and rearrange values between registers, memory, and the stack, but never change what the values mean. Every example on this page is real, pulled from the Sonic 2 disassembly and re-verified line by line.

Scope: these are the ten everyday movers. A few specialized ones sit elsewhere, MOVEP (byte-spreading to peripherals, rare in Genesis code), plus the SR/CCR/USP moves and the stack-frame pair LINK/UNLK that the Flow & System page covers.

1 · The MOVE family

Every one of these three instructions copies a value from somewhere to somewhere else. What separates them is where that somewhere is allowed to be, and what the CPU does to its status flags along the way. A register is the 68000's fastest working storage: eight general-purpose data registers (D0-D7) for values, and eight address registers (A0-A7, with A7 doubling as the stack pointer) for pointers. Reading or writing a register costs nothing extra in bus time, unlike memory, which is why so much 68000 code is a dance of getting values into registers, working on them there, and writing results back out.

Two reading conventions before the examples start. The 68000 writes operands source, then destination: move.w d0,d1 copies d0 into d1, so the right-hand operand is where the value lands. And the punctuation carries meaning: a # marks an immediate literal (#$100 is the number itself), a $ marks hexadecimal, and parentheses mean the memory at an address ((a0) is what a0 points at, not a0 itself). The Addressing Modes page carries the full notation key.

New here? · What do .b, .w, and .l mean?

Almost every 68000 instruction can be told how much data to move: a .b byte (8 bits), a .w word (16 bits), or a .l longword (32 bits). The suffix goes right after the mnemonic, so move.b copies one byte, move.w copies two, and move.l copies four. Registers are always 32 bits wide internally, but an instruction sized .b or .w only touches the low part of that 32 bits and leaves the rest of the register alone, which is a common source of stale-data bugs for anyone assuming a byte operation clears the whole register the way it might on other chips.

Picking the size that fits the job matters on real hardware, but not the way beginners often assume. A register-to-register move is 2 bytes and 4 cycles at every size, so the suffix alone does not make it smaller or faster. Where size pays off is memory and long immediates: a byte or word transfer to memory moves less across the bus (fewer cycles) than a long, and a long immediate costs two extension words where a byte or word immediate costs one (and MOVEQ costs none).

New here? · How can a register hold a negative number?

There is no minus sign in hardware; a register only holds bits. Negative numbers work by wrap-around, like a car odometer rolling backwards. A byte counts 0, 1, 2, up to $FF (255), and then wraps back to 0; run it the other way and 0 minus 1 wraps to $FF. So the bit pattern $FF means 255 or -1 depending entirely on how the code chooses to read it: the bits themselves do not know. This scheme is called two's complement, and every number on the 68000 uses it. The tell is the top bit: read as signed, $00-$7F are 0 to 127, and $80-$FF run from -128 up to -1. That top bit is exactly what the N flag copies.

And here is the part that trips everyone once: the same negative number needs different digits at different sizes. Think of the odometer again: roll 5 below zero on a three-digit odometer and it reads 995; roll the same 5 below zero on a four-digit odometer and it reads 9995. Those leading 9s are what the leading F digits are in hex. So -128 written as a byte is $80, the same -128 written as a word is $FF80, and as a long it is $FFFFFF80. The FF digits are not decoration and they do not change the value; they are simply what that value looks like with more room. Keeping them correct when a small value moves into a bigger slot is called sign extension, and the EXT instruction at the end of this page exists to do exactly that.

New here? · What is an "effective address"?

Almost every instruction on this page names its source or destination as an effective address (often abbreviated <ea> in syntax), which is really a small grammar for "where the value lives," not just a single number. It might be a register itself (d0), a plain memory address ((Some_Label).w), an address held in a register with a symbol added to it (x_pos(a0), the classic "struct field" pattern), or an immediate constant baked right into the instruction (#4). The 68000 supports fourteen addressing modes, twelve of them general effective-address forms; this page uses several of them in its examples without dwelling on the mechanics of each, since the Addressing Modes page covers exactly how every form resolves to a real address.

New here? · What's an immediate value?

An immediate is a constant baked directly into the instruction's own bytes in ROM, written with a leading #, like #0, #60, or #$100. It is the opposite of an address: move.l #4,d0 puts the number 4 into d0, while move.l 4,d0 (no #) would instead read whatever four-byte value sits at memory address 4. Mixing these up is one of the most common typos in 68000 code, and one the assembler cannot catch for you, since both are syntactically legal instructions that just do very different things.

New here? · What is a label, and who decides its address?

A label like Score, Return_Here, or ObjCC is just a name for a spot: a place in the code, or a slot in RAM. You do not hand it a number yourself (you can bind a name to a value directly with equ or =, but a location label like these is a different tool). The assembler works out its address at build time, by plain bookkeeping: it walks the source top to bottom, adds up the size of everything it emits, and whatever address the count has reached when it meets your label IS that label's value from then on. Write the same code with one extra instruction above the label and the label's address quietly moves; nobody edits it by hand.

So when an example says "Return_Here is at $00020010," that number came from the assembler's bookkeeping, not from anyone declaring it. And the (pc) suffix in pea Return_Here(pc) changes none of this: it only tells the assembler to ENCODE the reach as "this many bytes ahead of the current instruction" instead of as the full address, arithmetic it likewise does itself. Either way, you write the name and the assembler does the numbers.

So why ever write the (pc)? Both forms are legal and reach the same place; the difference is cost and survival. The full-address form spends two extension words (pea (Label).l: 6 bytes, 20 cycles); the distance form spends one (pea Label(pc): 4 bytes, 16 cycles), and because it says "this far from me" rather than "at this fixed spot," it keeps working if the whole block of code is moved somewhere else in ROM. The catch is reach: the distance field is 16 bits signed, so (pc) only works for labels within 32KB of the instruction, which is also why work-RAM variables in Genesis code cannot use it: the code runs from ROM, and RAM is nowhere near it. Hence the pattern all over Sonic-era source: nearby code labels get (pc), work-RAM variables in the high $FFxxxx range get (xxx).w (those addresses sign-extend from a single word), far-away data gets (xxx).l.

New here? · What is the program counter?

The program counter (PC) is the CPU's finger on the page: a special register holding the address of the next instruction to fetch. Execute an instruction and the finger has already slid past it; that is the entire mechanism of "running a program." It is not one of the d0-d7 / a0-a7 registers and you cannot MOVE into it: the only instructions allowed to write the PC are the jumps, branches, calls, and returns, which exist precisely to do that one job. A jmp ($00020010).l is the 68000's way of spelling "move this address into the PC," and rts is "pop 4 bytes into the PC."

The PC also moonlights as an addressing base: a d(PC) operand like SetupValues(pc) means "this far from where the PC is right now," which is what lets code find its own tables no matter where in ROM it ends up. The Programming Model page covers the full register set.

MOVE takes a value from one place and copies it into another. The place it copies FROM keeps its value; the place it copies INTO is overwritten with the copy. That is the whole idea. It works in any of the three sizes and is the single most common instruction in any 68000 program: the general-purpose "get this value from here to there." Setting an object's speed, arming a timer, handing a command to the VDP: nearly everything a game does, every frame, starts with a MOVE putting a value where the next routine expects to find it.

New here? · What are these X N Z V C flags?
The CPU keeps five status bits called the condition codes, or flags. Most instructions update them as a side effect of whatever they just computed: N (the result was negative, its top bit set), Z (the result was zero), V (a signed overflow happened), C (an unsigned carry or borrow happened), and X (the extend bit, a saved copy of carry that multi-precision math uses). On their own they do nothing; their whole point is that the branch instructions read them a moment later to decide where to go. So "MOVE sets N and Z" means that right after a MOVE you can branch on whether the value was negative or zero without a separate compare, which is exactly what the read-and-branch example below does. The Flags row on each instruction shows which bits it touches: set means updated from the result, clear means forced to zero, unaff. means left exactly as it was. The Programming Model page covers the status register in full.

In real object code a MOVE usually deposits a value that some other routine acts on later. These two are from Sonic 2's ending cutscene: the object is the Tornado, Tails' biplane, and this is the exact moment it starts drifting across the ending sky:

	move.w	#$100,x_vel(a0)
	move.w	#-$80,y_vel(a0)
s2disasm/s2.asm:13543-13544 (ObjCC, the ending-sequence Tornado)

a0 points at the plane's object data block, and x_vel / y_vel are its horizontal and vertical speed fields. Nothing moves this instant: the MOVEs just drop in the constants. These are velocities in Sonic's fixed-point format, where one whole pixel per frame is $100 (256 subpixel units), so #$100 is 1 pixel per frame rightward and #-$80 is half a pixel per frame upward, not 256 and 128 pixels. Every frame afterward the engine adds those speeds to the plane's position, and that is what actually carries it up and to the right, out over the clouds.

A MOVE is also how you arm a countdown. The retracting spikes hold still, then snap out on a timer:

	move.w	#60,spikes_retract_timer(a0)	; reset timer
s2disasm/s2.asm:29357 (MoveSpikes, the retracting spikes object)

This copies 60 into the object's retract timer, which is 60 frames, about one second on an NTSC console (PAL hardware runs at 50 frames a second, so a little longer there). The move itself changes nothing you can see; each following frame the object counts that timer down, and when it hits zero the spikes move. One MOVE sets the fuse; the countdown elsewhere in the object burns it.

And because a MOVE sets the N and Z flags from the value it copied, you can react to that value on the spot, with no separate compare. This is ProcessDMAQueue, the routine that flushes the frame's queued video transfers to the VDP; it reads the next entry of the command buffer and stops at the zero end-marker:

	move.w	(a1)+,d0
	beq.s	.done	; branch if we reached a stop token
s2disasm/s2.asm:1775-1776 (ProcessDMAQueue)

The MOVE copies a word from the buffer into d0 and, as a side effect, sets Z if that word was zero. The beq on the very next line reads that flag: a zero entry marks the end of the queue, so it branches away and stops. Reading and testing happen in one step.

Gotcha MOVE copies; it does not move. After move.w d0,d1 the source d0 is unchanged, the point the visualizer above makes on purpose. Next, a .w or .l MOVE whose address is odd does not just misbehave, it crashes: the 68000 takes an address-error exception for any word or long access to an odd address. Only byte moves are exempt. And the privileged forms (MOVE <ea>,SR and MOVE USP,An) are supervisor-only; running one in user mode traps instead of executing.
Cheaper cousins Loading a small constant (-128 to 127) into a data register? MOVEQ does it in half the size. Zeroing a whole data register? moveq #0, always: 4 cycles against clr.l's 6. Zeroing memory is CLR's job. Loading an address register? That is MOVEA; loading a label's address rather than its contents, LEA. Saving or restoring a pile of registers? One MOVEM beats a run of MOVEs.
Encoding

MOVE packs into a single 16-bit word: bits 15-14 are 00, bits 13-12 are the size (byte 01, word 11, long 10), then the DESTINATION as register-then-mode, then the SOURCE as mode-then-register. Storing the destination fields in the opposite order from every other instruction is MOVE's famous encoding quirk.

move.w d0,d1 assembles to $3200: the size is word (11), the destination is d1 (register 001, mode 000), and the source is d0 (mode 000, register 000).

2 · Moving addresses and whole register sets

The three instructions below step outside the plain "copy a value" pattern. LEA and PEA move an address itself rather than the data sitting at that address, and MOVEM moves an entire list of registers in one shot, which matters most at the boundary of a subroutine or an interrupt handler, where every register in use has to be saved and later put back exactly as it was.

New here? · What's the difference between an address and the data at that address?

Think of an address as a house number and the data as what's inside the house. If a variable named Score lives at memory location $FF1000, then "Score's address" is the number $FF1000 itself, while "Score's value" might be some completely different number, say 4200, sitting at that location. move.w Score,d0 walks into the house and brings back what's inside (4200 lands in d0); lea Score,a0 just writes down the house number ($FF1000 lands in a0). Confusing the two is one of the most common early mistakes in 68000 code, because both instructions are one word long and look almost identical on the page.

New here? · What is the stack, and why does the CPU need one?

The stack is a region of work RAM that grows downward and is always addressed through A7, the stack pointer. "Push" (the -(sp) form) decreases A7 by the value's size first, then writes the value at the new address; "pop" or "pull" (the (sp)+ form) reads from where A7 points first, then increases A7 by the size. This predecrement-then-write, read-then-postincrement ordering is what keeps A7 pointing at the current top item. Because it always operates on whatever A7 points to right now, the stack is naturally last-in-first-out: the most recent thing pushed is the first thing that comes back off.

That discipline is exactly what a subroutine call needs. jsr pushes a return address before it jumps, and rts pops that same address to come back, so nested calls just nest their return addresses on the stack automatically, no bookkeeping required by the programmer. One thing worth getting straight early: rts is not actually paired with jsr by any hidden record of who called. It simply pops whatever 4 bytes sit on top of the stack and jumps there. When pushes and pops are balanced, that is the caller's address; when they are not, rts jumps into your leftover data and the game crashes; and when you place an address there on purpose, you steer rts yourself, which is exactly the trick PEA's entry below pulls. MOVEM's predecrement and postincrement forms, covered below, lean on the same mechanism to save and restore whole banks of registers around a call.

Computes an effective address and loads that address itself, not the data living there, into an address register. Always a longword operation; there is no size suffix. It is the standard opening move of a routine that is about to work on something: point a register at the table, the object, or the port first, and everything that follows gets shorter and cheaper.

A PC-relative table load, right at reset, the standard position-independent way to address a local table:

	lea	SetupValues(pc),a5	; Load setup values array address.
s2disasm/s2.asm:206

And loading a hardware port address once, for repeated reuse through the register afterward:

	lea	(VDP_control_port).l,a5
s2disasm/s2.asm:704

That second example is the reason to use LEA at all. Every later write through (a5) is a 2-byte, 8-cycle instruction; spelling out (VDP_control_port).l again would cost 6 bytes and 16 cycles, every single time. Pay for the address once, and every use after that is nearly free. Whenever code is about to touch the same port, table, or structure more than once or twice, a LEA up front is the setup move that makes everything after it cheap.

Gotcha Beginners commonly confuse lea (foo).w,a0 (load the address of foo) with move.w (foo).w,d0 (load the value stored at foo); the difference between the two, "where" versus "what," is the entire point of this instruction.
Cheaper cousins PEA computes the same address but pushes it to the stack instead of loading a register. And LEA moonlights as pointer math: lea 64(a0),a0 adds 64 to a0 in 8 cycles, 4 cheaper than adda.w #64,a0 at the same size. For tiny bumps of 1 to 8, addq still wins on size; none of the An-destination adds touch the flags either way.

Teaching example · not engine code

PEA's everyday job is passing a pointer: push the address of a thing so a subroutine can find the thing itself. Here are both halves, the caller and the callee:

	pea	(Score_Digits).w	; push the ADDRESS of the score buffer
	bsr.w	DrawNumber	; call the routine
	addq.l	#4,sp	; done with the argument, discard it

DrawNumber:
	movea.l	4(sp),a0	; fetch the argument off the stack (it sits 4 bytes
			; up, because bsr pushed the return address on top)
	move.b	(a0)+,d0	; NOW the address gets used: read a digit through it
	...
	rts

Follow what the PEA actually changed: the score buffer itself is untouched, no register moved, and nothing was copied anywhere except the address, which now sits on the stack as a plain 4-byte number. It waits there, inert, until DrawNumber picks it up with MOVEA and dereferences it. That calling style is everywhere in compiled C code. Sonic-era game code passes pointers in address registers instead, which is faster, so PEA barely appears in it at all.

Which is why the one real use in the Sonic 2 disassembly is not parameter passing: it is a deliberate, commented "fake call" trick. The routine pushes the address right after itself, then branches (not calls) into shared code, so that shared code's own rts returns to the label following the PEA, exactly as if the PEA'd branch had been a subroutine call all along:

	; This is just a fancy efficient way of doing 'if true then call this, else call that'.
	pea	+(pc)
	tst.w	(Two_player_mode).w
	beq.w	CalculateVRAMAddressOfBlockForPlayer1.AbsoluteXAbsoluteY
	bra.w	CalculateVRAMAddressOfBlockForPlayer1.AbsoluteXAbsoluteY_DoubleResolution
+
s2disasm/s2.asm:20004-20009 (DrawInitialBG_LoadWholeBackground_512x256)
Gotcha This is an unusual, advanced idiom, not PEA's typical job of parameter passing, and it is worth reading twice: the PEA supplies the return address a plain bsr/jsr would have pushed automatically, so a bra can be used instead, picking between two destinations with a branch rather than duplicating the call. The consumer of the pushed address is the rts at the end of the shared code: an rts pops whatever 4 bytes sit on top of the stack and jumps there, with no idea whether a jsr or a PEA put them there. A second, structurally identical use of the same trick exists later in the same file, at line 29666.
Cheaper cousins If the address will be used more than once, LEA it into a spare register and push that with move.l An,-(sp); PEA only wins when the address is needed exactly once, on the stack, right now.

MOVEM Move Multiple Registers MOVEM.w/.l <register list>,<ea>  |  MOVEM.w/.l <ea>,<register list>

Flags Xunaff. Nunaff. Zunaff. Vunaff. Cunaff. touches no flags
Cost 8 + 8n cycles 4 bytes saving n registers in .l; the price scales with the list, not the opcode:
Full timing and size (scales with the register count)
Direction.w.l
save, registers to (An) / -(An)8 + 4n cycles8 + 8n cycles
restore, (An) / (An)+ to registers12 + 4n cycles12 + 8n cycles

n = how many registers the mask names; fancier addressing modes add their usual price on top. The size is always 4 bytes (opcode word plus register-mask word) before extension words. Do the math on the interrupt handler below: movem.l d0-a6,-(sp) moves 15 registers, 8 + 8 × 15 = 128 cycles, one of the most expensive single instructions in the whole game. That is not a flaw; 15 separate MOVEs would cost more and read worse. But it is why hot loops avoid saving registers they do not use.

Operands
Save into DnAn(An)(An)+-(An)d(An)d(An,Xn)(xx)#nd(PC)
Restore from DnAn(An)(An)+-(An)d(An)d(An,Xn)(xx)#nd(PC)

-(An) exists only when saving and (An)+ only when restoring, matching the push and pop directions. A restore may even come straight from a PC-relative table in ROM, since loading registers never needs to write anywhere.

Transfers a bitmask-selected list of registers to or from consecutive memory in a single instruction. This is the standard way to save every register a routine will clobber on entry, and restore them all on exit, without a separate MOVE per register.

Saving all fifteen working registers (d0-a6, everything except the stack pointer itself) at the very top of the vertical interrupt handler, predecrementing onto the stack:

V_Int:
	movem.l	d0-a6,-(sp)
	tst.b	(Vint_routine).w
	beq.w	Vint_Lag

And the matching restore, postincrementing back off the stack right before returning from the interrupt:

VintRet:
	addq.l	#1,(Vint_runcount).w
	movem.l	(sp)+,d0-a6
	rte

MOVEM is not limited to whole-stack saves; it also loads a fixed setup table in two passes, a word-sized group followed by a long-sized group, both postincrementing through the same pointer:

	lea	SetupValues(pc),a5	; Load setup values array address.
	movem.w	(a5)+,d5-d7
	movem.l	(a5)+,a0-a4
s2disasm/s2.asm:206-208

One subtlety in that word-sized load: movem.w into data registers sign-extends each 16-bit word to fill all 32 bits, whereas a word-sized save stores just the low 16 bits of each register. So movem.w (a5)+,d5-d7 is meant for values read as signed words; a raw bit pattern with its top bit set comes back with its high half filled with ones.

Gotcha With a predecrement destination (register-to-memory), the registers are stored in descending order, A7 down through D0, specifically so that a matching postincrement reload restores them in their original order. This is a real encoding quirk, not just a documentation note: the register-list bits mean something different for -(An) than they do for every other addressing mode, which surprises anyone reading raw opcode bytes expecting one consistent bit order. Separately, a .w load into a register sign-extends each 16-bit value to fill the full 32 bits, not zero-extends, a classic bug source when the intent was an unsigned word. And the most practical MOVEM bug of all: a routine with an early-exit path that branches past the restoring MOVEM straight to the rts. The pushes and pops no longer balance, so the rts pops a saved register's value instead of the return address and the CPU jumps into it. When a routine has more than one way out, check that every path restores exactly what the entry saved.
Cheaper cousins Saving just one register? move.l d0,-(sp) is 12 cycles against MOVEM's 16, and two plain MOVEs still tie a two-register MOVEM. The crossover is three registers; below that, reach for MOVE.

3 · Clearing and rearranging

The last four instructions don't move data between two different places at all; they reshape a value that is already sitting where it needs to be. CLR wipes a location to zero, SWAP and EXG rearrange bits or whole registers in place, and EXT changes how many bits of a register actually carry meaning, without touching memory at all.

Representative forms; the dearest, clr.l to an absolute long address, runs 28 cycles in 6 bytes. Memory forms cost more than the equivalent MOVE of a zero would suggest, because of the extra read described in the gotcha below.

Operands
Destination DnAn(An)(An)+-(An)d(An)d(An,Xn)(xx)#nd(PC)

One operand only, and no An: clearing an address register is spelled suba.l a0,a0 or a moveq/movea pair. No constants or PC-relative destinations, since nothing in ROM is writable.

Sets the destination operand to zero, in any of the three sizes. It reads as "clear," and does exactly that: whatever was there before is gone, replaced with 0. Resetting a timer, dropping a flag, stopping an object dead by zeroing its velocity: whenever a field needs to become 0 in one step, this is the tool.

Zeroing a flag byte before a batch of work runs:

loc_1072:
	clr.b	(Do_Updates_in_H_int).w
	movem.l	d0-a6,-(sp)
	bsr.w	Do_Updates

And the canonical "clear a struct field" pattern, stopping an object's velocity by zeroing both its axes directly:

	st.b	(CutScene+objoff_34).w
	clr.w	x_vel(a0)
	clr.w	y_vel(a0)
	bra.s	-

Gotcha On the original MC68000 specifically (not the later 68010 and up), CLR to a memory destination performs a read-then-write bus cycle, not a write-only one. That is invisible for ordinary work RAM, but it matters if the destination is a hardware register with a read side effect, since clearing it can silently trigger that side effect as an unwanted extra read. Later 680x0 family members fixed CLR to be write-only.
Cheaper cousins Zeroing a whole data register? moveq #0,d0 (see MOVEQ) clears all 32 bits in 4 cycles; clr.l d0 takes 6 and clr.w leaves the top half dirty. CLR earns its keep on memory and on deliberately partial clears.

SWAP Swap Register Halves SWAP Dn

Flags Xunaff. Nset Zset Vclear Cclear
Cost 4 cycles 2 bytes always.
Operands
Operand DnAn(An)(An)+-(An)d(An)d(An,Xn)(xx)#nd(PC)

A data register and nothing else; SWAP never touches memory or an address register.

Exchanges the high 16 bits and the low 16 bits of a data register, in place. There is no size suffix; it always operates on the full 32-bit register, and it only ever targets a register, never memory. Its everyday job is values that are really two 16-bit halves glued together, which on the Genesis usually means one thing: VDP command words, built half by half.

Both real uses here build a DMA command word out of two 16-bit halves computed separately, which is exactly the situation SWAP is built for. First, shaping a VRAM destination address into the top half of a DMA command:

	andi.l	#$FFFF,d2 ; command to specify destination address and begin DMA
	lsl.l	#2,d2
	lsr.w	#2,d2
	swap	d2
	ori.l	#vdpComm($0000,VRAM,DMA),d2 ; set bits to specify VRAM transfer

The same idiom again, in a different routine, moving a low-half command into position before writing the full longword to the VDP control port:

	ori.w	#vdpComm($0000,VRAM,WRITE)>>16,d0
	swap	d0
	move.l	d0,(a4)

Gotcha SWAP never touches memory, only a data register. It's a cheap way to assemble a 32-bit value out of two independently-computed 16-bit halves (compute the low half, swap it into the top, OR or add in the real low half) instead of reaching for shifted 32-bit arithmetic, which is exactly the pattern both examples above share: building a VDP command word half by half.
Cheaper cousins SWAP rearranges one register's halves; exchanging two whole registers is EXG. There is no cheaper way to reach a register's top half on the 68000; shifting right by 16 costs far more.

EXG Exchange Registers EXG Dn,Dn  |  EXG An,An  |  EXG Dn,An

Flags Xunaff. Nunaff. Zunaff. Vunaff. Cunaff. touches no flags
Cost 6 cycles 2 bytes always.
Operands
Both operands DnAn(An)(An)+-(An)d(An)d(An,Xn)(xx)#nd(PC)

Two registers in any mix (Dn,Dn / An,An / Dn,An), never memory.

Exchanges the full 32-bit contents of two registers, in any mix of data and address registers. Always a full-long swap, regardless of the "logical" size of the values held.

EXG is genuinely uncommon in hand-written 68000 game code, but not because MOVEs are better: it is uncommon because needing to swap two whole registers is rare. When a genuine swap is required, EXG beats the alternative on every axis, 6 cycles and 2 bytes with no scratch register and no effect on the condition codes, against three MOVEs through a spare register (12 cycles, 6 bytes, a register tied up, and N and Z clobbered). Across the entire 91,000-line Sonic 2 disassembly it appears only a handful of times, always as a plain two-register swap inside physics code. Here is the end of a floor-sensor routine whose contract with its callers is "the winning distance comes back in d1, its angle in d3." The excerpt runs all the way to the rts so you can see who the swap is for:

loc_1ECC6:
	move.b	(Secondary_Angle).w,d3
	cmp.w	d0,d1
	ble.s	loc_1ECD4
	move.b	(Primary_Angle).w,d3
	exg	d0,d1

loc_1ECD4:
	btst	#0,d3
	beq.s	+
	move.b	d2,d3
+
	rts
s2disasm/s2.asm:43601-43613 (Sonic_CheckFloor)

Why swap at all, instead of just using the values where they sit? Because register names are baked into the instructions. This routine ran two floor sensors and ended up with two distances, one in d0 and one in d1. If d1 already holds the smaller one, fine: fall through to the rts. If d0 won instead, the routine has a problem: every caller is already written to read the answer out of d1, and code cannot relabel registers at runtime, only move values between them. So it swaps, the winner lands in d1, and every caller stays oblivious. That is EXG's whole niche: two code paths converging on one register contract.

Gotcha Four more near-identical exg d1,d2 call sites exist elsewhere in the same disassembly, at lines 47667, 47692, 54498, and 54548, all the same register-swap idiom in different physics routines. That rarity is itself a useful signal: reach for a temp-register MOVE first, and treat EXG as a specialist tool for the rare spot where swapping two full registers in one step is genuinely worth it, not a routine replacement for MOVE.

EXT Sign Extend EXT.w Dn  (byte→word)   |  EXT.l Dn  (word→long)

Flags Xunaff. Nset Zset Vclear Cclear
Cost 4 cycles 2 bytes always.
Operands
Operand DnAn(An)(An)+-(An)d(An)d(An,Xn)(xx)#nd(PC)

A data register only. A signed byte in memory gets MOVE.b'd into a register first, then extended there; both examples below have exactly that shape.

Sign-extends the low portion of a data register to fill the rest of the register, in place. EXT.w takes the low byte and extends it to a full word; EXT.l takes the low word and extends it to a full long. You need it whenever a signed byte from object data (a radius, a height, a speed delta) is about to meet word-sized math; skip it and negative values silently turn into large positive ones.

New here? · What is sign extension, and why does it matter?

If "$FF means -1" is itself the confusing part, read the negative-number explainer back in section 1 first (the odometer story); this one builds on it.

A signed byte like -1 is stored as $FF, and a signed word holding -1 is stored as $FFFF. If you simply copied that byte $FF into the low 8 bits of a 16-bit value and left the top 8 bits zero, you would get $00FF, which is 255, not -1: a completely different number. Sign extension fixes this by copying the value's top bit (the sign bit) into every new bit added above it, so $FF extended to 16 bits correctly becomes $FFFF, still -1.

This matters constantly on the 68000 because so much game data is packed into bytes to save ROM space (an object's radius, a velocity, an angle delta), but arithmetic on that data usually needs to happen at word or long size. EXT exists specifically to do this conversion correctly; getting it wrong (or skipping it, and doing math on an unextended byte) is a classic source of "works for positive values, breaks for negative ones" bugs.

A byte radius field read off an object, extended to a word before it takes part in word-sized arithmetic:

	moveq	#0,d0
	move.b	y_radius(a0),d0
	ext.w	d0
	add.w	d0,d2
s2disasm/s2.asm:43587-43590

And a decoded code-length byte extended for use in a word-sized subtraction, inside a compression decoder's bit-reading loop:

	move.b	(a1,d1.w),d0	; get code length in bits
	ext.w	d0
	sub.w	d0,d6	; subtracted from the shift value, such that the next time around the next code is read

Gotcha There is no single-step byte-to-long form on the plain 68000: going all the way from a byte to a long takes two instructions, ext.w then ext.l. Reference material for later 680x0 chips sometimes mentions EXTB.L, a direct byte-to-long extension, but that instruction is 68020-and-up only and does not exist on the plain 68000 this page covers.
Cheaper cousins If the byte is a constant rather than data, skip the extension entirely: MOVEQ arrives already sign-extended to 32 bits.
