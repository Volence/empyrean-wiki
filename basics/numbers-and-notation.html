1 · Bits, bytes, nibbles

Everything a computer stores, from a score to a color to a letter of text, is built out of one tiny piece: a thing that can be in one of two states. Get that one idea and the rest of this page is just naming the bigger pieces you build from it.

That smallest piece is a bit. A bit is a single choice between two possibilities, and by long convention we write those two possibilities as 0 and 1. That is the whole definition: two values, nothing more. You will often hear a bit described as "one on/off wire," and that picture is genuinely useful, but be careful, because it is a mental model and not the definition. A wire can carry a bit, the same way a single bit can be held in a chip's memory, written onto a disc, or flipped from one value to the other a moment later. The wire is one place a bit can live; the bit is the piece of information itself.

One bit on its own barely says anything: yes or no, on or off. The power comes from grouping bits together, because each bit you add doubles how many different things the group can spell out. Two common group sizes have names. A byte is eight bits, and it is the everyday unit computers move around: memory is measured in bytes, files are measured in bytes, and most of the values in a game fit in one or a few bytes. Half of a byte, four bits, is a nibble (yes, really, because it is a small bite). Four bits can be arranged sixteen different ways, counting from 0000 through 1111, so a nibble has sixteen possible values. That number sixteen is not an accident you have to memorize here: a later section shows that one nibble lines up exactly with one hexadecimal digit, which is the reason the nibble is worth naming at all.

So the sizes build up in a simple ladder: a bit is the atom, a nibble is four of them, and a byte is two nibbles. These particular sizes keep coming back not because the universe demands them, but because early computers settled on the byte as a convenient chunk to handle at once, and the rest of the field agreed to keep using it. When a later page says a value is "one byte" or "two bytes," it is telling you exactly how many of these two-state atoms it takes to write that value down.

New here? · Why do computers use only two values?

Imagine you are trying to send a number down a wire to a friend using nothing but voltage, and the wire is a little noisy: the voltage sags a bit, picks up interference, drifts with temperature. If you agreed to use ten different voltage levels for the digits 0 through 9, you now have to tell a "3" apart from a "4" apart from a "5" on a signal that will not sit still, and a small wobble quietly turns one digit into its neighbor. Now agree instead on just two states, call them "clearly low" and "clearly high," with a wide no-man's-land between them. A wobble has to be enormous before a low ever looks like a high. You have traded away spelling richness for reliability, and reliability is what a machine doing billions of these a second desperately needs.

That is why modern digital computers favor two states: they are comparatively simple and cheap to build and, above all, easy to tell apart even when the electronics are imperfect. It is worth being precise, though, because this is a spot where people overstate the rule. Two states are favored, not forced. Machines have been built that store more than two levels per cell, and older analog computers represented numbers as smoothly varying quantities with no digits at all. Binary won on engineering merit, reliability and simplicity, not because a computer is physically incapable of anything else. For the rest of this wiki, though, you can safely assume two: every value you meet is spelled out in bits.

2 · Binary

You already count every day in base ten: ten digits, 0 through 9, and when you run out of them you carry into the next column. A computer counts the very same way, but it only ever has the two values a bit can hold, so it counts in base two. That is binary: the ordinary idea of digits and carrying, played out with just 0 and 1.

What makes any counting system work is place value: a digit is worth more or less depending on which column it sits in. In base ten the columns are worth 1, 10, 100, 1000, each one ten times the column to its right, because ten is where you run out of digits and have to carry. Binary carries as soon as it runs out of its two digits, so each column is worth twice the one to its right instead of ten times. Reading from the right, the binary columns are worth 1, 2, 4, 8, 16, 32, 64, 128, and on doubling forever.

To read a binary number, add up the column values wherever there is a 1, and ignore every column holding a 0. Take the pattern 0000 1010. Its columns, from the right, are worth 1, 2, 4, 8, 16, and so on; there is a 1 in the 8 column and a 1 in the 2 column, and 0 everywhere else, so the whole thing is 8 + 2 = 10. That is the entire trick: a binary number is just the sum of the place values whose bit is turned on.

But now there is a real danger of confusion. Written on its own, 1010 could be the binary number we just read as ten, or it could be the everyday decimal number one thousand and ten. The digits look identical, and only the base tells them apart, so we need a way to mark a written number as binary. Throughout this wiki we put a leading percent sign on a binary number: %00001010 is unmistakably the binary value, ten, and never one thousand and ten.

Notation The number in base two is real and true of every digital computer: it is how the machine actually counts. The way you mark a written number as binary, though, is a convention we chose. This wiki and its assembler use a leading %; many programming languages instead use a leading 0b, and would write the same value as 0b00001010. Same number, different spelling. The processor itself never sees either mark: it only ever holds the raw bits, and the prefix is purely for us humans reading and writing the code.

Counting up in binary works exactly like an odometer whose wheels carry only the digits 0 and 1. Start at %0 and add one at a time. First %0, then %1, and now the rightmost wheel is out of digits, so it rolls back to 0 and carries a 1 into the next column: %10, which is two, not ten. Keep going and you get %11 for three, then a carry into a fresh column gives %100 for four, %101 for five, and so on. Every time you fill a column you carry to the left, and because there are only two digits it happens twice as often as in decimal, which is why binary numbers grow long so quickly.

New here? · Why does each column double, and why does that cover every number?

Picture a kitchen balance scale and a box of weights, but a strange box: the weights come in 1 gram, 2 grams, 4 grams, 8 grams, 16 grams, and so on, each one double the last. You weigh something by piling weights on the empty pan until the scale balances. Here is the neat part of a doubling set like that: every whole number of grams can be made in exactly one way, and for each weight you only ever face a yes-or-no question, do I put this one on the pan or not?

Say the object weighs ten grams. The 16 alone would tip the scale too far, so it stays off. The 8? Put it on. That leaves two grams still to find. The 4 is too much, so leave it off. The 2? Put it on, and now the scale balances, so the 1 stays off as well. Your answers for the 8, 4, 2, 1 weights were yes, no, yes, no, and 8 + 2 is exactly ten. Write 1 for each yes and 0 for each no and those answers are the bits of the binary number for ten: %1010. Binary is not a strange new way to count. It is the smallest possible set of digits, just two, and a bit in each column is nothing more than that same yes-or-no: is this doubling amount part of the total?

This is the payoff of the sizes from section 1. Because a byte is eight bits, it has exactly these eight columns, 128 down to 1. Turn every bit off and the total is 0; turn every bit on and you add 128 + 64 + 32 + 16 + 8 + 4 + 2 + 1 = 255, the largest value a single byte can hold. So one byte counts from 0 to 255 and no further, a range we come back to and pin down properly in section 4.

3 · Hexadecimal

Binary is how the machine counts, but a long row of ones and zeros is miserable for a person to read. Hexadecimal is the shorthand programmers actually write, and it is built to line up with binary perfectly.

Decimal has ten digits and binary has two; hexadecimal has sixteen. It counts in base 16, which means each column runs through sixteen values before it carries, instead of ten or two. The trouble is that we only have ten number symbols, 0 through 9, and base 16 needs six more. So hex borrows the first six letters of the alphabet: after 9 comes A for ten, B for eleven, C for twelve, D for thirteen, E for fourteen, and F for fifteen. Those letters are digits here, not text, and F is simply the hex way to write fifteen. Count one past F and, just like decimal rolling from 9 to 10, hex rolls over to 10, its next carry, which is worth sixteen.

New here? · Why letters, and why exactly these six?

A counting system needs one distinct symbol for every value a single column can hold before it carries. Decimal holds ten, so it uses ten symbols, 0 to 9. Base 16 has to hold sixteen values in one column, so it needs sixteen symbols, and after 0 to 9 it is six short. Nothing about a computer requires those six extras to be letters; the people who settled hex just reached for the symbols already sitting on every keyboard and in every character set, and the letters A to F were right there, in order, easy to remember.

So read A to F as the numbers ten to fifteen and nothing more. C is not a letter that snuck into a number; in hex it is the number twelve, the same way 7 is the number seven. The confusing part is only that we reused shapes we already knew as letters. Their meaning here is purely numeric.

Here is the real reason hex is everywhere, and it is worth reading slowly, because it is the whole point. Sixteen is two multiplied by itself four times, so one hex digit covers exactly the sixteen values that four bits can spell out. Four bits is a nibble, the half-byte you met in section 1, so one hex digit is exactly one nibble, no rounding and no leftovers. That makes hex a clean stand-in for binary: chop a binary number into groups of four bits and rewrite each group as its single hex digit.

Take the running byte from section 2, %0101 1010, which we worked out as 90. A byte is eight bits, so it is two nibbles, and the two nibbles turn into two hex digits:

That is the mapping in full: the left nibble 0101 is 5, the right nibble 1010 is A (ten), so the byte is $5A. And $5A as a plain number is 5 times 16 plus 10, which is 90, exactly what the binary read out. One byte, always two hex digits, one digit per nibble. Once you can see the two nibbles, you can read the byte's hex straight off the bits without ever doing decimal arithmetic, which is precisely why people who stare at bytes all day reach for hex.

There is a smaller, second convenience worth naming, but keep it in its place: hex is just far shorter and easier to hold in your head. $5A is two characters where %0101 1010 is eight, and "five-A" is an easier thing to say aloud and remember than a string of ones and zeros. Handy, and real, but it is the readability bonus riding on top of the true reason. The nibble mapping above is why hex won.

Notation Base 16 is a real idea, true of every computer: it is a genuine way to count, the same sixteen values however you spell them. The way you mark a written number as hexadecimal, though, is a convention people picked. This wiki, this wiki's assembler, and much 68000 documentation put a leading $ on a hex number, so ninety is $5A. The C language instead writes a leading 0x and would spell the same value 0x5A; some other assemblers use a trailing h and write 5Ah. Same number, different spelling. The processor never sees any of these marks: it only ever holds the raw bits, and the $ is purely for us humans reading and writing the code.
Gotcha The trap that catches everyone once: $10 is sixteen, not ten. Your eye reads the shape "10" as the number ten out of pure habit, but the $ says this is hex, and in base 16 the columns are worth 1 and 16, not 1 and 10. So $10 is one sixteen and zero ones, which is sixteen. It is the same rollover you already know: decimal writes 10 when it runs out of digits at ten, binary writes %10 when it runs out at two, and hex writes $10 when it runs out at sixteen, one past $F. When you see a $, stop reading the digits as decimal.
New here? · How do I turn a byte into hex by hand?

You never have to juggle all eight bits at once, and that is the whole gift of the nibble mapping. Split the byte down the middle into its two nibbles and deal with four bits at a time. Take %01011010. Cut it into a left nibble 0101 and a right nibble 1010. Now read each nibble on its own as a small binary number: 0101 is 4 + 1 = 5, and 1010 is 8 + 2 = 10. A single nibble only ever lands between 0 and 15, and that is exactly the span of one hex digit, so turn each into its digit: 5 stays 5, and 10 is A. Read them back in the same order, left then right, and the byte is $5A.

That is the entire method, and it never gets harder: left nibble to a hex digit, right nibble to a hex digit, done. Going the other way is just as mechanical. Take $5A, expand each digit into its four bits (5 is 0101, A is 1010), and stick them back together for %01011010. The tool below does exactly this, so once you have tried a couple by hand you can check yourself against it.

Three spellings, one number: $5A, %0101 1010, and 90 are the same byte written three ways. The surest way to make that click is to convert a few yourself and watch all three move together, which is exactly what the tool below lets you do.

4 · Place value and range

Section 2 showed a byte counting from 0 to 255, and section 1 promised that every bit you add doubles how many things a group can spell out. This section makes that promise exact, because it is the one rule the next section leans on completely. Here it is, the rule this whole cluster of pages rests on, worth memorizing as the 2n rule: n bits give 2n different patterns. That is the whole of it. Everything about ranges and sizes below is this single line, applied.

Build the intuition from the bottom. One bit has two possible patterns, 0 and 1: that is 2 patterns, and 21 is 2. Add a second bit and each of those two patterns can be followed by either a 0 or a 1, so the count doubles to 4: 00, 01, 10, 11, and 22 is 4. Add a third bit and it doubles again to 8, a fourth and it is 16. Each new bit doubles the count because it splits every pattern you already had into two, the old pattern with a fresh 0 on the end and the same pattern with a 1. Doubling starting from 1, n times over, is exactly what 2n means, so n bits give 2n patterns.

Teaching example · not engine code

Take 3 bits and write out every pattern in counting order, and you can check the rule by hand. There are 23 = 8 of them, no more and no fewer:

000 001 010 011 100 101 110 111

Count them: eight patterns. Notice the shape of the list, the first four all start with 0 and the last four all start with 1. That is the doubling caught in the act: the four 2-bit patterns each appear once with a leading 0 and once with a leading 1, so adding the third bit turned 4 patterns into 8.

Now spend the rule on the byte. A byte is 8 bits, so it has 28 = 256 patterns, and each pattern is one value the byte can hold. Section 2 read those values straight off the place columns: all bits off is 0, all bits on is 255. So an unsigned byte holds the 256 values from 0 to 255, written $00 to $FF in hex, and not one value more.

Two hundred fifty-six patterns, but the largest value is 255, not 256. That gap of one trips up nearly everyone the first time, and it is worth pinning down rather than waving past. The count of values and the largest value are two different numbers, and they differ by one for a single reason: the counting starts at 0, not at 1.

New here? · Why is the biggest byte value 255 and not 256?

Because 0 is one of the values. Picture a short street with 256 houses on it, but the builder numbered the first house 0 instead of 1: house 0, house 1, house 2, and on. By the time you have counted out all 256 houses, the last one is numbered 255, because you spent one of your numbers, the very first, on 0. There are still 256 houses; the highest house number is just one less than the count, since the count includes the zero.

The byte works the same way. Its 256 patterns are the values 0, 1, 2, all the way up, and because the list starts at 0 the 256th and last value is 255. So "256 patterns" and "0 to 255" are not in conflict, they are the same fact said two ways: 256 values, the first of which is 0, ending at 255. Any time you meet a range that starts at 0, expect the top to be the count minus one: 28 = 256 patterns means 0 to 255, 216 patterns means 0 to 65,535, and so on.

Nothing about the rule was special to 8. Feed it a wider group of bits and the range grows fast, because 2n doubles at every step. Sixteen bits give 216 patterns, which is 65,536 values, so 0 to 65,535. Thirty-two bits give 232, over four billion values. The next section lines these widths up in a table and gives each one its name, so treat this as the trailer: the same rule you just learned, run at 16 and 32 bits.

Powers of two, and where "64KB" comes from

One more payoff of the rule explains a number you will meet everywhere in this material: sizes like 64KB. Because memory is addressed with bits, the round numbers of computing are the powers of two, not the powers of ten. The one worth carrying in your head is 210 = 1024, ten bits' worth of patterns, which lands tantalizingly close to 1000 without being 1000. That near-miss is the whole story behind the kilobyte.

Here honesty matters, because this is a spot where casual explanations quietly fib. Strictly, "64 kB" with a lowercase metric k can mean 64,000 bytes, since in the metric system kilo means exactly 1000. The amount the hardware actually deals in is 64 times 1024, which is 65,536 bytes, and the exact name for that quantity is 64 KiB, 64 kibibytes. Retro computing, and this wiki, write "64 KB" to mean that second thing: 64 times 1024 = 65,536 bytes. Read it that way on these pages, but hold it as a convention this field settled on and not a law of arithmetic, because "64 KB" written somewhere else is not guaranteed to mean exactly 65,536.

So the powers of two are not decoration. 256, 1024, and 65,536 keep turning up because each one is 2 raised to a whole number of bits: 256 is 28, 1024 is 210, 65,536 is 216. Whenever a count comes out to exactly 2n, it is the number of patterns that n bits can make, which is why these particular values sit at the round-number spots with nothing tidier nearby. It is the same 2n rule wearing a different hat.

5 · Widths: byte, word, long

Now apply that rule to the sizes you will actually see. A byte is one handy size, but plenty of values will not fit in one: a score climbs past 255, a position on a big level runs into the thousands, an address that points anywhere in memory needs far more room still. So bytes get grouped into larger fixed sizes, and just three of them cover almost everything in this material. Each one is built by doubling the last, which is exactly the move the 2n rule rewards.

A byte you already know: 8 bits, and that meaning is nearly universal. The larger sizes, though, are named relative to a particular processor. On the 68000, the CPU this wiki is about, a word is two bytes, so 16 bits, and a long word is two words, four bytes, so 32 bits. That is the whole ladder, byte to word to long word, each rung twice as wide as the one below it. One caution worth carrying: the word "word" is not fixed across the industry, other processor families use it for other sizes, so throughout this wiki "word" always means the 68000's 16 bits and "long word" its 32. When a page here says a value is "a word," it means precisely 16 bits, no more guessing.

Feed each width into the 2n rule and its range drops straight out, no new idea required. A byte is 8 bits, so 28 = 256 patterns, the values 0 to 255. A word is 16 bits, so 216 = 65,536 patterns, the values 0 to 65,535. A long word is 32 bits, so 232 patterns, whose top value is 4,294,967,295, a shade over four billion. Nothing here is a fresh fact to memorize: it is the single rule from the last section, run at three different widths.

The hex column lines up with the bit count in a handy way. Because one hex digit is exactly one nibble, four bits (section 3), a value that fills a byte needs 2 hex digits, a word 4, and a long word 8. Writers lean on this: padding a number out with leading zeros, writing $00FF or $000000FF, is a common way to signal to a reader which width is intended. But treat those digits as a hint, not the fact, because leading zeros change nothing about the value. $1, $01, $0001, and $00000001 are all the exact same number, one. What actually decides how many bits get stored or worked on is never how the number is written down but the instruction or data directive that uses it, which is the subject of the next note.

New here? · Why not just always use the biggest width?

You could. A long word holds every value a byte or a word can hold and far more, so if space were free you would reach for it every time and never think about widths again. But space is not free. Every value a program keeps has to live somewhere in memory, and a long word takes four times the room of a byte. On a machine of this era memory is genuinely scarce, so spending four bytes to remember a number that never climbs above 255 is four times the storage for no extra reach, multiplied across the thousands of little values a game juggles at once.

So the habit is the opposite of "always go big": pick the smallest width that comfortably fits the values you expect. A flag that is only ever on or off wants a byte, or even a single bit. A ring count that could reach a few hundred wants a word. An address that can point anywhere in a large memory wants a long word, because nothing narrower can name every location. Choosing the width is choosing exactly how much room a value is worth, and picking well is a quiet part of writing tight code.

Notation Grouping bits into fixed sizes is general, but the exact sizes and their names belong to a processor. On the 68000 you choose a size for an operation and write it as a short suffix: .b for byte, .w for word, .l for long word. That suffix notation is standard 68000 assembly, the same on any 68000 assembler, not a private habit of this wiki. And it is more than a note to the assembler. The text .w does disappear when the code is assembled, but the size choice does not: the assembler bakes it into a size field inside the instruction's opcode, and the processor reads that field and acts on it, touching one, two, or four bytes accordingly. So the written suffix is notation for us to read, but the size it selects is real, and the CPU genuinely sees it. Where the suffix rides on a real instruction, and what changes when you swap one for another, is the 68000's story: the 68000 track (see the 68000 Programming Model page) shows how these size suffixes attach to real instructions.

6 · Signed or unsigned

Every range so far started at 0 and climbed: a byte holds 0 to 255, a word 0 to 65,535. But programs also need negative numbers, a velocity that points left, a score that just dropped. The bits have no room to spare, so the machine reuses the ones it already has, and here is the whole idea: the same pattern of bits can stand for different numbers depending on how it is read. Take the byte %11111111, which section 2 read as 255. Read that pattern as unsigned and it is 255; read the very same pattern as signed and it is -1. The bits do not change. Only the interpretation does.

Nothing in the byte says which reading is meant, so the choice belongs to the program, not the byte. And most operations never have to make it: adding, subtracting, and copying produce the identical bits whether you picture the operands as signed or unsigned, so a plain add serves both and never needs to know. The signed-or-unsigned decision only surfaces where it genuinely changes the answer, a signed versus unsigned multiply or divide, stretching a small value into a wider one, or which way a comparison branches. Those come as separate signed and unsigned instructions on the 68000, and the Moving Data page covers them; the point here is only that the bits carry no sign of their own, and the program's intent supplies it.

The scheme almost every computer uses to make this work, the 68000 included, is called two's complement, and its universal shape is worth owning right here. In an n-bit signed value the top bit is not merely a flag that says "negative"; it carries negative weight. A byte's eight columns are normally worth 128, 64, 32, 16, 8, 4, 2, 1; read as signed, that top column flips to minus 128 while the lower seven stay positive. So the most negative byte is %10000000, just the top bit, which is -128, and the most positive is %01111111, the lower seven, which is +127. That is the pattern at any width: n bits run from -2n-1 up to +2n-1 - 1. A byte is -128 to 127; a word is -32,768 to 32,767.

Two patterns are worth memorizing. $80 is %10000000, only the top bit set, so as a signed byte it is exactly -128, the most negative value. $FF is %11111111, every bit set, which is -128 + 64 + 32 + 16 + 8 + 4 + 2 + 1 = -1, the pattern we started with. There is also a shortcut for any signed byte whose top bit is set: read it as an ordinary unsigned value and subtract 256. $FF is 255 unsigned, and 255 - 256 = -1; $80 is 128 unsigned, and 128 - 256 = -128. The subtract-256 trick is just the negative-weight idea in disguise, because turning the top column from +128 into -128 is a swing of exactly 256.

New here? · How can the same byte be 255 and -1?

Because "255" and "-1" are two ways of reading one pattern, not two different patterns. A byte is eight bits sitting there; on its own it means nothing until some instruction reads it, and the reading picks unsigned or signed. Under the unsigned reading of %11111111 the top bit is worth +128, so the total is 128 + 127 = 255. Under the signed reading that same top bit is worth -128, so the total is -128 + 127 = -1. Nothing moved; only the weight of the top bit changed sign.

Carry one habit away from this: whenever you see a byte with its top bit set, anything from $80 through $FF, it is a candidate negative number, and only the code around it settles whether that is what is meant.

One last piece of the universal picture, and an elegant one: two's complement has a single zero. The tempting alternative is to steal the top bit as a plain minus sign, but that leaves you with two zeros, a +0 and a -0, which wastes a pattern and complicates the arithmetic. Two's complement instead spends every pattern on a distinct number, so a byte's 256 patterns become the 256 distinct values -128 through 127 with exactly one zero among them. That tidiness, one pattern per number and ordinary addition that just works across the sign, is a big part of why the scheme won.

That is the whole universal picture: a fixed width, the top bit carrying negative weight, one zero, and a range of -2n-1 to 2n-1 - 1. What is left is specific to the chip: how the 68000 records, after each operation, whether a result went negative or overflowed (its condition flags), and which instructions pick the signed or the unsigned reading. Those live on the 68000 track: see the Moving Data page's negative-numbers explainer.

7 · Letters are numbers too

A byte can hold any value from 0 to 255, and so far we have read those values as counts: a score, a position, a velocity. But a letter of text is stored in a byte too, and the trick is simple once you see it: somebody drew up a table that pairs each letter, digit, and symbol with a number, and from then on a letter is just that number sitting in a byte.

The table this wiki's examples use is ASCII. In it the capital letter A is 65, which written in hex is $41. The next letter, B, is 66; C is 67, and on up the alphabet. Digits get numbers too (the character 0 is 48, not the value zero), and so does every space, comma, and bracket. Once every character has a number, a whole line of text is nothing more exotic than a run of bytes, one byte per character, each holding that character's number from the table.

ASCII is a foundational, broadly used standard, but it is worth being exact about what it is and is not. It is not a law of computing the way base two is: nothing forces a machine to store A as 65. It is a table people agreed on, one that caught on so widely that it underlies much of what you will read here and elsewhere. Other encodings exist and always have, and modern text mostly uses Unicode, a far larger scheme built to cover the writing of the whole world rather than one keyboard's worth of Latin letters. So read ASCII as the encoding this wiki's examples assume, a common and important one, not as the single way every computer everywhere stores text.

New here? · If letters are numbers, how does the computer know 65 is 'A' and not the number 65?

It does not, and that is the honest answer. Nothing in the byte says whether it is a letter or a count. The byte holding 65 is exactly the same eight bits whether you meant the character A or the plain number sixty-five. What decides is the context: the instruction that touches the byte. Code that is about to draw text on screen reads that 65 through the ASCII table and paints an A; code that is adding up a score reads the same 65 as the number and adds it. The byte never changes. The reading does.

You have already met this exact idea one section back. There, %11111111 was 255 read as unsigned or -1 read as signed, and the bits carried no built-in answer, only the code's intent settled it. This is the same lesson wearing different clothes: same bits, different meaning, and the surrounding code is what commits to one. "Is this byte a letter or a number?" has no answer you can find by staring at the byte, exactly as "is this byte positive or negative?" did not.

There is a real place in this wiki's own code where a value that looks like a plain number is, byte for byte, a word of text. On the later Genesis models a lock called the TMSS guards the video hardware: shortly after boot, the running game must write the letters S, E, G, A to a specific hardware register at $A14000, or the very next attempt to touch the video chip freezes the machine. Here is the exact line that does it:

        move.l  #$53454741, (TMSS_REGISTER).l   ; "SEGA"
engine/system/boot.asm:32

The single hex value $53454741 is a long word, four bytes. Split it into those bytes, most significant first, and they are $53, $45, $47, $41. Now look them up as ASCII: $53 is 83, the letter S; $45 is 69, E; $47 is 71, G; and $41 is 65, the A we started with. So the number $53454741 is not really a number here at all, it is the word SEGA spelled one byte per letter. When the 68000 stores that long word it writes the $53 first, at the lowest address, because the 68000 is big-endian: it puts the most significant byte first, which is exactly why the bytes land in reading order S, E, G, A rather than reversed. The register is watching for precisely those four bytes, and the comment ; "SEGA" is the programmer telling you what the digits secretly say.

One clarification, because the word SEGA does double duty here. A TMSS console actually looks for SEGA in two separate places. At power-on, before it will boot the cartridge at all, the system firmware checks that the ROM's header contains the text "SEGA". That header check is what authorizes the boot. The write above is the second, later step: the game, now running, writes "SEGA" to the $A14000 register to unlock the video hardware. The line of code shown is that video unlock, not the initial header check.

Notation The idea that letters are numbers is general: any encoding is just a table pairing characters with values, and ASCII is one widely used table. What is particular to this project is the line above, the way we write the value with a leading $, and the choice to pack four letters into one long word to feed the console's check. That line is not universal notation you will find in every computer; it is a real spot in this wiki's code where the general idea shows up wearing this project's conventions. The same letters live in a plainer form elsewhere too: the ROM header field GAME_CONSOLE equ "SEGA GENESIS " (games/sonic4/config/game.asm:4) is another run of bytes that is really text, this time written as characters between quotes and left for the assembler to turn into their ASCII numbers.

That is the whole of it. Give every character a number through an agreed table, store each number in a byte, and text becomes just data, a run of bytes like any other. Whether a given byte is a letter or a count is never written in the byte itself; the code that reads it decides, the same way it decided signed from unsigned in the last section.
