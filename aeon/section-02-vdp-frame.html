The VDP & the Frame

Everything a player sees comes from one chip: the VDP, walking its own private memory sixty times a second. This page explains what that chip actually is, then follows one frame all the way through the engine, from the moment the game loop finishes work to the moment pixels leave for the television. Verified line-by-line against engine/system/ and engine/level/.

1 · What the VDP draws with

The VDP never sees pixels arrive from the CPU one at a time. Instead it owns 64KB of private memory and, sixty times a second, walks that memory and composites a picture out of four kinds of thing living inside it: tiles, nametables, palettes, and sprites. Everything else in this page, the queues, the buffers, the interrupts, exists to get bytes into that memory in the right shape, at the right time.

New here? · What is a tile, really?

A tile is the Genesis's atomic unit of picture: an 8×8 block of pixels, and nothing the VDP draws is any smaller than one. Each pixel in a tile is 4 bits, not a full color, a number from 0 to 15 that means "slot 15 of whichever palette this tile is told to use." That indirection is the whole trick: the same 32 bytes of tile data can look like four different things depending which of the four 16-color palettes gets attached to it at draw time.

32 bytes per tile (8 rows × 8 pixels × 4 bits, packed two pixels per byte) is small enough that the unified art pool holds well over a thousand of them, resident, permanently, in VRAM. There is no "load this sprite's graphics" step during gameplay the way a modern engine streams a texture; the tile is already sitting in video memory, and drawing it is just telling the VDP which slot to read.

A nametable is the map that turns loose tiles into a picture: a grid of 16-bit words, one per 8×8 cell, where each word names a tile index and four small flags riding along with it. A plane is one such grid, scrollable as a whole; Aeon uses two, Plane A and Plane B, each a full 64×64-cell map in VRAM. Nothing about a nametable word touches pixels directly; it is a pointer plus some paint instructions, and the VDP does the actual pixel lookup every scanline as it draws.

Fifteen bits, laid out high to low: priority (draw above or below the other plane and most sprites), a 2-bit palette select (which of the four 16-color palettes), vertical flip, horizontal flip, and an 11-bit tile index (0-2047, which is exactly why the unified art pool tops out where it does). New here? · What is a palette, and why only four?

The Genesis can show 512 possible colors (9-bit color: 3 bits each of red, green, blue), but never more than 64 on screen at once, held in a small dedicated memory called CRAM. Those 64 slots are split into four palettes of 16 colors each, and every tile, every sprite, picks exactly one of the four for its lifetime on screen. Slot 0 of any palette is transparent (or the backdrop color, for a whole-plane background), which is why a 4-bit tile only really shows 15 colors.

Four palettes sounds tight, but it buys something modern engines pay dearly for: instant, zero-cost palette swaps. Recolor Sonic's shield by rewriting 16 bytes of CRAM, not a single pixel of tile data. The wiki's own palette story lives on the Art & Compression page.

Sprites are the fourth ingredient, and they are built from the same tiles as the planes but positioned independently, one entry per sprite in the Sprite Attribute Table (SAT) in VRAM: up to 80 on screen, each naming a position, a size in tiles, and a starting tile index plus the same palette/flip/priority bits a nametable word carries. The SAT's anatomy, byte by byte, belongs to the Object System page; this page cares about it only as one more thing that has to reach VRAM every frame, and reliably.

2 · Two ports and a shadow table

The CPU cannot address VRAM, CRAM, or the VDP's 24 configuration registers directly. Every access goes through two 16-bit mail slots at fixed addresses: the data port (where bytes and words actually flow) and the control port (where you first say where those bytes are going, or, using a different write pattern, which register to set). A "register write" is really a magic word sent to the control port; there is no address named "register 5", only the convention that a control-port word shaped like $85xx means "put xx in register 5".

New here? · Why can't the CPU just read VDP memory directly?

On a modern computer, video memory is just another region of the same address space; the CPU reads and writes it like anything else. The Genesis's VDP is a separate chip with its own 64KB of memory wired to nothing but itself, so the 68000 physically cannot point at a VRAM address and read a byte the way it reads work RAM. The only door in is two narrow ports: tell the control port where you want to go and what kind of access, then push bytes through the data port.

That narrow door is why so much of this page exists. Every technique described here, the shadow table, the DMA queue, the deferred plane buffer, is ultimately about getting bytes through that one door as efficiently and safely as possible, because there is no other way in.

That convention has a sharp edge: VDP registers are write-only. The chip never lets the CPU ask "what's in register 1 right now?" So Aeon keeps its own answer in RAM, a 19-byte VDP_Shadow struct mirroring registers $00-$12 (the five DMA registers $13-$17 are per-transfer state the DMA queue owns, not settled configuration, so they are deliberately not shadowed):

Game code writes registers through a SetVDPReg macro that updates the shadow and sets a bit in a dirty mask; once per VBlank, Flush_VDP_Shadow walks all 19 fields and writes only the ones whose dirty bit is set. On a typical gameplay frame zero to three registers actually change, so the flush skips sixteen-plus writes it would otherwise have to make blind.

Decision The shadow is the only sanctioned path for settled state, with two narrow exceptions. A raw control-port write to a shadowed register is legal exactly twice: transient pre-DMA setup (the level streamer's auto-increment tweak right before a transfer, where nothing reads the value back as authoritative), and mid-frame raster handlers, which may hit registers directly because that is the entire point of a raster effect, but must never touch the shadow or dirty mask, since the next VBlank flush would just re-assert the old value over the raster write. Everything else goes through SetVDPReg.

3 · The frame and its sliver

A television doesn't receive a picture all at once; it receives one scanline at a time, top to bottom, and then the beam has to fly back up to start over. On NTSC, 224 of 262 lines per frame are actually visible; the remaining 38 are dead time from the display's point of view, but they are the only window where the CPU can safely rewrite video memory without the change tearing across a half-drawn picture. That window is called VBlank, and almost everything on the rest of this page is about using it well.

New here? · What actually happens if you write to VRAM outside VBlank?

The VDP is reading from its own memory to build the picture at the exact same moment the CPU would be trying to write to it, since the chip is drawing continuously, line after line, the whole time the screen is lit. A write that lands mid-scanline doesn't crash anything, but it can change a tile or a nametable entry the VDP is about to read for the very next pixel, so the result is visible on screen as a fleeting tear or glitch, a texture that flickers wrong for one frame and then looks fine again.

VBlank sidesteps the whole problem by being the one moment nothing is being drawn at all. Writes that land there are guaranteed to be visible whole, never half-applied, starting with the very next frame. That guarantee is why this page treats the 38-line sliver as sacred ground and builds an entire queueing system just to make sure everything that needs to reach video memory fits inside it.

The sliver is not idle time to fill with anything convenient. It is the one place all of section 5's DMA queue, section 6's dirty buffers, and section 7's plane buffer are required to run, every single frame, or the player sees the consequence directly: a half-updated screen, or a frame that simply never got its work done at all (section 4's lag frame).

4 · The VBlank pipeline

IRQ6 fires once per frame, at the start of the sliver, and VBlank_Handler is the first thing that runs. It saves every register, checks one flag, and picks one of exactly two paths:

VBlank_Handler:
        movem.l d0-a6, -(sp)
        tst.b   (VBlank_Ready).w
        beq.s   .lag
        movea.l (VInt_Ptr).w, a0
        jsr     (a0)
        bra.s   .done
.lag:
        bsr.w   VInt_Lag
engine/system/vblank.asm:8

VBlank_Ready is the main loop's promise: "I finished this frame's work before you fired." When that promise holds, the handler dispatches through VInt_Ptr, which gameplay leaves pointed at VInt_Level, the full pipeline. When the promise doesn't hold, a lag frame, the handler runs VInt_Lag instead, a reduced path covered below.

VInt_Level runs in one fixed order, and the order is load-bearing: each stage either produces something the next stage depends on, or is the exact point where visual correctness would break if it moved.

Two things about that order are easy to guess wrong. VSRAM (step 6) runs after Critical DMA, not before, because the write must follow the HScroll DMA it is paired with. And the plane buffer drain (step 4) runs before Critical DMA, not after, simply because nothing later in the list depends on it and nothing about the plane buffer waits on DMA either; putting visual-stability work first is the whole ordering philosophy of this handler.

New here? · What's a "lag frame", and why does it need its own code path?

Every VBlank is a deadline. Most frames, the main loop finishes its physics, camera, and object work well before the VDP starts its blanking interval, and by the time IRQ6 fires, everything is ready to draw. Sometimes the main loop is still working when VBlank arrives anyway, a heavy frame, too many objects, a slow decompression, and that is a lag frame: the deadline was missed.

Firing the full pipeline anyway would be actively dangerous, not just wasteful. The main loop might be mid-write into a buffer the pipeline is about to read, and draining a half-written buffer sends garbage to the VDP. So the interrupt handler checks a flag the main loop sets only when it's actually done, and runs a smaller, safer pipeline instead when that flag isn't set. The player still gets a frame; it's just a frame that skips exactly the parts that would be unsafe to run early.

VInt_Lag is not a do-nothing stub; it is nearly the whole pipeline with two steps removed. It still runs the shadow flush, the dirty-buffer enqueue, Critical DMA, and the VSRAM write, so palette and sprite updates never visibly stall on a lag frame. Controllers still read and the frame counter still advances. What it omits is exactly the pieces that are unsafe to run against a main loop still mid-write: the plane-buffer drain (section 7 explains why draining a half-written buffer is actively dangerous, not just wasted work) and the budget-gated Important and Deferrable DMA drains, which simply persist to the next normal frame instead.

Decision Lag_Frame_Count is the real performance meter, not a cycle profiler. A cycle-counting profiler tells you a routine got slower; it does not tell you whether that slowdown ever cost a frame. Lag_Frame_Count (debug builds only, ifdef __DEBUG__, incremented at the tail of VInt_Lag) counts the one number that actually matters: how many times the deadline was missed. Zero lag frames during a worst-case diagonal fast-scroll is the real pass condition for any streaming or rendering change, not a cycle budget on paper.

A lag frame leaves one hazard behind that took real debugging to find and fix. Because VInt_Lag also sets VBlank_Flag, the main loop's wait routine (VSync_Wait) has to clear any stale flag before it starts waiting, and the clear-flag / raise-VBlank_Ready pair has to land as a single atomic act. If a VBlank fires between those two writes, with Ready still 0, it runs VInt_Lag, which sets the flag, so the wait falls straight through with no real sync at all, while Ready is left armed for the next VBlank to fully dispatch VInt_Level and drain a plane buffer the main loop is still mid-fill on. This is exactly the kind of Genesis-timing gotcha that only shows up under load, and the fix (commit b96c861) is a few dozen cycles of interrupt masking around the pair, on a once-per-frame path where the cost is free:

        move.w  sr, -(sp)
        move.w  #$2700, sr             ; mask: the pair below must be atomic
        move.b  d0, (VBlank_Flag).w     ; clear any stale lag-frame flag
        move.b  #1, (VBlank_Ready).w    ; this frame's work is done
        move.w  (sp)+, sr               ; unmask BEFORE waiting; IRQ6 sets the flag
.wait:
        tst.b   (VBlank_Flag).w
        beq.s   .wait
engine/system/vblank.asm:182

It is a small fix precisely because the underlying design was already close: only the clear-and-arm pair needed to be atomic, not the whole wait. That is the value of building the lag path deliberately instead of bolting it on afterward.

5 · The DMA queue

Steps 5, 8, and 9 of the pipeline above all drain from the same structure: a 32-slot, three-priority DMA queue that the main loop feeds all frame long and VBlank empties in one pass. Nothing in the game loop ever DMAs directly; it enqueues a request (source, destination, length) and moves on, so a slow VBlank never stalls gameplay logic, only graphics delivery.

New here? · Why interleave register markers into the data at all?

The VDP's DMA-source registers ($13-$17) have to be set one byte at a time, in a specific order, before a transfer can start, and normally that means five separate instructions per transfer just for setup. The movep instruction is a 68000 oddity built for exactly this: it moves a register's bytes to memory (or back) at every other address, skipping one byte between each. Interleave a VDP register-select marker into the gaps of a data structure, and one movep can write the marker bytes and the payload bytes in a single instruction, because they were never really separate in memory to begin with.

The entry format on this page borrows that trick directly (credited to Flamewing's Ultra DMA Queue design): the register markers are baked in once at boot, and the drain loop below never has to think about them again, just fire pre-assembled words at the VDP's control port.

The three drain routines share the goal but not the mechanism, and the split is deliberate:

Decision Hybrid drain: unrolled jump table for Critical, address-compare loop for the rest. Process_DMA_Critical converts the slot pointer into a byte offset and jumps straight into a fully unrolled sequence of move.l/move.w per slot, zero comparisons per entry, about 514 cycles to drain all 8. Important and Deferrable share Drain_Budgeted_Queue, a loop that checks DMA_Budget_Remaining before each entry and stops by comparing the current pointer against the queue's first-free address, not a dbf-counted loop. Unrolling all 32 slots the Critical way would cost 704 bytes of ROM; the hybrid costs roughly 280, putting the fully-unrolled speed only where every cycle affects visual stability, and a compact shared loop where one extra frame of latency is tolerable.

A shrink-after-a-lag-frame scheme and a temporary post-lag boost to Deferrable were both part of the original design; neither is built. This is a real gap, not a secret one: a self-tuning budget remains future work, noted here rather than claimed as shipped.

The packer below lets you feel the budget directly: it's a fixed-size box, and real transfers either fit or they spill.

Two safety details round out the enqueue path, both in QueueDMATransfer, the shared core all three QueueDMA_* entry points fall through to. First, a RAM source address ($FF0000+) can set bit 23 after being shifted for the VDP, which the chip would misinterpret as a VRAM-to-VRAM copy flag instead of a normal 68K-to-VDP transfer; the routine clears that bit explicitly. Second, the whole enqueue runs with interrupts masked, so a VBlank landing mid-write into a queue slot can never tear an entry.

Decision 128KB boundary splitting is implemented, not deferred. The VDP's DMA source address only auto-increments its low 17 bits; a transfer that crosses a 128KB boundary wraps within the same block and corrupts. QueueDMATransfer detects the crossing with a sub+sub carry check and, on a hit, splits the request into two queue entries in the same sub-queue: the first finishing exactly at the boundary, the second resuming from the wrapped source with the correct destination offset. If only one free slot remains, the second half is quietly dropped and only the first part enqueues, a documented, vanishingly rare edge case for small DPLC transfers that happen to straddle a boundary.

6 · Dirty buffers and static DMA

Some transfers never change shape: the sprite table is always 640 bytes from the same RAM buffer to the same VRAM address, and each palette line is always a fixed 32-byte copy. For these, BuildStaticDMA pre-computes the entire 14-byte queue entry once, at boot, and Enqueue_Dirty_Buffers just block-copies the pre-built entry into the Critical queue whenever the corresponding dirty flag says something actually changed. No source/dest/length math, no boundary check, no SR masking, none of QueueDMATransfer's per-call overhead; just a macro-driven copy.

New here? · Why bother pre-computing something the CPU could just calculate each time?

A DMA transfer needs five pieces of information written into VDP registers before it can start: source address (split across three bytes), length, and a destination command word. If the source, destination, and length never change, and for the sprite table and each palette line they genuinely never do, then computing those bytes over and over, every single frame, is pure waste. The values are the same every time.

This is the same instinct behind compiling a program once instead of re-interpreting it on every run: do the work exactly once, at a moment that costs nothing (boot, before the player is even looking at a screen), and every later use is just a fast copy of the finished answer. Aeon leans on this build-time-and-boot-time philosophy everywhere; static DMA entries are one of its smallest, clearest examples.

BuildStaticDMA:
        moveq   #-$6C, d0                      ; $94 sign-extended
        move.l  #$93979695, d5

Palette_Dirty is a 4-bit mask, one bit per 32-byte palette line, so a frame where only the water line's colors cycle sends 32 bytes, not 128. HScroll works differently: the Hscroll_Dirty_Start/Hscroll_Dirty_End fields exist in RAM but are dead, nothing reads them. Decision Known issue The dirty flag clears even when the DMA was dropped. Look again at the excerpt above: Palette_Dirty and Sprite_Table_Dirty are cleared unconditionally, immediately after the queueStaticDMA call, regardless of whether the Critical queue actually had room. If the Critical queue was already full (eight slots, all spoken for by something else that frame), the enqueue silently fails and the dirty bit that would have triggered a retry next frame is already gone. The failure scenario is concrete: a palette change lands on a frame where the Critical queue happens to be saturated, the DMA never fires, the dirty flag is cleared anyway, and the stale colors persist on screen indefinitely with no automatic path back to correct, until something else marks that line dirty again. 7 · The plane buffer

Nametable writes never touch the VDP during the game loop. Instead, the camera scrolling into new territory appends tile updates to one shared 1,536-byte RAM buffer (Plane_Buffer, 768 words, PLANE_BUFFER_SIZE), and step 4 of the VBlank pipeline drains that buffer to VRAM with direct CPU writes, once per frame, in one pass.

New here? · What does "the camera scrolling reveals new tiles" actually mean?

A scroll plane is much bigger than the screen, 64 tiles wide and 64 tall here, but the screen only ever shows a 40-tile-wide window into it. As Sonic runs right, that window slides across the plane, and the VDP's own hardware scroll registers handle sliding the picture smoothly, pixel by pixel, for free. What the hardware cannot do on its own is invent new content: the tiles that scroll into view from off the right edge have to already be sitting in that plane's nametable before the window reaches them, or the player sees the plane's old, wrapped-around content instead of the level.

So "the camera scrolling into new territory" means exactly that: every time the visible window crosses into a strip of the plane it hasn't shown yet, something has to write fresh tile data into that strip, in VRAM, before the scroll register reaches it. That something is the plane buffer described in this section.

Three producer routines feed it, all doing cache lookup, VRAM-address math, and the buffer write in a single pass, no separate setup step:

  • Draw_TileColumn: Plane A, one new tile column, word-mode drain
  • Draw_TileRow_FromCache: Plane A, one new tile row, longword-mode drain
  • Draw_BG_TileColumn: Plane B (background), always a fixed 32-word column strip

Plane A and Plane B entries interleave in this same buffer, distinguished only by which VRAM destination address each entry carries, not by separate buffers. Each entry starts with a VRAM address word and a flags/count word, bit 15 set for column mode (word-at-a-time drain, autoincrement $80) or clear for row mode (longword-packed drain, autoincrement $02), and the whole buffer is terminated by a zero address:

VInt_DrawLevel:
        tst.w   Plane_Buffer_Ptr
        beq     .reset                             // nothing to draw

        lea     Plane_Buffer, a0
        lea     VDP_DATA, a6
        lea     VDP_CTRL_OFF(a6), a5

    .next:
        moveq   #0, d0
        move.w  (a0)+, d0                          // VRAM addr (0 = end)
        beq     .done
        move.w  (a0)+, d1                          // flags | count
        bmi     .col_entry

        // -- ROW entry: autoincrement $02, longword drain --
        move.w  #$8F02, (a5)
        lsl.l   #2, d0
        addq.w  #1, d0
        ror.w   #2, d0
        swap    d0
        move.l  d0, (a5)
    .drain_row:
        move.l  (a0)+, (a6)
        dbf     d1, .drain_row
        jbra    .next
engine/level/plane_buffer.emp:381

Each producer checks remaining capacity before writing (worst case for a column: two headers plus 64 data words). On overflow, the update is silently dropped for that frame; the routines' own header comments say so directly (Out: none (silently drops if buffer full)), and there is no re-queue mechanism. In practice a dropped update reads as a momentary pop at the screen edge, not corruption, but it is genuinely lost, not delayed.

Teaching example · not engine code

Say the camera whips sideways fast enough in one frame to cross four 16-pixel block boundaries at once. Four calls to Draw_TileColumn queue four column entries, each up to 136 bytes worst case (two headers plus 64 rows). If the buffer already had, say, 1,400 of its 1,536 bytes spoken for from earlier in the frame, the third or fourth column simply won't fit and its cmpi.w capacity check fails before it writes a single word. The first two columns still land at VBlank; the third and fourth are gone. The player sees the newly-revealed strip pop in a frame late at the very edge of the screen, exactly where a fast pan is least likely to be scrutinized.

Decision One shared buffer and one shared pointer, not two per-plane buffers with independent dirty flags. The original plan called for Plane_A_Buffer/Plane_B_Buffer and matching Plane_A_Dirty/Plane_B_Dirty flags. Neither exists in the shipped engine. VInt_DrawLevel gates on a single check, tst.w Plane_Buffer_Ptr, and drains everything in one pass regardless of which plane contributed which entry. This is the largest gap between the original design and what actually shipped: not a regression, just a simplification that turned out to need nothing the two-buffer plan offered, since VRAM destination address already disambiguates every entry.

Who does what

Aeon ✓ one buffer, entries tagged by destination address

Two more design ideas from the original plan never got built and are worth naming so nobody goes looking for them in the code. A Plane_Double_Update_Flag would have queued two sequential updates when the camera moved more than 16 pixels in a single frame; it does not exist, and each producer always queues exactly one update per call regardless of how far the camera moved. And the buffer's generous size, 768 words against S.C.E.'s 576, was sized for a worst-case diagonal fast-scroll around 400 words per plane, with headroom for section transitions; that sizing rationale is carried over design reasoning, not something independently re-measured against this engine's own worst case.

8 · The Z80 and DMA

The classic rule, repeated in every Genesis hardware reference, is: halt the Z80 before starting any DMA, release it after. The two CPUs share one bus, and a Z80 fetch landing mid-transfer can corrupt the data in flight. Every reference engine follows this rule literally: hold stopZ80 across the entire DMA window.

New here? · Why do two CPUs sharing a bus need any of this at all?

The 68000 and the Z80 are two independent processors, but there is only one set of wires running out to the cartridge and its ROM. Ordinarily that's fine: the 68000 does almost all the work, and the Z80 occasionally needs to fetch a byte of sound data across that same set of wires, a request that gets quietly relayed onto the 68000's bus and handled in between other traffic.

A DMA transfer is a burst of the VDP reading many bytes off that same bus, back to back, as fast as the hardware allows. On some board revisions, a Z80 fetch that happens to land inside that burst doesn't just wait politely; it can interleave with the transfer and corrupt the data actually arriving in video memory, showing up as garbled tiles or shredded sprites. "Stop the Z80 during DMA" is the blunt, always-safe fix. Section 8 explains why Aeon narrows that fix to something more precise instead.

Aeon's sound-on build does not. Holding the Z80 for the whole VBlank pipeline starves the sound driver of the steady feed its DAC drum channel depends on, audibly: sampled drums skip. So instead of a held fence, VInt_Level and VInt_Lag both bracket the whole VDP/DMA window with a single flag in Z80 RAM, SND_CTRL_DMA_ACTIVE. Raising that flag at the start and clearing it at the end each need a momentary stopZ80, just long enough to write one byte (you have to own the bus to touch Z80 RAM at all). That pair of one-byte writes is the only bus-hold in the sound build. The DMA burst itself, everything between the brackets, runs with the Z80 free:

    ifdef SOUND_DRIVER_ENABLED
        stopZ80
        move.b  #1, (SND_Z80_BASE+SND_CTRL_DMA_ACTIVE).l   ; raise: DMA window open (Z80 -> DRAIN)
        startZ80
    endif
    ifndef SOUND_DRIVER_ENABLED
        stopZ80
    endif
engine/system/vblank.asm:49
Decision A flag bracket, not a held fence, in sound-on builds. While SND_CTRL_DMA_ACTIVE is up, the Z80's sound driver feeds the DAC only from a ring buffer already sitting in its own RAM and never reads ROM across the shared bus, so it cannot land inside a DMA burst even though it's technically still running. Only the two one-byte flag writes themselves briefly hold the bus, a few cycles each, not the whole pipeline. The rule the hardware actually enforces (no Z80 bus access during a transfer) is honored precisely; the classic full-DMA fence enforces a blunter version of the same rule (no Z80 activity at all) that costs far more than the hardware requires. Sound-off builds keep the classic fence, since there's no driver to starve.

Who does what

Aeon (sound-on) ✓ flag bracket, Z80 free during DMA (MegaPCM-2-informed drain model) Classic Sonic drivers: stopZ80 around every DMA Aeon (sound-off) ✓ classic fence retained, no driver to protect

The bracket opens as the very first act of the VBlank pipeline, before any VDP work at all, and closes only after the last DMA drain (step 10, section 4), so its coverage matches the pipeline's DMA-touching steps exactly, no wider and no narrower.

9 · The HBlank seam

Where VBlank is the once-per-frame window for bulk updates, HBlank is the tiny gap at the end of every single scanline, up to 224 times a frame, and it's the hook raster effects use to change VDP state mid-picture: a different scroll value on line 100 than line 99 is what makes parallax bend a background instead of just sliding it.

New here? · How does changing a scroll value mid-frame bend a background?

A plane's horizontal scroll value is a single number: shift everything on this plane left or right by this many pixels. Normally that number is set once and holds for the whole frame, so the plane slides as a rigid sheet. But the VDP reads the scroll value fresh, from a table in VRAM, for every scanline it draws, not just once at the top of the frame.

An HBlank interrupt firing between scanlines is the moment to rewrite that table entry before the next line gets drawn. Give line 50 a different scroll value than line 49, and the picture literally bends at that seam, foreground rows scrolling faster than a distant background above them, or a wavy heat-shimmer effect where each line offsets by a slightly different amount. It's not a special hardware mode; it's the ordinary scroll register, just rewritten far more often than once per frame.

The dispatch mechanism is built and wired today; nothing installed on top of it yet is. The HBlank vector points at a tiny, permanent ROM stub that reads a handler pointer from RAM and calls through it:

HBlank_Dispatch:
        movem.l d0-d1/a0, -(sp)
        movea.l (HBlank_Handler_Ptr).w, a0
        jsr     (a0)
        movem.l (sp)+, d0-d1/a0
        rte
engine/system/hblank.asm:12

The contract is strict on purpose: an installed handler may clobber only d0-d1/a0, because this stub can run 224 times in a single frame and every register it saved would be paid for on every one of those lines. Boot installs HBlank_Null, a bare rts, as the default:

HBlank_Null:
        rts
engine/system/hblank.asm:22

That "does nothing" handler is not free: the dispatch overhead plus the interrupt entry and exit, movem save, jsr, rts, movem restore, rte, runs roughly 180 cycles per scanline, not the 20 or so you might assume from reading the handler itself. A proposed RAM-trampoline fix, not yet applied, would shortcut the indirect call for the common empty-table case and bring that down to roughly 76 cycles. Today, every scanline of every frame pays the full 180.

The raster command table walker, the system that would actually install per-section stackable effects (shadow/highlight lighting, palette gradients, water-line splits) through this seam, is design-stage: zero handlers exist beyond HBlank_Null as of this page. The pointer indirection above is ready to receive it; nothing currently does. Four of the five commercial disassemblies studied while designing Aeon (Vectorman, Thunder Force IV, Gunstar Heroes, Alien Soldier) RAM-patch HBlank the same way, which is why the dispatch shape was worth building ahead of the walker that will eventually use it.
