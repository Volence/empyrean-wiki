Boot & Hardware Init

What happens between power-on and the first frame of gameplay: the 68000 reads two longwords from address zero, proves the cartridge is legitimate, programs the video chip, wakes the sound hardware, and wipes every byte of memory. The biggest wipe runs in parallel, on a second clock, while the CPU keeps working. This page documents the boot sequence as built, verified line-by-line against engine/system/.

1 · The first 256 bytes

A Sega Genesis ROM does not begin with code. It begins with a table: 64 longword addresses, occupying exactly the first 256 bytes of the cartridge. When power is applied, the 68000 reads the first entry into its stack pointer and the second into its program counter, and those two reads are the boot process, as far as the CPU is concerned. Everything else on this page is just what the code at that second address chooses to do.

New here? · What is a vector table?

A CPU sometimes needs to drop everything and run a special routine: the machine was just reset, a piece of hardware raised its hand, or the program did something illegal (like dividing by zero). The 68000's designers gave each of these events a number, and reserved the first 256 bytes of memory as a lookup table. Event number 3 happened? Jump to the address stored in slot 3. Each slot is called a vector because it points somewhere.

On the Genesis, "the first 256 bytes of memory" means the first 256 bytes of the cartridge, so every game ships its own table. Filling in all 64 slots, even the ones "that should never fire", is the difference between a crash you can diagnose and a machine that silently leaps into garbage.

New here? · What is the stack, and why does a console need one?

When code calls a subroutine, the CPU has to remember where to come back to. It stores that return address in a region of RAM called the stack, which grows downward: each call pushes an address on, each return pops one off. Interrupts use it too. When the video chip interrupts the CPU mid-frame, the CPU pushes its current position onto the stack so it can resume exactly where it left off.

The stack has to live somewhere in the 64KB of Work RAM, and the very first longword of the ROM decides where. Put it in a bad place and a deep call chain silently overwrites your game's variables. That is one of the nastiest classes of bug on this hardware, because the corruption appears far away from its cause.

The table itself is plain data, and deliberately so. The org-0 image (the ROM bytes assembled at address zero) is pure data until address $200 (vectors, then the ROM header), so even the trivial rte instruction behind NullInterrupt is emitted later in the ROM, not squeezed between the tables. Here is the real top of the file:

Two details worth making explicit. First, exceptions do not all funnel into one monolithic ExceptionHandler: each CPU exception gets its own label (BusError, AddressError, ZeroDivide, …), the reserved vectors up through the spurious-interrupt slot go to ErrorExcept, and all sixteen TRAPs plus the tail reserved block go to ErrorTrap. Per-vector labels let the debug error handler name exactly what fired. Second, the ROM header at $100-$1FF is not a hardcoded block: it is emitted by a gameHeader macro from game-declared strings, and every field width is asserted at build time (more on that in section 9).

What those per-vector labels buy is a real crash screen. Every exception stub feeds a named message ("BUS ERROR", "ADDRESS ERROR", "ILLEGAL INSTRUCTION", ...) into Vladikcomper's MD Debugger and Error Handler v2.6, a third-party debugging module the engine embeds at the very end of the ROM. When any of those vectors fires, the handler takes over the machine: it sets up its own video state from an internal register table (so it can draw even if the crash happened before boot configured the VDP), then prints the exception name, the program counter, the registers, the caller, and resolves addresses to symbol names using a table the build appends to every ROM. On the error screen, button B shows an exception backtrace and button A decodes the address registers. This is not a debug-build luxury: the handler ships in release builds too (only assertions and the KDebug logging interface are gated to DEBUG), so a crash on real hardware is a readable screen, never a silent freeze.

New here? · What happens when a Genesis game crashes?

On a modern computer a crashing program gets caught: the operating system shows a dialog, writes a crash log, and the machine carries on. A Genesis has no operating system. When the CPU hits a fault (a read from a nonsense address, an opcode that is not an instruction), it jumps through the vector table, and whatever sits at the other end of that vector is the crash handling. Games that left those slots pointing at garbage simply froze, or worse, leapt into random memory and kept "running" while corrupting everything they touched.

That is why filling in all 64 vectors matters, and why pointing them at a real error screen is one of the highest-value things a Genesis engine can do. The crash still happens, but instead of a black screen you get the exception's name, where it fired, and the trail of calls that led there: a 1990s console showing you something very close to a modern stack trace.

Before the decisions below, it helps to see the terrain they live in. The 68000 addresses 16MB, and every piece of hardware this page touches sits at a fixed spot inside that space; "talking to the VDP" or "copying a program into the Z80" just means writing to the right addresses. Click a region to see what lives there and where this page uses it.

Decision Initial stack pointer = $FFFFFF00, not $00000000. The stack grows downward from near the top of Work RAM, far from game data at low RAM addresses, and the top 256 bytes ($FFFFFF00-$FFFFFFFF) stay out of the stack's path (Aeon parks its cold-boot marker at $FFFFFF00). A common alternative is SSP = $0: the first push predecrements $00000000 to $FFFFFFFC, which the 68000's 24-bit bus addresses as $FFFFFC, the top of work RAM, so the stack simply lives at the very top of RAM (the same address wrap this boot code uses on purpose for the RAM clear in section 5). Several shipping games do exactly this. Aeon instead fixes SSP at $FFFFFF00 so the top 256 bytes stay reserved for the cold-boot marker, not because $0 is unsafe.

2 · Proving we're a licensed game

Before Aeon touches the video chip, it has to satisfy the console's lock. Later Genesis revisions ship with the Trademark Security System (TMSS): if the hardware revision nibble is non-zero, the boot code must write the ASCII longword "SEGA" to $A14000, or the next access to the VDP data port hangs the CPU permanently. Revision 0 machines (early Model 1s) have no TMSS and no such register, so the write is gated on the revision check.

New here? · Why did Sega put a lock on its own console?

In 1989 Sega lost control of who could make Genesis games. A publisher called Accolade reverse-engineered the console and started shipping unlicensed cartridges: perfectly playable, zero royalties to Sega. So newer console revisions shipped with a lock: on boot, the hardware demands that the cartridge write the word "SEGA" into a magic register, and in exchange the console displays "PRODUCED BY OR UNDER LICENSE FROM SEGA ENTERPRISES LTD." on a splash screen. The legal theory was that an unlicensed publisher now had to either fail the lock (game won't boot) or falsely display Sega's trademark.

It backfired. Accolade satisfied the lock anyway, Sega sued, and in 1992 the Ninth Circuit ruled (Sega v. Accolade) that reverse engineering for compatibility is fair use, a landmark decision that still protects emulator and interoperability work today. The lock stayed in the hardware, though, which is why every game since, including Aeon, must perform this little handshake, and why so many 90s games greet you with that license screen.

Two footnotes to the handshake. The magic register is only half of TMSS: the console also checks that the ROM header's first field, the console-name string at $100 (which gameHeader emits from the game's GAME_CONSOLE symbol), begins with "SEGA". And on TMSS machines the cartridge is not actually the first thing to run: the console's internal boot ROM executes before EntryPoint, performs those checks, and draws the license splash. By the time Aeon's first instruction runs, the player has already been looking at that screen.

Decision Soft-reset checks run before TMSS. Pressing RESET only resets the 68000; the VDP may still be mid-DMA, and reprogramming it blind would corrupt the transfer in flight. So the very first instructions test the port A and expansion control registers (non-zero means the machine was already running), and the warm path waits out any in-flight DMA before anything else touches the hardware.

EntryPoint:
        tst.l   (HW_PORT_A_CTRL_FULL).l
        bne.s   Warm_Boot
        tst.w   (HW_EXPANSION_CTRL_FULL).l

Warm_Boot:
        ; Wait for any in-progress DMA
.wait_dma:

        btst    #1, d0
        bne.s   .wait_dma
        ; Fall through to cold boot.

Cold_Boot:
        ; TMSS handshake (§0.2)
        move.b  (HW_VERSION).l, d0
        andi.b  #$F, d0
        beq.s   .no_tmss

.no_tmss:
engine/system/boot.asm:6

Note the fall-through at the end of Warm_Boot: it waits for any in-flight DMA to finish, then continues straight into Cold_Boot. A soft reset runs the same full hardware initialization as power-on, including the 64KB RAM wipe; the only thing "warm" about it is the DMA wait.

A warm-reset persistence scheme, preserving a small RAM region (lives, score) across RESET presses, is designed but not yet built. Today every boot, warm or cold, clears all 64KB of RAM, including the reserved marker address at $FFFFFF00, and then unconditionally rewrites the CROSS_RESET_MAGIC longword ('INIT') there; nothing survives a soft reset. The address reservation is real, and the preservation mechanism will build on it.

3 · Programming the video chip

The VDP has 24 configuration registers, $00 through $17, and after a reset their contents are undefined. Aeon writes all 24 from a compile-time table in one tight loop: 24 bytes of data, five instructions of code. Two properties of the chosen values matter for everything that follows: the display is OFF (nothing ugly is visible while we work) and DMA is ON (so the VRAM fill in section 5 can start immediately).

New here? · What is the VDP?

The Genesis has a dedicated graphics processor: the Video Display Processor, or VDP. It owns its own 64KB of video memory that the main CPU cannot address directly; everything goes through two narrow mail slots (a data port and a control port). Every frame, entirely on its own, the VDP walks its memory and composites the picture: two scrolling tile planes, up to 80 sprites, and a window layer.

Its 24 registers are the chip's settings panel: where in video memory each plane's map lives, how big the planes are, what the backdrop color is, whether the display is even on. You set them by writing magic words to the control port. There is no "register 5" address, just a convention where $85xx means "put xx in register 5". Get one value wrong and the symptom appears somewhere else entirely, which is why engines treat this table as precious, reviewed data.

Decision A RAM-resident shadow copy of the VDP registers. The VDP's registers are write-only: you can never ask the chip what a register currently holds. So Aeon keeps the authoritative copy in RAM: a 19-byte VDP_Shadow struct mirroring registers $00-$12 (the five DMA registers $13-$17 are per-transfer state owned by the DMA queue, and deliberately not shadowed). Game code writes registers through a setVDPReg macro that updates the shadow and sets a dirty bit; each VBlank, Flush_VDP_Shadow writes only the dirty registers to hardware. The dirty mask is a single longword (VDP_Dirty_Mask: ds.l 1, bits 0-18), so one beq skips the whole flush on the many frames where nothing changed.

The shadow comes with a written rule about bypassing it. setVDPReg is the only sanctioned path for settled state on registers $00-$12 (display enable, scroll mode, plane bases); a direct control-port write to those registers is legal in exactly two cases. Transient pre-DMA setup, like the auto-increment tweak the level streamer makes immediately before a transfer, may write directly, because nothing reads the value back as authoritative. And raster-effect handlers may write registers freely mid-frame (that is the entire point of a raster effect) but must not touch the shadow or the dirty mask: the shadow represents settled frame state, and the next VBlank flush re-asserts it. Everything else goes through the macro, because a hardware-only write to settled state risks being silently overwritten by the next dirty flush.

4 · Waking the sound hardware

Sound on the Genesis is a second computer. A Zilog Z80, the CPU of the Master System, lives on its own 8KB of RAM and drives the YM2612 FM synthesizer and the SN76489 PSG. At power-on the Z80 is executing garbage, the PSG may be shrieking, and the FM chip's state is undefined. Boot has to bring all three to a known, silent state.

New here? · Why does a Genesis have two CPUs?

Music is relentless. A synthesizer chip doesn't hold a note the way a piano does; the driver has to feed it register writes constantly, on a strict schedule, or the music stutters. If the main CPU did that job, every game would have to interleave "update the bassline" with "run the physics," and a slow frame would warp the music.

Sega's answer was to put a whole second CPU in the console: a Z80, cheap and well-understood, which also doubled as the brain for Master System backward compatibility. It runs the sound driver as an independent program on its own little 8KB of RAM. The main CPU just drops short messages ("play song 3", "ring chime") into a shared mailbox and gets on with the game. Aeon leans on this hard: its sound driver is fully Z80-autonomous.

The bus dance is precise. The 68000 asserts the Z80's reset line, requests the shared bus, releases reset, and polls until the bus is granted. Only then can it copy a program into Z80 RAM, byte by byte, because Z80 RAM tolerates only byte writes. What gets copied is the full sound driver in the default build (SOUND_DRIVER_ENABLED is on by default); only sound-off builds load the tiny idle program instead. Then reset is asserted again and held for ~264 cycles of the 68K clock before the Z80 is released to run. The YM2612, which shares that reset line, needs its reset held for at least 192 cycles of its own clock; on the Genesis the YM2612 is clocked at the same rate as the 68K, so the two figures compare directly.

Where does the driver come from? It is not a separate file the CPU has to locate at runtime: the assembled Z80 blob is physically embedded inside the boot data table, between the DMA fill command and the PSG silence bytes. Boot reads that table strictly sequentially through (a5)+, so when the copy loop starts, a5 is already pointing at the first driver byte. No address to load, nothing to look up.

Silencing the PSG takes four bytes: one "maximum attenuation" command per channel ($9F $BF $DF $FF) written to the PSG port at $C00011. The YM2612 gets an explicit key-off pass: register $28 is the key-on/off switchboard, and boot writes channel numbers 0-2 and then $04-$06 (the FM chip splits its six channels across two banks) with all operator bits clear. This step runs late, after the video-memory clears, and the whole pass is bracketed by stopZ80/startZ80, because the newly-started Z80 driver also talks to the FM chip and two writers on one chip is a race.

New here? · Why must the Z80 be stopped before touching video memory?

The two CPUs share one road. When the Z80 wants music data from the cartridge, its request is relayed onto the 68000's bus, the same bus the VDP uses when it DMA-copies art into video memory. On real hardware (especially early board revisions), a Z80 fetch landing in the middle of a DMA transfer corrupts the transferred data: garbage tiles, shredded sprites. The classic rule, repeated in every Genesis reference, is therefore: halt the Z80 (stopZ80) before starting any DMA, resume it after.

Aeon narrows this rule to what the hardware actually requires; see the decision block just below.

Decision The Z80 keeps running during DMA (sound builds). Halting the Z80 for every transfer starves the sound driver: sampled drums skip. So instead of the classic full-DMA stopZ80 fence, VBlank brackets its whole VDP/DMA window with a SND_CTRL_DMA_ACTIVE flag in Z80 RAM. While the flag is up, the Z80 keeps running but feeds the DAC (the chip's raw-sample channel, the part that plays drum recordings) only from a ring buffer in its own RAM, never fetching ROM across the shared bus; only the two one-byte flag writes themselves briefly hold the bus. Uninterrupted drums, zero DMA corruption risk, and the hardware rule is honored in its precise form (no Z80 bus access during DMA) rather than its blunt form (no Z80 at all). The full fence survives only in sound-off builds.

Decision Sound_Init waits forever, on purpose. The last sound step of boot (phase I in section 8) is a handshake: the 68000 polls a byte in Z80 RAM until the driver finishes its own init and writes an "alive" marker there. Each probe grabs the Z80 bus (the only reliable way for the 68000 to read Z80 RAM), compares, and then releases the bus before probing again; a continuous hold would deadlock the handshake, since a stopped Z80 can never finish booting and post the marker. There is deliberately no timeout: a driver that never comes up means the ROM itself is broken, and hanging at one well-defined address beats limping onward without sound (Sound_Init, engine/sound/sound_api.asm:39).

5 · Clearing 64KB the clever way

Boot must zero three memories: 64KB of video RAM, 64KB of work RAM, plus the VDP's two small internal memories (CRAM, the color palette, 128 bytes; VSRAM, the vertical-scroll table, 80 bytes). The naive order is serial: clear one, then the next. Aeon overlaps them.

New here? · What is DMA?

Direct Memory Access: hardware that copies (or fills) memory by itself, without the CPU executing a copy loop. On the Genesis, the DMA engine lives inside the video chip and runs on the video chip's clock. You tell it "fill video RAM with zeros, all 64KB", kick it off with one write, and then the VDP grinds through the job on its own while your CPU is free to do something else entirely.

That last clause is the whole trick of this section. Two processors, two clocks, one goal: get everything to zero. Modern engines call this async I/O; on a 1988 console it just takes knowing that the fill's "busy" bit is something you can come back and check later.

The VDP register table already primed a fill (register $17's fill mode, length $FFFF), so starting it takes four instructions: set auto-increment to 1 for the byte-wise fill, aim the command at VRAM address 0, and write the fill byte:

        ; Start VRAM DMA fill (§0.7) - runs in background on VDP clock
        move.w  #vdpReg($0F, $01), (a4)    ; auto-increment = 1 for byte-by-byte DMA fill
        move.l  (a5)+, (a4)                ; vdpComm(0, VRAM, DMA)
        moveq   #0, d0
        move.w  d0, (a3)                    ; trigger fill (fill byte = 0)
engine/system/boot.asm:51

While the VDP fills, the CPU does the entire previous section's work (the Z80 bus dance, the driver copy, the YM-safe reset), then clears its own 64KB of work RAM, then silences the PSG. The RAM clear itself is a lovely idiom: start the address register at $00000000 and write longwords with pre-decrement, letting 32-bit address arithmetic wrap around to $FFFFFFFC and walk down through all of RAM:

        ; Clear Work RAM - 64KB (§0.7)
        movea.l d0, a6                      ; a6 = 0
        move.w  d6, d2                      ; d2 = $3FFF (longword count)
.clear_ram:
        move.l  d0, -(a6)                   ; wraps: $00000000 → $FFFFFFFC → ... → $FFFF0000
        dbf     d2, .clear_ram
engine/system/boot.asm:86

Only after all that does boot poll the VDP status register's DMA-busy bit, and by then the fill is typically already done, so the "wait" costs almost nothing. Auto-increment goes back to 2 (restored from a trailing word in the boot data table), and the CPU hand-clears CRAM and VSRAM with short loops (DMA fill only works on VRAM; for 208 bytes total, a CPU loop is the right tool anyway).

For the record, those real durations: the 64KB work-RAM clear alone is roughly 393,000 CPU cycles, about 51 milliseconds at 7.67 MHz (16,384 iterations of a 14-cycle move.l d0,-(a6) plus a 10-cycle dbf), and it is the biggest single CPU chore in the sequence. Decision DMA-parallel boot clear (novel). Every reference disassembly clears memory serially: fill VRAM, wait, then do everything else. Aeon starts the VRAM fill first and hides the Z80 init, the 64KB RAM clear, and the PSG silence inside it, polling for completion only when there is nothing left to overlap. This is a modern async-I/O pattern applied to 1988 silicon; the insight is simply that the fill runs on the VDP's clock, not the CPU's.

6 · Where in the world is this console?

The same cartridge must run on a Japanese Genesis at 60Hz and a European Mega Drive at 50Hz. One byte tells the software which world it woke up in: the version register at $A10001. Bit 7 is domestic/overseas, bit 6 is NTSC/PAL, and the low nibble is the hardware revision (the same nibble the TMSS check read earlier). Boot stores the full byte to Hardware_Region, masks the top two bits into Region_Flags, and then makes the one decision that matters: NTSC or PAL.

New here? · 50Hz vs 60Hz, and why PAL games ran slower

Old consoles didn't render "when ready"; they rendered in lockstep with the television. American and Japanese TVs (NTSC) redraw 60 times a second; European TVs (PAL) redraw 50 times. Since nearly every 90s game ran its physics once per TV frame, a game tuned for NTSC and shipped to Europe unchanged simply ran at 5/6 speed: Sonic jogged, music dragged, and European players genuinely got a 17% slower game. Some studios compensated; many didn't.

PAL has a consolation prize, though: with 312 scanlines per frame instead of 262 but the same 224 visible, the invisible gap between frames (the window where it's safe to blast data into video memory) is nearly twice as long. A PAL console can move almost double the graphics data per frame. That's why Aeon stores a per-region DMA budget, not just a speed factor.

Decision PAL compensation via a fixed-timestep accumulator. Each frame adds Timing_Step to an 8.8 fixed-point Frame_Accumulator, and whole 1/60s ticks are consumed from it. On NTSC the step is exactly 1.0. On PAL it's $0133 ≈ 1.199, so roughly every fifth frame the accumulator yields a second tick (≈60 physics ticks per 50 PAL frames), and every tick that runs is bit-exact NTSC physics. Boot's share is picking the step and zeroing the accumulator, and as of this writing that is the only part wired up: nothing consumes the accumulator yet, and GameLoop still runs one state dispatch per VSync on both regions.

Last piece of world-facing setup: the controller ports. Boot writes $40 to all three port control registers, configuring the TH pin as an output (the pin the joypad-reading protocol toggles to select which button bank the pad reports), and then also writes $40 to all three port data registers, driving TH high as its initial state. The data-port writes matter, since the first controller read assumes a known TH level. Actual button polling happens every VBlank, not at boot.

7 · The interrupt switchboard

Two interrupts drive everything periodic in the engine. IRQ4 fires at horizontal blanking, between scanlines, and is the hook for raster effects. IRQ6 fires at vertical blanking, between frames, and is where all VDP-touching work happens. Aeon wires them asymmetrically, and the asymmetry is the design.

New here? · What is an interrupt, and what is "blanking"?

An interrupt is hardware tapping the CPU on the shoulder: stop what you're doing, run this routine, then resume. The video chip taps twice on a schedule. Once per scanline it can raise HBlank: the beam (real or emulated) has finished a line and is flying back to start the next, a microscopic gap you can use to change video settings mid-picture (that's how one background can scroll at three speeds at once). And once per frame it raises VBlank: the whole picture is done and the beam is returning to the top.

VBlank is sacred in console programming: it's the only comfortable window where video memory can be rewritten without the change tearing across a half-drawn frame. Everything a Genesis game shows you was smuggled into video memory during VBlanks.

Seen as scanlines, a frame is mostly picture. On NTSC, 224 of the 262 lines are active display; VInt fires at the start of line 224, and the remaining 38 lines are the whole window before the next frame begins:

HBlank: RAM-dispatched, contract-bound. The vector points at this ROM stub, which never changes; per-section raster programs are installed by writing HBlank_Handler_Ptr. Handlers may clobber only d0-d1/a0; the stub saves nothing else, because this code can run up to 224 times per frame and every saved register is paid for on every line:

HBlank_Dispatch:
        movem.l d0-d1/a0, -(sp)
        movea.l (HBlank_Handler_Ptr).w, a0
        jsr     (a0)
        movem.l (sp)+, d0-d1/a0
        rte
engine/system/hblank.asm:12

Boot installs HBlank_Null, a bare rts, so the dispatch costs a handful of cycles until a level section wants more.

VBlank: ROM entry, RAM-selected body, built-in lag detection. The handler saves everything, then checks VBlank_Ready, a flag the main loop sets when it finishes a frame's work. If the main loop finished, the handler dispatches through VInt_Ptr (boot installs VInt_Level); if the main loop is still running (a lag frame), it takes a reduced path instead:

VBlank_Handler:
        movem.l d0-a6, -(sp)
        tst.b   (VBlank_Ready).w
        beq.s   .lag
        movea.l (VInt_Ptr).w, a0
        jsr     (a0)
        bra.s   .done
.lag:
        bsr.w   VInt_Lag
engine/system/vblank.asm:8

VInt_Level, the normal-frame body, runs in this exact order: raise the sound DMA flag → Flush_VDP_Shadow → Enqueue_Dirty_Buffers (palette, sprites, HScroll) → VInt_DrawLevel (drain the plane buffer to the nametables) → Process_DMA_Critical → Vscroll_Write (the VSRAM write deliberately comes after the critical DMA drain, because it must follow the HScroll DMA), then the budgeted Important and Deferrable DMA tiers, lower the sound flag, read controllers, latch accumulated press edges, bump the frame counter, and set VBlank_Flag for the waiting main loop. Similar names, opposite directions: VBlank_Ready is the main loop telling the interrupt "my frame's work is done", while VBlank_Flag is the interrupt telling the main loop "VBlank has run: go".

VInt_Lag is not a do-nothing stub. It runs the shadow flush, the dirty-buffer enqueue, Critical DMA, the VSRAM write, controllers, and the frame counter, and it still sets VBlank_Flag at the end: nearly the whole pipeline. The one thing it deliberately skips is the plane-buffer drain: on a lag frame the main loop is mid-write into that buffer, and draining a half-written entry would send tiles to a garbage VDP address. The dirty-flag-gated buffers are safe to drain (their flags are set only after a complete write); the pointer-gated plane buffer is not, so it waits for the next complete frame.

That lag-frame VBlank_Flag is a loaded gun, and the main loop's wait routine is built around not getting shot by it. On a lag frame nobody is waiting; the flag the main loop eventually checks would be stale, left over from an interrupt that already came and went. So VSync_Wait clears the flag before it starts waiting. And there is a second, sharper hazard: the clear-flag and raise-VBlank_Ready writes must land as one atomic act. If a VBlank fires between them (Ready still 0), it runs VInt_Lag, which sets the flag, so the "wait" falls straight through with no real vsync, while Ready is left up for the next VBlank to fully drain a plane buffer the main loop is still writing. The fix costs a few dozen cycles on a once-per-frame path: mask interrupts across the pair, then wait with them live.

        move.w  sr, -(sp)
        move.w  #$2700, sr             ; mask: the pair below must be atomic
        move.b  d0, (VBlank_Flag).w     ; clear any stale lag-frame flag
        move.b  #1, (VBlank_Ready).w    ; this frame's work is done
        move.w  (sp)+, sr               ; unmask BEFORE waiting; IRQ6 sets the flag
.wait:
        tst.b   (VBlank_Flag).w
        beq.s   .wait
engine/system/vblank.asm:182

One loose end the excerpts above leave out: the interrupt side lowers VBlank_Ready itself on the way out of every VBlank, normal or lag, so the ready signal is consumed exactly once per interrupt and can never carry over into a frame that did not earn it.

8 · The whole sequence

Here is the boot order: all thirty-four steps of boot.asm, grouped into phases. Phase C is the parallel window from section 5: everything in it executes while the VDP is filling VRAM.

  1. Reset & TMSS · boot.asm:6-36

    • Soft-reset checks: tst.l port A control, tst.w expansion control
    • Warm path only: poll VDP status until any in-flight DMA finishes, then fall through
    • TMSS: read the $A10001 revision nibble; write "SEGA" to $A14000 if non-zero
    • Read the VDP control port once, resetting its command-word state machine

  2. VDP registers, fill armed · boot.asm:39-55

    • Preload hardware addresses into d5-d7/a0-a4 with one movem
    • Write all 24 VDP registers from BootData_VDPRegs (display OFF, DMA ON)
    • Set auto-increment = 1, then trigger the 64KB VRAM DMA fill (fill byte 0)

  3. Parallel window: CPU works while the VDP fills · boot.asm:60-97

    • Z80 init: reset/bus-request dance, copy in the sound driver (or idle blob), reset again with the ~264-cycle YM2612-safe delay, release
    • Clear all 64KB of Work RAM via the wrapping pre-decrement loop
    • Silence the PSG: 4 attenuation bytes to $C00011

  4. Fill completes, small memories cleared · boot.asm:101-121

    • Poll VDP status bit 1 until the DMA fill is done
    • Restore auto-increment = 2
    • Clear CRAM (128 bytes) with a CPU loop
    • Clear VSRAM (80 bytes) with a CPU loop

  5. Silence FM, zero the CPU · boot.asm:124-143

    • YM2612 key-off on all six FM channels, bracketed by stopZ80/startZ80
    • Clear all 68K registers with one movem.l from freshly-zeroed RAM
    • disableInts: masked while the engine assembles itself

  6. Engine subsystems · boot.asm:146-158

    • VDP_Shadow_Init: copy the boot register table into the RAM shadow
    • Init_DMA_Queue
    • Init_SpriteTable: sprite link chain
    • BuildStaticDMA: precomputed DMA command words
    • Install VInt_Ptr = VInt_Level

  7. World facts · boot.asm:161-185

    • Region detection → Hardware_Region, Region_Flags, Timing_Step, DMA_Budget_Default; zero Frame_Accumulator
    • Controller init: $40 to three control registers and three data ports (TH output, driven high)
    • Install HBlank_Handler_Ptr = HBlank_Null

  8. Interrupts live · boot.asm:188-194

    • setVDPReg mode 2 = $34: VBlank interrupt enabled in the VDP, display still OFF
    • Flush_VDP_Shadow: the VDP must have VInt enabled before the CPU unmasks
    • enableInts: the engine goes live

  9. Handoff to the game · boot.asm:197-220

    • Write CROSS_RESET_MAGIC ('INIT') to $FFFFFF00, the cold-boot-complete marker
    • DEBUG builds: CompressionSelfTest, the golden decompressor self-test
    • Sound builds: Sound_Init, a mailbox idle handshake with the running Z80 driver
    • gameBootHook: game-supplied, may be empty
    • Set Game_State = Game_Entry, Game_State_ID = GAME_ENTRY_ID, clear Game_State_Init
    • bra.w GameLoop: never returns

Note what the last phase says about the engine/game split: boot ends at GameLoop with a game-supplied entry state. The engine has no idea whether the game is Sonic 4 or the demo's white box on a blue field; it just runs whatever Game_Entry the game's manifest declared. And the handoff is live: VBlank is enabled in the VDP and unmasked in the CPU before the branch (phase H), so the game's first state runs with interrupts already firing. There is no masked handoff to unwind.

And what is on screen through all of this? Nothing. The display bit in VDP register $01 is never set during boot: phase H turns $14 into $34, which enables the VBlank interrupt but leaves the display off, and no later boot step touches it. Turning the picture on is the game's job, in its first state, once it has loaded something worth showing. So the player's experience of this entire page is: the console's own TMSS license screen (drawn by the internal boot ROM before EntryPoint even runs), then a black screen for a few frames, then whatever the game puts up. Boot itself is invisible.

GameLoop, the destination of that final branch, is four steps repeated forever: wait for VBlank (VSync_Wait, section 7), drain one pending sound effect into the Z80 mailbox (sound builds), run the game's per-frame debug hook (gameDebugTick), and jump through Game_State. Everything the engine does after boot happens inside one of those four steps.

The boot-time contract

Boot is also where the engine/game wall is easiest to see, because boot consumes the whole game-facing contract in one pass. A game is a manifest: its main.asm declares seven include-macros (config, RAM, engine-block code, object-bank code, data, sound data, game states) and then includes engine/engine.inc, which owns the entire ROM layout from the vector table to the end-of-ROM marker. What boot itself asks of the game is small and precise:

  • GAME_* header strings, consumed by gameHeader at build time (section 9)
  • gameBootHook, invoked once, after Sound_Init: defined in the game's contract file (games/<game>/config/game.asm), and may be empty
  • Game_Entry and GAME_ENTRY_ID, the first state routine and its id, installed as the final act before GameLoop

Two consequences of the wall are worth spelling out. First, a game must boot with sound compiled out: the game-states macro sits outside the engine's sound conditional, so Game_Entry has to resolve even in a sound-off build. Second, the 64KB RAM wipe in phase C covers game RAM too; game variables live in a region that continues from the engine's Engine_RAM_End seam, inside the same 64KB, so every game starts from zeroed memory without writing its own clear loop.

9 · The assembler does the work

A recurring Aeon principle: never compute at runtime what the assembler can compute at build time, and never let a mistake survive to hardware that a build error could have caught. §0 is full of small examples: the VDP register commands are built by a vdpReg function, the boot data block is read sequentially with all offsets resolved at assembly, and the ROM header is generated, not typed.

New here? · What does an assembler actually do, and what is "build time"?

An assembler translates human-readable mnemonics (move.w #$8F02, (a4)) into the raw bytes the CPU executes. But a good macro assembler is also a small programming language that runs while translating: it can compute values, expand templates (macros), and, crucially, refuse to produce a ROM at all if an assertion fails.

"Build time" is that moment, on the developer's machine, before any console is involved. A check that runs at build time costs the finished game nothing (zero bytes, zero cycles), and a failure is a readable error message instead of a black screen. Aeon treats the assembler the way modern engines treat their build pipeline: as the first and cheapest place to catch bugs.

The ROM header shows the pattern best. The gameHeader macro takes nine game-declared strings and refuses to assemble if any has the wrong width; a wrong-length serial becomes a build failure, not a corrupt header field discovered on hardware:

        if strlen(GAME_SERIAL) <> 14
          fatal "GAME_SERIAL must be exactly 14 chars"
        endif
engine/system/header.inc:25
Decision No boot-time checksum check. The header's checksum word at $18E is emitted as zero and patched into the finished ROM by tools/fixheader after assembly. Nothing ever verifies it: boot does not walk the ROM summing words, and a wrong checksum changes nothing at runtime. Classic Sonic games spend their first moments on exactly that walk and halt on a red screen if it fails; it was a defense against corrupted dumps and flaky cartridge hardware, and it costs boot time on every clean start. Aeon treats ROM integrity as the build pipeline's job and keeps the field honest for tools that read it.

The shadow-struct assert is the quiet hero: the flush loop, the dirty mask width, and the "registers $00-$12 only" rule all assume that struct is exactly 19 bytes, and anyone who adds a field learns about all three dependents at build time.

DEBUG builds extend the principle past build time, into the first moments of runtime. Right after interrupts go live (phase I in section 8), boot runs CompressionSelfTest: the build emits one fixed payload compressed three ways (S4LZ v3 plain, S4LZ v3 with a dictionary, ZX0) along with its expected checksum, and the test decompresses each vector into the art staging buffer, then asserts both the checksum and a full byte-for-byte compare. Between vectors the buffer is poisoned with $A5A5: all three vectors share one payload, so without the poison a decoder that did nothing at all would pass on the previous vector's output. Any mismatch between a 68000 decompressor and its build-side encoder halts on the assert screen before any game state has touched the buffer.
